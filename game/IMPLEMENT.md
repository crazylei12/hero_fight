# IMPLEMENT.md

最后更新：2026-04-20

## 文件用途

这份文件用于指导 `game/` 目录下的实际编码与实现工作。

它回答的是：
- 开始写 Unity 工程时先做什么
- 改动不同类型内容时最少要验证什么
- 什么时候该继续实现，什么时候该先停下来修问题
- 如何判断当前改动已经达到可交付状态

这份文件与 `game/AGENTS.md` 的关系是：
- `game/AGENTS.md` 负责长期规则与边界
- `game/IMPLEMENT.md` 负责日常实施顺序与验收动作

## 开始实现前必须确认

正式开始编码前，至少确认以下几点：
- 已阅读 `game/AGENTS.md`
- 已阅读 `../docs/planning/project-foundation.md`
- 已阅读阶段一的战斗、英雄、架构、实施计划和启动清单文档
- 如果本次任务涉及持续区域技能、普通攻击飞行物、技能范围特效或顶视角战斗 VFX，已阅读 `../docs/planning/stage-01-skill-area-presentation-usage.md`
- 当前任务确实属于第一阶段最小原型范围
- 当前没有未解决的文档冲突

如果以上任一项不满足，不应直接开始大规模编码。

## 实施总原则

- 先搭骨架，再补内容
- 先跑通主循环，再补细节
- 先保编译，再加功能
- 先保统一系统，再写特例
- 先保日志和可解释性，再补表现层精修
- 特效任务默认先确认对象和期望效果，再查项目现有 prefab 与素材库
- 特效任务默认先给用户候选方案或自制方案说明，确认后再产出项目专用 prefab，再由程序调用
- 资源包 prefab 默认当素材来源，不默认当最终交付物

## 推荐实施顺序

### 1. 工程与目录

先完成：
- Unity 工程初始化
- `Assets/` 目录结构
- 基础场景
- 基础脚本目录

只有目录和场景骨架稳定后，才开始大量写逻辑脚本。

### 2. 静态数据结构

先写：
- 英雄定义
- 技能定义
- 普攻定义
- 战斗输入对象
- 战斗结果对象
- 枚举和基础类型

先让数据结构稳定，再写运行时逻辑。

### 3. 战斗主循环

先写：
- `BattleManager`
- `BattleClock`
- `BattleScoreSystem`
- `BattleEndResolver`
- `BattleEventBus`

先确保一场战斗有开始、推进、结束主干。

### 4. 单位基础循环

再写：
- 运行时英雄实例
- 出生
- 移动
- 索敌
- 普攻
- 死亡
- 复活

先保证单位能闭环运行，再补技能系统。

### 5. 数值与状态系统

再写：
- 伤害结算
- 暴击结算
- 治疗结算
- 防御修正
- 状态系统
- 第一种控制 `眩晕`

### 6. 技能系统

再写：
- 小技能释放
- 大招释放
- 技能目标选择
- 技能效果执行
- 技能 CD 管理

### 7. 职业 AI 模板

再写：
- 六职业默认行为模板
- 大招时机判断差异
- 默认索敌差异

### 8. 首批英雄配置

最后再创建：
- 六个模板英雄
- 对应的说明文档
- 基础占位资源绑定

### 9. 最小 UI 通路

在核心战斗稳定后接：
- HeroSelect
- Battle HUD
- Result UI
- 日志面板

## 每次实现时的工作流

### 1. 先确认本次改动类型

先判断本次主要改动属于：
- 数据改动
- 战斗逻辑改动
- UI 改动
- 工程结构改动

不同类型改动，最低验证路径不同。

### 2. 控制单次改动范围

单次实现尽量只处理一个明确子目标，例如：
- 只建立 BattleInput / BattleResult
- 只建立 RuntimeHero
- 只建立死亡与复活流程
- 只建立 HeroSelect 的基础选人提交

不要一口气同时改战斗、UI、状态系统和多个英雄特例。

### 3. 完成后立刻做最小验证

改完后不要直接继续堆代码，先做与本次改动对应的最低验证。

### 4. 验证通过后再进入下一步

如果最小验证失败：
- 先修复
- 再继续

## 不同改动类型的最小验证路径

### 数据改动

如果本次改动涉及：
- ScriptableObject
- 英雄/技能/普攻定义
- 枚举
- 输入/结果对象

至少验证：
- 结构能正常创建或读取
- 字段命名和用途没有偏离既有规范
- 运行时可以消费这些结构
- 没有破坏既有配置的一致性

### 战斗逻辑改动

如果本次改动涉及：
- BattleManager
- RuntimeHero
- 索敌
- 普攻
- 死亡/复活
- 数值结算
- 状态系统
- 技能系统
- AI 决策

至少验证：
- 战斗可以正常开始
- 单位能正常执行本次改动涉及的动作
- 60 秒计时和击杀计分规则未被破坏
- 死亡后 5 秒复活流程未被破坏
- 同分加时规则入口仍然存在
- 小技能与大招仍有不同释放逻辑

### UI 改动

如果本次改动涉及：
- HeroSelect
- HUD
- Result UI
- 日志界面

至少验证：
- 选人 -> 开战 -> 结果主通路仍能走通
- UI 没有直接承担战斗规则判定
- UI 展示的时间、比分、结果与逻辑层一致

### 技能区域表现改动

如果本次改动涉及：
- 持续区域技能特效
- 技能区域 prefab
- 技能区域自定义表现 controller
- 技能范围视觉与逻辑对齐
- 技能区域排序层级

至少验证：
- 技能实际落点正确
- 视觉范围大致覆盖逻辑范围
- 持续时间与 tick 节奏未偏离技能配置
- 英雄仍然位于技能区域特效上层
- `Stage01SampleContentBuilder.cs` 已与当前资产配置同步

### 战斗特效 prefab 改动

如果本次改动涉及：
- 新建项目专用技能特效 prefab
- 新建项目专用飞行物 prefab
- 资源包 prefab 的二次组装
- 程序从资源包原始 prefab 切换到项目专用 prefab

至少验证：
- 程序实际引用的是项目专用 prefab，而不是资源包原始 prefab
- prefab 的视角、层级和尺寸适合当前顶视角战斗
- 如果有逻辑联动，相关通用驱动层没有改坏原有战斗节奏
- `Stage01SampleContentBuilder.cs` 已与当前资产配置同步

### 工程结构改动

如果本次改动涉及：
- 场景
- 目录
- 包配置
- ProjectSettings

至少验证：
- 工程仍可打开
- 场景引用没有断裂
- 没有破坏现有目录约定
- 没有引入与当前阶段无关的复杂依赖

## 出现编译错误时的处理顺序

出现编译错误时，必须优先处理，顺序如下：

1. 当前改动直接导致的语法错误
2. 命名空间、引用、路径、类型名错误
3. 数据结构变更引发的连锁错误
4. 本次任务顺带暴露的中等严重问题
5. 与当前任务无关的历史次要问题

规则：
- 编译不过时，不继续堆新功能
- 不用临时脏修破坏架构边界来掩盖错误
- 如果错误暴露了文档与实现冲突，先修冲突再继续

## 遇到运行异常时的处理顺序

如果代码能编译但运行异常，优先顺序如下：

1. 先看战斗事件日志
2. 再看调试日志
3. 再确认输入对象和运行时实例是否正确
4. 再确认是不是状态系统、数值结算或技能释放条件出错
5. 最后才考虑表现层问题

不要先凭感觉改一堆逻辑。

## 提交前最低验收标准

每次准备交付一个阶段性改动前，至少确认：
- 可以编译
- 当前改动目标已经完成
- 对应最小验证路径已经走通
- 没有明显破坏第一阶段边界
- 没有绕开统一输入、结果、事件、状态、结算体系
- 新增目录和文件位置符合既定结构

## 可交付状态的判断标准

只有满足以下条件，才能认为一次改动达到“可交付”：
- 本次子目标明确完成
- 编译通过
- 最小验证通过
- 没有引入明显架构倒退
- 日志或结果足够解释当前行为

## 不该在实现阶段做的事

- 不要边写核心框架边大量扩英雄内容
- 不要先做高保真 UI 或美术表现
- 不要提前实现联机流程
- 不要把复杂特例当成基础系统替代品
- 不要在核心规则未稳定前频繁重构目录

## 当前最推荐的实际落地顺序

如果现在马上开始做工程，推荐顺序是：

1. 初始化 Unity 工程到 `game/`
2. 建立 `Assets/` 目录结构和基础场景
3. 写静态数据结构与 ScriptableObject
4. 写 BattleInput / BattleResult / BattleManager
5. 写 RuntimeHero 和基础单位循环
6. 写数值结算与状态系统
7. 写技能系统
8. 写职业 AI 模板
9. 创建 6 个模板英雄
10. 接最小 UI 通路和日志

## 给后续 AI 的简短执行指令

如果你现在在 `game/` 里真正写代码：
- 先做一件小事并完成验证
- 不要一次改太多层
- 编译错误优先于新功能
- 主循环优先于表现
- 统一系统优先于英雄特例
