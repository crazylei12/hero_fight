# 战斗人数模式扩展规划（3v3 / 4v4 / 5v5）

最后更新：2026-04-22

## 文档用途

这份文档用于整理项目后续接入 `3v3` 和 `4v4` 时，需要改哪些模块、优先怎么改，以及为什么“大招释放逻辑”会成为主要改动点。

它的目标不是把第一阶段直接改成多模式同步交付，而是：
- 保持第一阶段当前的 `5v5` 原型目标不变
- 提前把未来多人数模式的影响面写清楚
- 让后续编码 AI 不再继续把队伍人数、密度判断和大招门槛写死成 `5`

## 范围定位

- 第一阶段当前正式交付目标仍然是 `本地单机 5v5 自动战斗原型`
- `3v3 / 4v4` 视为后续必须支持的正式对战模式
- 在没有额外文档改动前，`3v3 / 4v4 / 5v5` 默认沿用同一套基础战斗规则：
  - `60 秒` 常规时间
  - 死亡后 `5 秒` 复活
  - 常规时间结束比较击杀数
  - 同分进入下一杀加时
- 当前判断基本成立：`战斗界面需要改的地方不算最多`；真正需要重点重构的是：
  - 队伍人数相关的输入和 HUD 槽位显示数量
  - 团战密度与人数阈值
  - 英雄大招释放条件与兜底逻辑

## 已确认结论

- 不应为 `3v3`、`4v4`、`5v5` 各自复制一套独立战斗系统。
- 应新增一层正式的 `模式配置`，由它决定：
  - 队伍人数
  - 需要显示的阵容槽位数
  - 大招释放阈值缩放规则
- 已确认 `3v3 / 4v4 / 5v5` 继续共用当前竞技场尺寸和出生布局规则，不额外拆 `arena preset / spawn preset`。
- 已确认战斗中的侧边栏继续沿用当前卡片位置，不做密度重排；多出来的槽位只是不加载、不渲染。
- 当前大量实现仍把 `5` 当成固定事实，至少已经影响到：
  - `game/Assets/Scripts/Data/BattleInputConfig.cs`
  - `game/Assets/Scripts/UI/Flow/GameFlowState.cs`
  - `game/Assets/Scripts/UI/Flow/HeroSelectSceneController.cs`
  - `game/Assets/Scripts/UI/BattleSideHeroSidebarHud.cs`
  - `game/Assets/Scripts/Battle/BattleSkillSystem.cs`
  - `game/Assets/Scripts/Editor/Stage01SampleContentBuilder.cs`
- `Stage01ArenaSpec` 和当前出生布局逻辑原则上可继续复用；只有当某些入口仍通过 `DefaultTeamSize` 反推出生点时，才需要做兼容修正，但这不是本轮主要改点。
- 当前大招模板的数据核心仍然偏向绝对人数门槛，例如：
  - `requiredUnitCount`
  - `targetPriorityRequiredUnitCount`
  - `minTargetsToCast`
  - `overrideRequiredUnitCount`
- 如果直接把这些绝对人数复制到 `3v3 / 4v4`，会出现两类问题：
  - `3v3` 下很多团战型大招会因为“等不到 3 人”而明显捏大
  - `4v4` / `3v3` 下某些保护型大招又可能因为局部密度更高、时间更短而过早交掉

## 需要改哪些

### 1. 模式配置层

必须新增一个正式的对战模式配置对象，建议命名方向：
- `BattleModeDefinition`
- 或 `BattleModePreset`

它至少应包含：
- `modeId`
- `displayName`
- `teamSize`
- `ultimateDecisionProfileId`

约束：
- `BattleInputConfig` 不应再只靠 `DefaultTeamSize = 5` 判断合法性
- 战斗输入应显式引用模式，或至少显式保存本局 `teamSize`
- 选人、HUD、日志都从模式配置读取人数，不各自猜
- 竞技场尺寸和出生布局当前不通过模式切换，继续复用现有规则

### 2. 竞技场与出生

这一块当前不作为主要改点。

已确认规则：
- `3v3 / 4v4 / 5v5` 都继续使用当前 `arena_stage01_flat`
- 地图逻辑尺寸继续保持 `48 x 27 世界单位`
- 出生点仍沿用当前“按实际参战人数自动分布在己方出生区”的规则，不为不同人数模式单独拆新布局

实际需要检查的点只有：
- 不要因为接入 `3v3 / 4v4` 就反向修改竞技场尺寸
- 不要默认为了小人数模式去重做出生层次、镜头和出生区
- 如果某些代码入口仍通过 `DefaultTeamSize = 5` 计算出生相关参数，只做兼容修正，让它读取本局真实队伍人数

一句话结论：
- 地图不变，出生规则不重做，只消除“固定按 5 人推导”的残留入口

### 3. 选人与战斗 HUD

这部分改动不算最重，但必须做成动态人数。

明确需要改的地方：
- `HeroSelect` 的左队 / 右队槽位数不能再固定为 `5`
- “All 10 slots must be filled” 这类文案要改成随模式变化
- `GameFlowState` 的默认选人列表长度不能再硬编码 `5`
- `BattleSideHeroSidebarHud` 只渲染当前模式需要的英雄卡，不渲染多余槽位

建议原则：
- 计时、比分、结果展示可以继续沿用现有大结构
- 战斗侧边栏继续沿用当前卡片位置和当前布局密度
- `3v3` 或 `4v4` 下，只保留前 `3` 或前 `4` 个实际槽位，其余位置不加载、不显示

### 4. AI 与战斗基础假设

接入多人数模式后，不只是 UI 数量变了，很多“局势判断”也会被影响。

需要改的地方包括：
- “敌方是否成团”的人数判断
- “友方是否危险”的低血人数判断
- “某个区域值不值得交大”的密度判断
- “当前局势是否进入残局”的存活人数判断

这里的关键结论是：
- `3v3` 的一波局部接战，本质上更容易接近“全队参战”
- `5v5` 的局部接战，则更容易出现“只打到 2~3 个前排”

所以同样是“命中 2 个敌人”：
- 在 `3v3` 里往往已经接近大窗口
- 在 `5v5` 里可能只是普通前排接触

## 大招释放逻辑要怎么改

### 1. 先明确哪些字段不能再只用绝对人数

以下字段都不能继续只按固定整数理解：
- `UltimateConditionData.requiredUnitCount`
- `SkillData.targetPriorityRequiredUnitCount`
- `SkillData.minTargetsToCast`
- `UltimateFallbackData.overrideRequiredUnitCount`
- `UltimateFallbackData.secondaryOverrideRequiredUnitCount`

此外，当前 `BattleSkillSystem` 里的这些常量也需要模式化：
- `UltimateInitialLockoutSeconds`
- `UltimateDecisionIntervalSeconds`
- `UltimateBaseReleaseChance`
- `UltimateExtraUnitReleaseChance`
- `UltimateAllySuppressionWindowSeconds`

原因：
- 人少时，绝对人数阈值更难满足
- 人少时，“多命中 1 人”的额外收益占比更高
- 人少时，同阵营错峰开大的压制窗口也不应继续和 `5v5` 完全一样长

### 2. 推荐把人数门槛改成“5v5 基线值 + 模式缩放规则”

不建议第一步就把所有英雄文档改成三套手写阈值。

更稳的做法是：
- 当前文档和技能资产里的 `requiredUnitCount` 继续视为 `5v5 基线值`
- 运行时再根据模式把它解析成真实门槛

推荐新增一层共享策略，含义类似：
- `countScalePolicy`
- `countReference`
- `roundingMode`

至少要区分：
- 参考 `敌方队伍人数` 还是 `友方队伍人数`
- 参考 `开局配置人数` 还是 `当前存活人数`
- `向上取整`、`四舍五入` 还是 `向下取整`

### 3. 不同大招类型的缩放规则不能完全一样

#### 进攻型团控 / 群伤大招

典型例子：
- 法师范围爆发
- 坦克范围团控
- 以敌方密集区为核心的进攻型大招

建议规则：
- 保持“只有拿到明显团战价值才开”
- 人数阈值按 `敌方队伍人数比例` 缩放
- 默认使用 `向上取整`

建议口径：
- `5v5` 基线要求 `3` 人
- `4v4` 一般仍要求 `3` 人
- `3v3` 改为要求 `2` 人

原因：
- 这类技能如果在 `3v3` 还坚持等 `3` 人，通常会明显过晚
- 但如果把 `4v4` 也直接降到 `2`，又会过早交大

#### 保护型群疗 / 群体保命大招

典型例子：
- 群体治疗
- 群体护盾
- 围绕受压友军的反打保护大招

建议规则：
- 不要继续只按“至少 2 名低血友军”做绝对判断
- 在 `3v3` 里，`1` 名关键友军进入危险线就可能已经值得开
- 这类阈值应按 `友方队伍人数比例` 缩放
- 默认使用 `四舍五入` 或 `向下取整但最低为 1`

建议口径：
- `5v5` 基线要求 `2` 名危险友军
- `4v4` 仍多半保持 `2`
- `3v3` 通常降到 `1`

#### 单体收割 / 单体斩杀大招

典型例子：
- 刺客收割大招
- 低血点杀型大招

建议规则：
- 目标数量门槛通常仍是 `1`
- 主要调整项不是人数，而是：
  - `fallback` 提前
  - 同阵营压制时间缩短
  - 低血线判定是否需要略微放宽

结论：
- 这类大招不是“人数变了就要完全改逻辑”
- 但仍要改它们的时间兜底和开大节奏

#### 自身狂暴 / 残血反打型大招

典型例子：
- 狂战士这类自增益大招

建议规则：
- `SelfLowHealth` 这类自血线条件基本保持不变
- 周围敌人数的“优先级提升条件”要按模式缩放
- `fallback` 时间要比 `5v5` 更早

建议口径：
- `5v5` 的“周围至少 2 敌人”通常可以继续保留到 `4v4`
- `3v3` 中若该条件只是“提高优先级”而非硬门槛，可允许降到 `1~2`

#### 全敌方 / 全友方范围大招

典型例子：
- 全体嘲讽
- 全体拉拽
- 全体禁疗

建议规则：
- 目标选择本身不需要重写，因为“全敌方 / 全友方”语义不变
- 真正要改的是“什么时候值得开”
- 这里更适合参考 `当前存活人数比例`，而不是开局固定人数

建议口径：
- “至少 3 名敌方存活再开”在 `3v3` 必须改成“至少 2 名敌方存活”
- “至少 2 名低血敌方”在 `3v3` 常可改为 `1`

### 4. 释放时机的时间参数也要跟人数一起缩放

仅仅改人数阈值还不够。

建议同步按模式缩放：
- `UltimateInitialLockoutSeconds`
  - `5v5` 保持当前 `6 秒`
  - `4v4` 可略降
  - `3v3` 应更早允许开始判定
- `fallback.triggerAfterSeconds`
  - `3v3` 应比 `5v5` 更早进入兜底
- `UltimateAllySuppressionWindowSeconds`
  - `3v3` 不应继续使用和 `5v5` 同样长的错峰压制

建议方向：
- `3v3` 比 `5v5` 提前 `5 ~ 10 秒` 进入第一段兜底
- `4v4` 比 `5v5` 提前 `3 ~ 5 秒`
- 同阵营开大压制窗口可按人数减短，避免 `3v3` 因为一个人刚开大，另一个本该跟开的英雄被压太久

### 5. 额外人数收益应从“多 1 人加多少”改成“超额占比加多少”

当前 `BattleSkillSystem` 里存在：
- `UltimateExtraUnitReleaseChance`

它隐含的是：
- 比最低门槛多命中 `1` 人，就多一档收益

这个逻辑在 `5v5` 还算自然，但到了 `3v3` 会出现：
- 可超额的人数空间过小
- 同样的“多 1 人”占比过大

更稳的做法是：
- 不再只看 `extraUnitCount`
- 改看 `overflowRatio`

也就是：
- 当前命中人数超出最低门槛的比例有多高
- 再把这段比例换算成额外开大意愿

## 对英雄文档怎么改

不建议立刻把所有英雄文档都展开成三套完整数值。

更合理的顺序是：
- 先在共享规则层把“多人数模式如何缩放大招条件”定下来
- 再给对人数最敏感的英雄补模式说明

优先需要补模式说明的英雄类型：
- 团控坦克
- 群伤法师
- 群疗 / 群护盾辅助
- 全敌方影响型刺客或坦克

当前已知会明显受影响的样例包括：
- `Tidehunter`
- `Shieldwarden`
- `Butcher`
- 法师群伤型大招
- 群疗 / 群保护型辅助大招

相对不需要大改的类型：
- 纯自增益反打型大招
- 纯单点收割型大招

但它们仍然要补：
- 更早的 `fallback`
- 更早的判定时机
- 更短的同阵营压制窗口

## 推荐实施顺序

1. 先新增 `BattleModeDefinition`，把队伍人数从输入、选人、出生、HUD 中抽出来。
2. 在暂时仍只跑 `5v5` 的前提下，把输入、选人、侧边栏和大招判定改成不再写死 `5`。
3. 重构大招决策，把绝对人数条件收口成“基线值 + 模式缩放规则”。
4. 先接入 `3v3`，验证团控、群伤、群疗、全敌方影响这四类大招。
5. 再接入 `4v4`，做中间档位调校。
6. 最后补具体英雄文档和 demo 阵容资源。

补充约束：
- 上述步骤里不包含重做竞技场尺寸
- 也不包含重排战斗侧边栏卡片密度

## 最低验收要求

当后续真正开始做 `3v3 / 4v4` 时，至少要确认：
- `HeroSelect` 能按模式显示正确数量的英雄槽位
- 战斗侧边栏在 `3v3 / 4v4` 下只显示有效英雄卡，且仍保持当前位置体系
- 继续使用当前地图尺寸和出生规则时，不会出现出生点重叠或明显异常空位问题
- `3v3` 中团控 / 群伤大招不会明显捏到太晚
- `3v3` 中保护型大招不会因为坚持等 `2` 名低血友军而经常错过窗口
- `4v4` 中大招释放时机处于 `3v3` 与 `5v5` 之间，不出现极端偏早或偏晚
- `5v5` 旧行为不因多人模式支持而回归变形

## 对后续 AI 的约束

后续 AI 在真正实现多人数模式时，必须遵守：
- 不要把 `3v3 / 4v4 / 5v5` 做成三套平行战斗代码
- 不要只改 UI 槽位数，而忽略大招门槛和战斗输入
- 不要继续在英雄大招里直接手写“至少 3 人”“至少 2 个低血目标”而不声明缩放规则
- 不要在还没抽出模式配置前，就开始逐个英雄手工特判 `3v3` 和 `4v4`
- 不要为了支持小人数模式，把当前 `5v5` 原型基线直接改坏
- 不要擅自把竞技场尺寸、出生布局规则或战斗侧边栏位置体系改成分模式版本
