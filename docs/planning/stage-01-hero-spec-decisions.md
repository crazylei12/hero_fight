# 第一阶段英雄开发规范决策

最后更新：2026-04-22

## 文档用途

这份文档记录第一阶段关于英雄定义、配置、扩展和文档约束的正式决策。后续 AI 在新增英雄、调整英雄、设计技能、扩展职业模板时，必须以本文件为直接依据。

如果后续规则变化，应直接修改本文件，而不是继续依赖问题清单中的原始回答。

## 第一阶段目标

第一阶段英雄系统的目标是：
- 建立统一、可扩展的英雄配置规范
- 支持首批 6 个职业模板英雄
- 为后续持续新增英雄提供稳定格式
- 避免英雄逻辑完全碎片化
- 允许少量特例，但必须被显式管理

## 英雄数据分层

第一阶段英雄数据固定拆成三层：
- `基础数据`
- `战斗逻辑配置`
- `表现资源配置`

### 基础数据

用于描述英雄的身份和战斗基础信息，例如：
- 英雄 ID
- 英雄名称
- 职业
- 基础属性
- 标签
- 设计说明或调试备注

### 战斗逻辑配置

用于描述英雄如何参与战斗，例如：
- 普攻配置
- 小技能配置
- 大招配置
- AI 行为模板
- 特例逻辑标记

### 表现资源配置

用于描述英雄的表现资源引用，例如：
- 头像
- 立绘或占位图
- 2D 战斗单位预制体或平面角色资源
- 如有必要，也允许临时引用 3D 占位预制体
- 动画资源
- 投射物资源
- 特效资源
- 音效资源

说明：
- 这三层必须明确分离。
- 不允许把所有信息长期堆进一个混乱的大对象里。
- 第一阶段默认资源方向应优先兼容 `Sprite / SpriteRenderer / 2D 角色 prefab`。
- 如果某个字段同时能挂 2D 与 3D 资源，规划意图仍然是 `2D 正式表现优先，3D 仅作临时调试占位`。

## 英雄配置内容范围

第一阶段每个英雄配置文件至少应包含以下内容：
- `英雄ID`
- `英雄名称`
- `职业`
- `基础属性`
- `普攻配置`
- `小技能配置`
- `大招配置`
- `AI 行为标签`
- `表现资源引用`
- `调试备注`

约束：
- 第一阶段只保留和战斗相关的字段。
- 第一阶段不要求加入剧情、背景故事、台词、稀有度等包装字段。

## 英雄内容归属固定规则

第一阶段后续新增英雄时，必须先按“谁拥有这部分内容”来决定放置位置，不允许边做边猜。

固定归属如下：
- `HeroDefinition` 只作为英雄总入口和装配点
- `HeroStatsData` 只负责英雄基础属性
- `BasicAttackData` 只负责普攻规则与普攻表现参数
- `SkillData` 只负责该技能自己的战斗参数、效果参数与技能表现参数
- `HeroVisualConfig` 只负责英雄本体级表现资源
- `RuntimeHero` 只负责战斗中的临时状态
- 表现层控制脚本只负责把事件和配置转成画面，不负责决定战斗结果

### HeroDefinition 的固定职责

`HeroDefinition` 必须是“英雄装配清单”，不是大杂烩逻辑脚本。

它可以拥有：
- 英雄 ID、名称、职业、标签
- `baseStats`
- `basicAttack`
- `activeSkill`
- `ultimateSkill`
- 职业或英雄 AI 模板 ID
- 特例逻辑标记与说明
- `visualConfig`

它不应长期拥有：
- 技能内部效果细节
- 技能专属特效参数
- 运行时血量、CD、目标、位置
- 复杂技能执行代码
- 技能命中特殊判定的硬编码实现

说明：
- 英雄换技能时，应只替换 `SkillData` 引用，而不是再回头修改英雄本体里一堆技能细节字段。
- `HeroDefinition` 可以引用技能，但不应吞掉技能自己的配置边界。

### HeroVisualConfig 的固定职责

`HeroVisualConfig` 只负责“这个英雄本体长什么样、用什么基础表现资源”，不负责承载所有技能表现。

它适合拥有：
- 英雄头像
- 英雄战斗 prefab
- 英雄 animator controller
- 英雄本体常驻挂件或默认材质
- 普攻投射物资源
- 英雄通用受击表现或死亡表现的资源引用
- 可选的通用施法/受击兜底资源

它不应长期拥有：
- 某个小技能专属范围特效
- 某个大招专属持续区域特效
- 某个技能独有的音效节奏参数
- 依赖单个技能 ID 才能成立的表现配置

规则：
- 只要某个表现资源是“换了技能就应跟着一起换”的，它就不属于 `HeroVisualConfig`，而属于对应 `SkillData` 或普攻配置。

### BasicAttackData 的固定职责

普攻相关内容必须尽量跟普攻走，不应散落到英雄本体或 UI 脚本里。

`BasicAttackData` 至少应负责：
- 普攻伤害倍率
- 普攻效果类型，例如伤害或治疗
- 普攻目标类型，例如敌方或友方
- 攻速换算所需参数
- 普攻射程覆盖
- 是否投射物
- 投射物速度
- 普攻命中附带效果列表，例如附加状态、属性 Debuff 或其他共享 on-hit 载荷
- 普攻命中表现需要的参数或资源引用

说明：
- 如果未来某个英雄更换武器型普攻，优先改 `BasicAttackData` 或其关联表现配置，不要把变化写死在 `BattleView`。
- 第一阶段默认敌方普攻仍以造成伤害为主，但系统必须允许少量英雄把普攻目标切到友方，并把结算改为治疗。
- 对敌方普攻，`BasicAttackData.targetType` 除了默认最近敌人之外，也应允许显式声明“沿用共享职业偏好的敌方目标”，避免把刺客切后排、射手偏好收残等逻辑偷偷散写进个别英雄脚本。
- 如果某个英雄的普攻只是“普通伤害 + 命中附带状态”，这仍应优先通过 `BasicAttackData` 配置表达，不要额外拆成伪被动技能或单英雄脚本。

### SkillData 的固定职责

技能自己的东西，必须尽量都跟技能走。

`SkillData` 应负责：
- 技能 ID、名称、描述
- 技能类型、激活模式、目标类型、释放距离、范围、CD（若为主动技能）
- 目标优先过滤、威胁半径、回退目标等共享选目标参数
- 技能效果列表
- 被动监听条件或动态收益参数，例如当前生命值比例分段、缺血系数、命中事件等共享入口
- 技能附着型短时保护载荷，例如“挂在友军身上的一次性近身反制”
- 技能执行期间的正式动作覆写参数，例如临时射程、执行间隔、重复次数、伤害转治疗比例
- 技能大招决策模板
- 技能表现资源引用
- 技能表现所需的正式参数入口

技能表现资源引用至少应优先放在技能侧，而不是英雄侧，例如：
- 施法特效
- 命中特效
- 持续区域特效
- 技能专属投射物
- 技能音效

规则：
- 只要某个表现资源与某个技能强绑定，并且英雄换技能时它也应一起更换，这部分就必须放到 `SkillData` 或技能专属表现配置中。
- 技能的范围、持续时间、脉冲频率等正式参数，必须由技能配置或技能效果配置提供，不允许长期在表现层另存一套。
- 如果某个技能会在执行期间临时改变单位的普攻射程、施法射程、动作间隔或重复执行同一个动作，这些正式参数也必须跟技能走，不允许散写在 AI、表现层或单英雄脚本里。
- 持续区域类技能应优先拆成“创建持续区域”与“区域脉冲效果载荷”的组合，不要为范围治疗、范围 Buff、范围 Debuff 各自新增一组高度耦合的专用效果类型。
- 若某个保护技能需要“短时间附着在友军身上，敌人贴近时触发一次击飞/击退”，应优先作为 `SkillData` 下的共享运行时载荷表达，而不是把近身检测长期散写进辅助英雄 AI、表现脚本或手工 Update 轮询里。

### RuntimeHero 的固定职责

运行时状态只能放在 `RuntimeHero` 或其他运行时对象中。

至少包括：
- 当前生命值
- 当前目标
- 当前位置
- 技能冷却
- 死亡/复活状态
- 当前状态效果
- 对局内统计

规则：
- 不允许把这些运行时状态回写到 `HeroDefinition`。
- 不允许让表现层脚本自己偷偷维护一份战斗真状态作为长期方案。

## 英雄目录与资源放置规则

第一阶段后续新增英雄时，目录也要固定，不允许“能放哪就放哪”。

推荐固定结构如下：
- `Assets/Data/Heroes/<heroId>/`
- `Assets/Data/Skills/<heroId>/`
- `Assets/Prefabs/Heroes/<heroId>/`
- `Assets/Prefabs/Projectiles/<heroId>/`
- `Assets/Prefabs/VFX/Skills/<skillId>/`
- `Assets/Art/Heroes/<heroId>/`
- `Assets/Audio/Skills/<skillId>/`

放置规则：
- 英雄总配置资产放在 `Assets/Data/Heroes/<heroId>/`
- 该英雄默认使用的技能资产放在 `Assets/Data/Skills/<heroId>/`
- 英雄本体 prefab、角色展示资源放在 `Assets/Prefabs/Heroes/<heroId>/` 或 `Assets/Art/Heroes/<heroId>/`
- 普攻投射物 prefab 放在 `Assets/Prefabs/Projectiles/<heroId>/`
- 技能专属特效 prefab 放在 `Assets/Prefabs/VFX/Skills/<skillId>/`
- 技能专属音效放在 `Assets/Audio/Skills/<skillId>/`

补充规则：
- 同一个技能如果未来要给多个英雄复用，仍然以 `skillId` 为主目录，不转回英雄目录。
- 同一个英雄的“本体资源”与“技能资源”要分开，不要全部塞进一个英雄文件夹里。
- 第一阶段示例资源可以暂时简化命名，但后续新增正式英雄时应尽量按以上目录收束。

## 复杂表现特例的固定落点

复杂表现特例允许存在，但必须有固定落点，不能继续无限堆进 `BattleView` 这一类总视图脚本。

规则如下：
- `BattleView` 只负责通用视图同步、事件订阅和表现分发
- 通用技能特效优先走技能配置直接实例化
- 复杂技能表现应放到单独的表现控制脚本中
- 如果某个英雄的大招需要单独动画编排、分段特效或自定义时间轴，应建立独立脚本，不要把完整实现直接塞进总视图文件

推荐落点：
- `Assets/Scripts/UI/Presentation/Skills/`
- 或 `Assets/Scripts/UI/Presentation/Heroes/<heroId>/`

示例约束：
- 不应长期出现一串 `if skillId == xxx` 的大段特效实现都写在 `BattleView`
- 如果某个技能需要特殊表现，优先写成 `SkillPresentationController`、`MeteorPresentationController` 这类单独脚本
- `BattleView` 最多只保留“选择哪个控制器/哪个 prefab”的分发层职责

一句话规则：
- 英雄本体的东西跟英雄走
- 普攻的东西跟普攻走
- 技能的东西跟技能走
- 运行时状态只放运行时
- 复杂表现特例必须拆出总视图脚本

## 英雄 ID 规则

第一阶段英雄 ID 规则固定为：
- `职业前缀 + 三位序号 + 英文名字`

示例格式：
- `warrior_001_skybreaker`
- `mage_001_firemage`
- `assassin_001_shadowstep`
- `tank_001_ironwall`
- `support_001_sunpriest`
- `marksman_001_longshot`

约束：
- ID 必须稳定、唯一。
- 不允许后续随意改已有英雄 ID。
- 英雄英文名部分应简短、可读、可用于资源命名。

## 职业枚举

第一阶段代码中直接固定六类职业枚举：
- 战士
- 法师
- 刺客
- 坦克
- 辅助
- 射手

说明：
- 第一阶段不采用开放字符串分类。
- 如果未来真的要扩新职业，再升级数据结构和枚举体系。

## 六个职业的第一阶段基础定位

### 战士

- 近战或近距离持续作战
- 输出和生存相对均衡
- 可以带控制能力
- 如果带控制，优先是 `单点控制`

### 法师

- 相对脆弱
- 依赖技能打出高爆发或范围伤害
- 大招通常具备明显范围压制能力

### 刺客

- 高机动
- 优先威胁敌方后排脆皮
- 强调切入、突进、收割

### 坦克

- 前排承伤
- 限制敌方推进
- 通常具备明确控制能力，或稳定的团队保护 / 承伤能力
- 控制或保护收益往往更强，更偏向正面稳团

### 辅助

- 负责治疗、增益、保护或控制
- 也允许具备较强、范围较大的控制效果
- 不以高额爆发输出为主要职责

### 射手

- 远程持续输出
- 对攻速、暴击、输出位置更敏感
- 依赖稳定站位持续打伤害

## 默认职业 AI 模板

第一阶段要求每个职业至少有一个默认 AI 行为模板。

说明：
- 默认模板用于形成统一职业行为基线。
- 具体英雄可以在模板基础上扩展或覆盖。
- 这不是要求所有英雄完全相同，而是要求先有继承起点。

职业模板方向示例：
- 战士：压前排，优先接敌，持续输出
- 法师：保持中后排，找敌方密集区释放技能
- 刺客：寻找切入路径，优先后排脆皮
- 坦克：顶在前排，优先接敌并替队友承压；按英雄配置打出控制或保护技能
- 辅助：靠近友军，根据局势治疗、增益或控制
- 射手：维持安全输出位置，持续攻击可攻击目标

### 投射物普攻英雄的默认受压后撤基线

- 这是一条按 `普攻形态` 共享的 AI 基线，不只属于某个职业。
- 默认覆盖射手、法师，以及使用投射物普攻进行输出或治疗的辅助。
- 当英雄的普攻配置为 `usesProjectile = true` 时，应继承“被危险 threat 压近时短时后撤”的默认行为。
- 这个默认行为的目标是维持更安全的输出或治疗位置，而不是把单位做成长期逃跑或完整走 A 状态机。
- 正常远程互点不应触发机械式双后撤；只有近战 / 短手威胁，或已经压进异常近距离的远程 threat，才应触发。

## 特例逻辑规则

第一阶段允许少量特例逻辑，但必须显式标记。

规则如下：
- 默认必须先尝试用统一技能系统、统一目标系统、统一状态系统实现。
- 只有在统一系统无法表达时，才允许补充自定义逻辑。
- 所有特例逻辑必须在配置中显式标注。
- 所有特例逻辑必须在对应英雄文档中说明用途和原因。

说明：
- 不允许无节制地为每个英雄各写一套完全独立脚本。
- 允许特例，是为了避免统一系统刚起步时过度僵化。

## 技能配置模板化

第一阶段技能配置必须模板化。

每个技能都应走统一技能配置结构，而不是每个技能拥有完全不同的数据形态。

建议统一字段至少包含：
- `技能ID`
- `技能名称`
- `技能类型`
- `激活模式`
- `目标类型`
- `目标优先过滤 / 回退参数`
- `释放条件或被动监听条件`
- `射程`
- `范围半径`
- `伤害或治疗倍率`
- `冷却时间（若为主动技能）`
- `动作覆写参数`
- `状态效果列表`
- `动画/特效资源引用`

说明：
- 模板化是后续批量扩英雄的前提。
- 即使存在少量特例，也必须建立在统一技能模板之外的补充层，而不是替代模板本身。

### 被动型小技能能力

第一阶段技能系统允许极少量英雄把唯一的小技能位定义为 `被动型小技能`，用于表达常驻监听或条件型自我增益。

这套能力至少应能表达：
- 常驻生效且无主动施放动作
- 监听施法者当前生命值比例等共享运行时条件
- 监听最近一段时间内对施法者造成直接伤害的不同敌方来源数量
- 按分段阈值或线性系数持续提供属性修正
- 在死亡、复活和生命值变化后重新按当前状态即时结算

约束：
- 被动型小技能仍占用唯一的小技能位，不额外新增独立被动栏
- 如果收益本质只是“普攻命中附带效果”，仍应优先走 `BasicAttackData`
- 如果收益本质是正式属性变化、伤害转治疗或其他共享动作覆写，应继续走统一数值结算与日志机制，不要并行维护私有结算入口
- 第一阶段优先把它收敛在“自身条件驱动的属性变化 / 短时动作覆写”范围内，不扩成完整天赋树或复杂符文系统

### 连续动作序列能力

第一阶段技能系统应预留通用的“连续动作序列”能力，不只服务某个射手大招。

这套能力至少应能表达：
- 连续普攻
- 连续多次释放同一个技能
- 在固定次数或固定持续时间内重复执行同一个标准动作

建议纳入统一配置或统一运行时契约的最小字段包括：
- 动作载荷来源，例如 `普攻`、`当前技能` 或后续可扩展的标准动作
- 重复次数或持续时间
- 动作间隔
- 序列期间的临时参数覆写，例如临时射程
- 当前目标失效时的刷新目标策略
- 可被哪些控制或中断条件打断

约束：
- 这套能力的目标是成为共享执行入口，而不是每个英雄各写一套循环逻辑。
- 如果第一阶段暂时只落地一个样例，也应按“通用序列能力的第一个实例”来实现，不要直接写成射手专属分支。

### 径向扫波 / 延迟环带命中能力

第一阶段技能系统应允许少量技能以“记录一个中心点，再按距离分段延迟命中”的方式表达向外扩散或向内回卷的波纹、海浪或冲击圈。

这套能力至少应能表达：
- 中心点来源，例如 `施法者当前位置快照`、`指定目标点`
- 最大作用半径
- 波锋厚度或每层环带宽度
- 外放阶段持续时间
- 可选的外圈停顿
- 回卷阶段持续时间
- 外放与回卷各自的效果载荷
- 同一目标在每个阶段的命中上限

约束：
- 逻辑层应优先把它实现成“按时间顺序结算多层环带命中”的共享入口，而不是依赖物理碰撞体实时扫描的单英雄专属圆环脚本。
- 默认应以施法瞬间记录的中心点为整个序列的逻辑锚点，不要求实时跟随施法者后续移动；如果某个技能确实要跟随，必须在对应英雄文档中显式说明。
- 每段命中的伤害、治疗、状态和日志仍应继续复用统一结算、统一状态系统和统一事件机制，不要新增“海啸命中”“波纹状态”这类只服务单英雄的新底层。
- 如果第一阶段暂时只有一个样例，也应按共享的“径向扫波能力”落地，不要直接写成单英雄专属分支。

### 部署型固定战斗代理体能力

第一阶段技能系统应允许少量技能创建 `部署型固定战斗代理体`，用于表达沙兵、固定炮台、阵地触发点这类“留在场上、但不是英雄本体”的战场锚点。

这套能力至少应能表达：
- 生成锚点模式，例如 `指定位置`、`目标附近`、`施法者附近`
- 持续时间
- 同时存在上限
- 超上限时是否按最旧实例替换
- 是否可移动、是否可被选中、是否可受伤
- 作用半径或联动半径
- 代理体的动作模式，例如 `定时脉冲`、`跟随所有者普攻触发`、`跟随当前技能触发`

约束：
- 这类正式参数必须跟 `SkillData` 或对应技能效果配置走，不允许长期散写在单英雄脚本或表现层里。
- 如果某个代理体只是“在固定区域替所有者补打一段普攻 / 技能命中”，应优先走共享的 `owner-driven proxy strike` 语义，不要直接膨胀成独立 `RuntimeHero + 独立 AI`。
- 代理体造成的伤害、治疗、状态和日志，仍应继续复用统一结算、统一状态系统和统一事件机制，不要再为部署物并行维护第二套战斗流程。
- 如果某个代理体按设计不作为英雄单位，必须在共享规则里显式声明它不参与默认索敌、threat 后撤、最小间距、body blocking 和计分语义，而不是把这些排除条件散写成单英雄特例。

### 技能按已有状态层数结算与击杀后触发

第一阶段技能系统应允许在统一模板内表达“读取目标已有正式状态，再决定额外收益”的能力。

这套能力至少应能表达：
- 读取目标当前某类正式状态的有效层数或等价累计值
- 按该层数为技能追加伤害、治疗、护盾或其他正式收益
- 仅对带有指定状态的目标生效，而不是无条件命中所有目标
- 目标因指定技能或指定效果死亡后，在死亡位置追加一次范围效果

约束：
- 这类状态读取必须优先通过统一状态系统查询，不要在英雄脚本里长期并行维护第二套“私有层数计数器”
- 被读取的状态应按配置指定 `状态资产 / 主题键 / 来源技能组`，不要把所有同大类状态混算；例如“中毒结算”不应默认把燃烧、流血也一起算进去
- `主题键` 的职责是表达“这些正式状态能否被同一类技能统一读取”；它不应自动把不同来源的实例并进同一个叠层池
- 默认叠层语义仍按“同一来源、同一技能或同一普攻 on-hit 载荷”分别计数；如果要让同一来源内多个施毒入口共用一组层数上限，应通过显式的局部叠层池配置表达，而不是靠主题键偷偷合并
- 默认语义应是“读取但不消耗层数”；如果某个技能要清空、减少或转化层数，必须在配置和文档里显式声明
- “击杀后触发范围效果”应优先以 `死亡位置快照` 作为范围锚点，不要求先建立可长期存在的尸体对象
- 如果同一次技能结算会先对一批初始目标造成本体伤害，再由击杀触发后续范围效果，则应先让这批初始目标各自完成一次本体结算，再处理由这些击杀产生的后续效果；前一个目标触发的尸爆或等价连锁，不应让后一个初始目标跳过自己的本体结算
- 击杀后的后续效果仍应复用统一的范围伤害、范围附状态、持续区域或共享事件载荷，不要把连锁收割逻辑直接写成某个英雄专属的大段脚本

### 技能特效参数绑定规则

第一阶段技能特效必须与技能配置参数绑定，不允许长期独立维护一套仅供表现层使用的“假参数”。

至少遵守以下规则：
- `范围型特效` 的可见范围应绑定技能的 `范围半径` 或对应效果的 `radiusOverride`
- `持续型特效` 的存在时长应绑定技能效果的 `durationSeconds`
- `周期触发型特效` 的重复播放频率应绑定技能效果的 `tickIntervalSeconds`
- `投射物表现` 的飞行速度、起点与命中目标应绑定技能或普攻的投射物参数

约束：
- 不允许为了“先看起来差不多”而长期手写一套与技能参数脱节的特效缩放、持续时间或播放频率
- 如果某个特效资源本身尺寸与技能参数不一致，应通过统一换算规则适配，不要为单个英雄散写魔法数字
- 表现资源配置可以决定“用哪个特效”，但技能配置负责决定“这个特效按多大范围、持续多久、多久触发一次”

## 普攻配置模板化

第一阶段普攻也必须配置化，不允许长期写死在代码里。

建议普攻统一配置至少包含：
- `伤害倍率`
- `攻击间隔` 或攻速换算参数
- `射程`
- `是否投射物`
- `投射物速度`
- `命中附带效果列表`
- `命中特效引用`

说明：
- 这样才能稳定支持近战、远程、投射物、暴击表现，以及“普攻附加减攻 / 减防 / DOT / HOT”等共享扩展。

## 首批 6 个英雄的定位

第一阶段首批 6 个英雄直接按职业原型做“模板英雄”。

建议模板方向：
- 战士：近战持续输出型
- 法师：范围爆发型
- 刺客：切后排突进型
- 坦克：前排承伤控制 / 保护型
- 辅助：治疗增益或保护控制型
- 射手：远程持续输出型

说明：
- 第一批英雄不追求复杂主题包装。
- 第一批英雄的主要职责是验证框架、职业差异和技能系统。

## 英雄命名风格

第一阶段英雄命名采用：
- `职业原型名`
- 或 `开发代号`

说明：
- 第一阶段不以世界观包装为优先。
- 原型名和代号更利于快速开发和沟通。

## 标签系统

第一阶段需要简单英雄标签系统。

标签用于职业之外的补充描述，例如：
- `近战`
- `远程`
- `爆发`
- `持续输出`
- `治疗`
- `突进`
- `控制`

约束：
- 第一阶段标签系统保持简单。
- 可用字符串数组或枚举数组实现。
- 标签是补充分类，不替代职业字段。

## 英雄说明文档要求

第一阶段要求每个英雄都要有一份说明文档。

文档用途：
- 说明英雄定位
- 说明技能意图
- 说明目标职业模板
- 说明设计边界
- 说明是否存在特例逻辑

说明：
- 这会增加一点前期文档成本。
- 但对于 AI 协作开发，这能显著降低后续理解偏差。

### 英雄说明文档目录规则

英雄具体设定文档统一放在：

- `docs/heroes/<class>/<english-name>.md`

职业目录固定为：

- `warrior`
- `mage`
- `assassin`
- `tank`
- `support`
- `marksman`

说明：
- `docs/planning/` 继续承载规则、阶段决策和实现约束
- `docs/heroes/` 只承载单个英雄的具体设定、技能说明和设计边界
- 每份英雄文档至少应包含：英文名、中文名、英雄分类、普攻类型、小技能、大招、设计边界

## 设计边界要求

第一阶段必须为每个英雄定义设计边界，防止设计失控。

示例：
- 坦克不做超高全屏输出
- 辅助不做超高爆发核心输出
- 刺客不做超远程站桩输出
- 射手不做超大范围强控核心角色

说明：
- 设计边界是控制首批英雄复杂度和职业辨识度的重要工具。
- 每个英雄说明文档里都应明确记录“该英雄不做什么”。

## 首批英雄覆盖的系统特性

首批 6 个英雄不只是职业各来一个，还必须故意覆盖不同系统能力，确保战斗框架真正被验证到。

首批英雄整体至少要覆盖：
- 近战普攻
- 远程投射物普攻
- 范围伤害
- 位移
- 治疗或增益
- 眩晕控制
- 大招范围判断

说明：
- 如果首批英雄没有覆盖这些能力，系统即使看似完成，也仍然未经充分验证。

## 对后续 AI 的约束

后续 AI 在新增英雄或调整英雄时，必须遵守以下规则：
- 优先复用统一英雄结构、统一普攻模板、统一技能模板、统一状态系统。
- 没有充分理由时，不要新增完全特例化实现。
- 所有特例都必须标记，并写入英雄说明文档。
- 新英雄必须归属六类职业之一。
- 新英雄必须包含稳定 ID、标签和说明文档。
- 第一阶段新增英雄时，应优先补齐系统覆盖面，而不是盲目堆数量。
- 如果新增英雄依赖“普攻命中附带效果”，优先扩 `BasicAttackData` 的共享配置能力，不要把这类逻辑散写到 `RuntimeHero`、AI 或单英雄被动脚本里。
