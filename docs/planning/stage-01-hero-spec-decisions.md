# 第一阶段英雄开发规范决策

最后更新：2026-04-30

## 文档用途

这份文档记录第一阶段关于英雄定义、配置、扩展和文档约束的正式决策。后续 AI 在新增英雄、调整英雄、设计技能、扩展职业模板时，必须以本文件为直接依据。

如果后续规则变化，应直接修改本文件，而不是继续依赖问题清单中的原始回答。

## 第一阶段目标

第一阶段英雄系统的目标是：
- 建立统一、可扩展的英雄配置规范
- 支持首批 6 个职业模板英雄
- 为后续持续新增英雄提供稳定格式
- 避免英雄逻辑完全碎片化
- 允许少量特例，但必须被显式管理

## 英雄数据分层

第一阶段英雄数据固定拆成三层：
- `基础数据`
- `战斗逻辑配置`
- `表现资源配置`

### 基础数据

用于描述英雄的身份和战斗基础信息，例如：
- 英雄 ID
- 英雄名称
- 职业
- 基础属性
- 标签
- 设计说明或调试备注

### 战斗逻辑配置

用于描述英雄如何参与战斗，例如：
- 普攻配置
- 小技能配置
- 大招配置
- AI 行为模板
- 特例逻辑标记

### 表现资源配置

用于描述英雄的表现资源引用，例如：
- 头像
- 立绘或占位图
- 2D 战斗单位预制体或平面角色资源
- 如有必要，也允许临时引用 3D 占位预制体
- 动画资源
- 投射物资源
- 特效资源
- 音效资源

说明：
- 这三层必须明确分离。
- 不允许把所有信息长期堆进一个混乱的大对象里。
- 第一阶段默认资源方向应优先兼容 `Sprite / SpriteRenderer / 2D 角色 prefab`。
- 如果某个字段同时能挂 2D 与 3D 资源，规划意图仍然是 `2D 正式表现优先，3D 仅作临时调试占位`。

## 英雄配置内容范围

第一阶段每个英雄配置文件至少应包含以下内容：
- `英雄ID`
- `英雄名称`
- `职业`
- `基础属性`
- `普攻配置`
- `小技能配置`
- `大招配置`
- `AI 行为标签`
- `表现资源引用`
- `调试备注`

约束：
- 第一阶段只保留和战斗相关的字段。
- 第一阶段不要求加入剧情、背景故事、台词、稀有度等包装字段。

## 英雄内容归属固定规则

第一阶段后续新增英雄时，必须先按“谁拥有这部分内容”来决定放置位置，不允许边做边猜。

固定归属如下：
- `HeroDefinition` 只作为英雄总入口和装配点
- `HeroStatsData` 只负责英雄基础属性
- `BasicAttackData` 只负责普攻规则与普攻表现参数
- `SkillData` 只负责该技能自己的战斗参数、效果参数与技能表现参数
- `HeroVisualConfig` 只负责英雄本体级表现资源
- `RuntimeHero` 只负责战斗中的临时状态
- 表现层控制脚本只负责把事件和配置转成画面，不负责决定战斗结果

### HeroDefinition 的固定职责

`HeroDefinition` 必须是“英雄装配清单”，不是大杂烩逻辑脚本。

它可以拥有：
- 英雄 ID、名称、职业、标签
- `baseStats`
- `basicAttack`
- `activeSkill`
- `ultimateSkill`
- 职业或英雄 AI 模板 ID
- 特例逻辑标记与说明
- `visualConfig`

它不应长期拥有：
- 技能内部效果细节
- 技能专属特效参数
- 运行时血量、CD、目标、位置
- 复杂技能执行代码
- 技能命中特殊判定的硬编码实现

说明：
- 英雄换技能时，应只替换 `SkillData` 引用，而不是再回头修改英雄本体里一堆技能细节字段。
- `HeroDefinition` 可以引用技能，但不应吞掉技能自己的配置边界。

### HeroVisualConfig 的固定职责

`HeroVisualConfig` 只负责“这个英雄本体长什么样、用什么基础表现资源”，不负责承载所有技能表现。

它适合拥有：
- 英雄头像
- 英雄战斗 prefab
- 英雄 animator controller
- 英雄本体常驻挂件或默认材质
- 普攻投射物资源
- 英雄通用受击表现或死亡表现的资源引用
- 可选的通用施法/受击兜底资源

它不应长期拥有：
- 某个小技能专属范围特效
- 某个大招专属持续区域特效
- 某个技能独有的音效节奏参数
- 依赖单个技能 ID 才能成立的表现配置

规则：
- 只要某个表现资源是“换了技能就应跟着一起换”的，它就不属于 `HeroVisualConfig`，而属于对应 `SkillData` 或普攻配置。

### BasicAttackData 的固定职责

普攻相关内容必须尽量跟普攻走，不应散落到英雄本体或 UI 脚本里。

`BasicAttackData` 至少应负责：
- 普攻伤害倍率
- 普攻效果类型，例如伤害或治疗
- 普攻目标类型，例如敌方或友方
- 可选的固定顺序普攻子段列表；每个子段可独立声明效果类型、目标类型、目标策略、倍率、命中附带效果与表现标记
- 可选的连锁弹射参数，例如额外弹射次数、每跳搜索半径、每跳倍率、同一链内唯一命中规则与弹射表现标记
- 可选的同目标持续普攻叠层参数，例如绑定目标、每次命中加层、层数上限、层数收益和清层条件
- 攻速换算所需参数
- 普攻射程覆盖
- 是否投射物
- 投射物速度
- 普攻命中附带效果列表，例如附加状态、属性 Debuff 或其他共享 on-hit 载荷
- 普攻命中表现需要的参数、资源引用或子段表现标记

说明：
- 如果未来某个英雄更换武器型普攻，优先改 `BasicAttackData` 或其关联表现配置，不要把变化写死在 `BattleView`。
- 第一阶段默认敌方普攻仍以造成伤害为主，但系统必须允许少量英雄把普攻目标切到友方，并把结算改为治疗。
- 对少量英雄，`BasicAttackData` 应允许按固定顺序在多个正式普攻子段间轮转，例如上一发对敌造成伤害、下一发对友造成治疗；当前轮转序号属于运行时状态，不回写静态配置。
- 若轮转型普攻需要在出生或复活后回到固定起手顺序，这条重置规则也应作为正式配置能力表达，不要散写在单英雄脚本里偷偷维护。
- 若轮转型普攻需要在“当前子段目标阵营已完全没有合法目标”时回退到另一个正式子段继续出手，例如敌方全灭等待复活时允许从伤害段回退到治疗段继续奶人，这条回退语义也应作为 `BasicAttackData` 的正式配置能力表达，而不是让 AI 或表现层私下决定“这次先别等了，临时换弹”。
- 对敌方普攻，`BasicAttackData.targetType` 当前默认收敛为 `最近敌人`；若未来真要新增新的敌方普攻模板，应作为新的明确规则补文档和共享实现，不要把这类逻辑偷偷散写进个别英雄脚本。
- 如果某个英雄的普攻只是“普通伤害 + 命中附带状态”，这仍应优先通过 `BasicAttackData` 配置表达，不要额外拆成伪被动技能或单英雄脚本。
- 若某个英雄的普攻需要命中后继续向附近目标弹射，这也应优先通过 `BasicAttackData` 的正式链式能力表达；每跳命中仍应继续复用统一伤害、状态和日志入口，不要在单英雄脚本里私下补一套“伪技能命中”。
- 若某个英雄的普攻需要“持续攻击同一个目标时叠加收益，切换目标后清零”，也应优先作为 `BasicAttackData` 的正式同目标叠层能力表达；目标绑定、清层条件、层数收益和大招期间的临时满层覆写都应有显式数据，不要让单英雄脚本私下维护一份隐藏目标和隐藏层数。
- 若某个英雄需要两套普攻投射物、命中特效或音效，优先通过普攻子段上的显式 `variantKey` 或等价表现标记表达，让表现层直接区分“当前是哪一种普攻”，不要再靠 `heroId` 或运行时猜测补分支。
- 若连锁弹射需要首段、弹射段或不同跳数使用不同拖尾、音效或命中火花，也应继续通过显式的普攻表现标记、跳数或等价事件参数表达，不要让 `BattleView` 再靠 `heroId` 猜“这次是不是弹射”。
- 若友方治疗普攻需要优先补齐某个 `on-hit` 正向状态的覆盖，再回退到该状态剩余持续时间最短的友军，这条覆盖排序也应作为 `BasicAttackData` 可驱动的共享目标策略表达，不要在单英雄脚本里私下维护“谁身上还没这层 Buff”的专属名单。

### SkillData 的固定职责

技能自己的东西，必须尽量都跟技能走。

`SkillData` 应负责：
- 技能 ID、名称、描述
- 技能类型、激活模式、目标类型、释放距离、范围、CD（若为主动技能）
- 目标优先过滤、威胁半径、回退目标等共享选目标参数
- 技能效果列表
- 被动监听条件或动态收益参数，例如当前生命值比例分段、缺血系数、命中事件等共享入口
- 技能附着型短时保护载荷，例如“挂在友军身上的一次性近身反制”
- 技能执行期间的正式动作覆写参数，例如临时射程、执行间隔、重复次数、伤害转治疗比例
- 技能大招决策模板
- 技能表现资源引用
- 技能表现所需的正式参数入口

技能表现资源引用至少应优先放在技能侧，而不是英雄侧，例如：
- 施法特效
- 命中特效
- 持续区域特效
- 技能专属投射物
- 技能音效

规则：
- 只要某个表现资源与某个技能强绑定，并且英雄换技能时它也应一起更换，这部分就必须放到 `SkillData` 或技能专属表现配置中。
- 技能的范围、持续时间、脉冲频率等正式参数，必须由技能配置或技能效果配置提供，不允许长期在表现层另存一套。
- 如果某个技能会在执行期间临时改变单位的普攻射程、施法射程、动作间隔或重复执行同一个动作，这些正式参数也必须跟技能走，不允许散写在 AI、表现层或单英雄脚本里。
- 持续区域类技能应优先拆成“创建持续区域”与“区域脉冲效果载荷”的组合，不要为范围治疗、范围 Buff、范围 Debuff 各自新增一组高度耦合的专用效果类型。
- 若某个保护技能需要“短时间附着在友军身上，敌人贴近时触发一次击飞/击退”，应优先作为 `SkillData` 下的共享运行时载荷表达，而不是把近身检测长期散写进辅助英雄 AI、表现脚本或手工 Update 轮询里。
- 若某个技能需要在开始施法时先从多个正式变体里选中一个结果，例如随机菜品、随机弹种或随机仪式形态，那么这些变体的 `目标类型`、`回退目标` 与 `效果列表` 也应继续归属在 `SkillData` 下，并以显式 `variantKey` 或等价运行时对象继续传给事件、日志和命中结算，不要等表现层看到技能飞出去后再倒推本次到底是哪一种。

### RuntimeHero 的固定职责

运行时状态只能放在 `RuntimeHero` 或其他运行时对象中。

至少包括：
- 当前生命值
- 当前目标
- 当前位置
- 技能冷却
- 死亡/复活状态
- 当前状态效果
- 对局内统计

规则：
- 不允许把这些运行时状态回写到 `HeroDefinition`。
- 不允许让表现层脚本自己偷偷维护一份战斗真状态作为长期方案。

## 英雄目录与资源放置规则

第一阶段后续新增英雄时，目录也要固定，不允许“能放哪就放哪”。

推荐固定结构如下：
- `Assets/Data/Heroes/<heroId>/`
- `Assets/Data/Skills/<heroId>/`
- `Assets/Prefabs/Heroes/<heroId>/`
- `Assets/Prefabs/Projectiles/<heroId>/`
- `Assets/Prefabs/VFX/Skills/<skillId>/`
- `Assets/Art/Heroes/<heroId>/`
- `Assets/Audio/Skills/<skillId>/`

放置规则：
- 英雄总配置资产放在 `Assets/Data/Heroes/<heroId>/`
- 该英雄默认使用的技能资产放在 `Assets/Data/Skills/<heroId>/`
- 英雄本体 prefab、角色展示资源放在 `Assets/Prefabs/Heroes/<heroId>/` 或 `Assets/Art/Heroes/<heroId>/`
- 普攻投射物 prefab 放在 `Assets/Prefabs/Projectiles/<heroId>/`
- 技能专属特效 prefab 放在 `Assets/Prefabs/VFX/Skills/<skillId>/`
- 技能专属音效放在 `Assets/Audio/Skills/<skillId>/`

补充规则：
- 同一个技能如果未来要给多个英雄复用，仍然以 `skillId` 为主目录，不转回英雄目录。
- 同一个英雄的“本体资源”与“技能资源”要分开，不要全部塞进一个英雄文件夹里。
- 第一阶段示例资源可以暂时简化命名，但后续新增正式英雄时应尽量按以上目录收束。

## 复杂表现特例的固定落点

复杂表现特例允许存在，但必须有固定落点，不能继续无限堆进 `BattleView` 这一类总视图脚本。

规则如下：
- `BattleView` 只负责通用视图同步、事件订阅和表现分发
- 通用技能特效优先走技能配置直接实例化
- 复杂技能表现应放到单独的表现控制脚本中
- 如果某个英雄的大招需要单独动画编排、分段特效或自定义时间轴，应建立独立脚本，不要把完整实现直接塞进总视图文件

推荐落点：
- `Assets/Scripts/UI/Presentation/Skills/`
- 或 `Assets/Scripts/UI/Presentation/Heroes/<heroId>/`

示例约束：
- 不应长期出现一串 `if skillId == xxx` 的大段特效实现都写在 `BattleView`
- 如果某个技能需要特殊表现，优先写成 `SkillPresentationController`、`MeteorPresentationController` 这类单独脚本
- `BattleView` 最多只保留“选择哪个控制器/哪个 prefab”的分发层职责

一句话规则：
- 英雄本体的东西跟英雄走
- 普攻的东西跟普攻走
- 技能的东西跟技能走
- 运行时状态只放运行时
- 复杂表现特例必须拆出总视图脚本

## 英雄 ID 规则

第一阶段英雄 ID 规则固定为：
- `职业前缀 + 三位序号 + 英文名字`

示例格式：
- `warrior_001_skybreaker`
- `mage_001_firemage`
- `assassin_001_shadowstep`
- `tank_001_ironwall`
- `support_001_sunpriest`
- `marksman_001_longshot`

约束：
- ID 必须稳定、唯一。
- 不允许后续随意改已有英雄 ID。
- 英雄英文名部分应简短、可读、可用于资源命名。

## 职业枚举

第一阶段代码中直接固定六类职业枚举：
- 战士
- 法师
- 刺客
- 坦克
- 辅助
- 射手

说明：
- 第一阶段不采用开放字符串分类。
- 如果未来真的要扩新职业，再升级数据结构和枚举体系。

## 六个职业的第一阶段基础定位

### 战士

- 近战或近距离持续作战
- 输出和生存相对均衡
- 可以带控制能力
- 如果带控制，优先是 `单点控制`

### 法师

- 相对脆弱
- 依赖技能打出高爆发或范围伤害
- 大招通常具备明显范围压制能力

### 刺客

- 高机动
- 优先威胁敌方后排脆皮
- 强调切入、突进、收割

### 坦克

- 前排承伤
- 限制敌方推进
- 常见方向包括：明确控制、稳定的团队保护 / 承伤，或极强的自我续航站场
- 即使不承担强控制或显式保护，也必须能在正面形成明确拖线价值，并写清楚反制点

### 辅助

- 负责治疗、增益、保护或控制
- 也允许具备较强、范围较大的控制效果
- 不以高额爆发输出为主要职责

### 射手

- 远程持续输出
- 对攻速、暴击、输出位置更敏感
- 依赖稳定站位持续打伤害

## 默认职业 AI 模板

第一阶段要求每个职业至少有一个默认 AI 行为模板。

说明：
- 默认模板用于形成统一职业行为基线。
- 具体英雄可以在模板基础上扩展或覆盖。
- 这不是要求所有英雄完全相同，而是要求先有继承起点。

职业模板方向示例：
- 战士：压前排，优先接敌，持续输出
- 法师：保持中后排，找敌方密集区释放技能
- 刺客：寻找切入路径，优先后排脆皮
- 坦克：顶在前排，优先接敌并替队友承压；按英雄配置打出控制、保护或自我续航技能
- 辅助：靠近友军，根据局势治疗、增益或控制
- 射手：维持安全输出位置，持续攻击可攻击目标

### 投射物普攻英雄的默认受压后撤基线

- 这是一条按 `普攻形态` 共享的 AI 基线，不只属于某个职业。
- 默认覆盖射手、法师，以及使用投射物普攻进行输出或治疗的辅助。
- 当英雄的普攻配置为 `usesProjectile = true` 时，应继承“被危险 threat 压近时短时后撤”的默认行为。
- 这个默认行为的目标是维持更安全的输出或治疗位置，而不是把单位做成长期逃跑或完整走 A 状态机。
- 正常远程互点不应触发机械式双后撤；只有近战 / 短手威胁，或已经压进异常近距离的远程 threat，才应触发。

## 特例逻辑规则

第一阶段允许少量特例逻辑，但必须显式标记。

规则如下：
- 默认必须先尝试用统一技能系统、统一目标系统、统一状态系统实现。
- 只有在统一系统无法表达时，才允许补充自定义逻辑。
- 所有特例逻辑必须在配置中显式标注。
- 所有特例逻辑必须在对应英雄文档中说明用途和原因。

说明：
- 不允许无节制地为每个英雄各写一套完全独立脚本。
- 允许特例，是为了避免统一系统刚起步时过度僵化。

## 技能配置模板化

第一阶段技能配置必须模板化。

每个技能都应走统一技能配置结构，而不是每个技能拥有完全不同的数据形态。

建议统一字段至少包含：
- `技能ID`
- `技能名称`
- `技能类型`
- `激活模式`
- `目标类型`
- `目标优先过滤 / 回退参数`
- `释放条件或被动监听条件`
- `射程`
- `范围半径`
- `伤害或治疗倍率`
- `冷却时间（若为主动技能）`
- `动作覆写参数`
- `状态效果列表`
- `动画/特效资源引用`

说明：
- 模板化是后续批量扩英雄的前提。
- 即使存在少量特例，也必须建立在统一技能模板之外的补充层，而不是替代模板本身。

### 被动型小技能能力

第一阶段技能系统允许极少量英雄把唯一的小技能位定义为 `被动型小技能`，用于表达常驻监听或条件型自我增益。

这套能力至少应能表达：
- 常驻生效且无主动施放动作
- 监听施法者当前生命值比例等共享运行时条件
- 监听最近一段时间内对施法者造成直接伤害的不同敌方来源数量
- 按分段阈值或线性系数持续提供属性修正
- 在死亡、复活和生命值变化后重新按当前状态即时结算

约束：
- 被动型小技能仍占用唯一的小技能位，不额外新增独立被动栏
- 如果收益本质只是“普攻命中附带效果”，仍应优先走 `BasicAttackData`
- 如果收益本质是正式属性变化、伤害转治疗或其他共享动作覆写，应继续走统一数值结算与日志机制，不要并行维护私有结算入口
- 第一阶段优先把它收敛在“自身条件驱动的属性变化 / 短时动作覆写”范围内，不扩成完整天赋树或复杂符文系统

### 被动或触发型大招特例

第一阶段大招默认仍以 `主动释放型、一局一次` 为主，但允许极少量英雄把大招位定义为 `被动型` 或 `条件触发型`。

这类特例至少应满足：
- 它承载的是该英雄最高优先级的核心战局机制，而不是偷偷额外塞一个免费被动
- 触发条件、收益上限和运行时监听入口都能写成清晰文档，不依赖聊天上下文补充
- 即使没有主动施放动作，它仍然占用正式大招位，不再同时获得同级主动大招

补充约束：
- 英雄文档必须显式写清：触发条件、持续方式、是否依赖共享击杀参与 / 生命值 / 状态监听，以及为何不适合改写为主动释放型大招
- 这类大招若涉及属性成长、自我治疗、伤害转治疗或其他正式收益，仍应继续复用统一数值结算、统一事件和统一日志入口，不要私下维护第二套隐藏成长系统
- `docs/planning/stage-01-ultimate-numeric-guidelines.md` 中针对“一局一次主动大招”的结果锚点，对这类特例只作参考，不应机械套用

### 连续动作序列能力

第一阶段技能系统应预留通用的“连续动作序列”能力，不只服务某个射手大招。

这套能力至少应能表达：
- 连续普攻
- 连续多次释放同一个技能
- 在固定次数或固定持续时间内重复执行同一个标准动作

建议纳入统一配置或统一运行时契约的最小字段包括：
- 动作载荷来源，例如 `普攻`、`当前技能` 或后续可扩展的标准动作
- 重复次数或持续时间
- 动作间隔
- 序列期间的临时参数覆写，例如临时射程
- 当前目标失效时的刷新目标策略
- 可被哪些控制或中断条件打断

约束：
- 这套能力的目标是成为共享执行入口，而不是每个英雄各写一套循环逻辑。
- 如果第一阶段暂时只落地一个样例，也应按“通用序列能力的第一个实例”来实现，不要直接写成射手专属分支。

### 分段动作中的自有大招插入窗口能力

第一阶段技能系统允许极少量英雄在一个主动小技能的分段动作中声明“自有大招插入窗口”，用于表达先完成位移 / 落位，再立即接自己的大招，最后继续完成小技能后续动作的连携。

这套能力至少应能表达：
- 小技能分段名称，例如 `起手位移`、`插入窗口`、`后续冲锋`
- 插入窗口可持续的最短时间或逻辑检查点
- 允许插入的技能范围，第一阶段默认只允许插入施法者自己的大招
- 大招成功释放后的恢复动作，例如继续执行小技能后续冲锋、斩击或强制位移
- 大招未释放、资源已消耗或条件不满足时的回退动作
- 施法者死亡、硬控或目标失效时，当前序列是否取消以及已结算效果是否保留

约束：
- 这不是通用连招系统，不允许借此让任意小技能和任意大招自由穿插。
- 插入窗口必须由技能配置或英雄文档显式声明；没有声明的英雄仍按普通小技能和大招释放流程执行。
- 小技能的 CD、动作锁和后续动作归属必须清楚，不应因为插入大招而额外获得第二次小技能释放。
- 被插入的大招仍占用正式大招位和一局一次资源，不应被视为小技能附带的免费效果。
- 插入前后发生的位移、伤害、状态和日志仍应继续复用统一技能、状态、位移和事件入口，不要在单英雄脚本中私下串完整连招。
- 如果第一阶段暂时只有一个样例，也应把它作为“分段动作插入窗口”的第一个实例落地，而不是把大招插入逻辑硬编码进某个英雄。

### 路径推挤型强制位移能力

第一阶段技能系统应允许少量近战开团技能以“记录起点、方向和路径宽度，沿固定方向把一个或多个敌方目标推到终点附近”的方式表达冲锋顶回、撞阵、群体推挤等效果。

这类效果在第一阶段语义上属于 `击退 / 水平强制位移`，不属于状态效果本体；如果还需要击飞或眩晕，必须额外组合正式 `KnockUp` 或 `Stun` 状态。

这套能力至少应能表达：
- 推挤起点，例如施法者当前位置快照、闪烁落点或指定锚点
- 推挤方向来源，例如朝施法前位置、朝当前目标反方向或朝敌方 / 己方半场方向
- 推挤最大距离
- 路径宽度或碰撞半径
- 初始被推动目标集合
- 路径中追加收集目标的规则
- 多目标最终落点分配范围
- 强制位移持续时间
- 可选的直接伤害、状态或打断载荷

约束：
- 逻辑层应优先把它实现成“路径扫掠收集目标 + 统一强制位移 + 合法落点分配”的共享入口，不要依赖物理碰撞体实时互推作为规则判定。
- 路径中遇到的敌人可以加入同一次推挤集合，但同一目标在同一次推挤中最多被加入一次。
- 多目标被推到终点附近时，必须通过共享的合法落点分配或最小间距收束处理，避免把多个敌人堆到同一个坐标导致重叠、抖动或近战距离异常。
- 推挤本身不默认等同于 `KnockUp`、眩晕、嘲讽、不可选中或无敌；如果需要这些效果，必须作为额外正式状态或效果显式配置。
- 推挤期间若施法者死亡、被硬控或路径被边界截断，应按技能配置的取消 / 截断规则处理，不要由表现层补一个看起来完成的假位移。
- 推挤命中、追加收集、最终落点和每个目标的实际位移都应进入统一事件或调试日志，方便复查路径和落点是否符合预期。

### 径向扫波 / 延迟环带命中能力

第一阶段技能系统应允许少量技能以“记录一个中心点，再按距离分段延迟命中”的方式表达向外扩散或向内回卷的波纹、海浪或冲击圈。

这套能力至少应能表达：
- 中心点来源，例如 `施法者当前位置快照`、`指定目标点`
- 最大作用半径
- 波锋厚度或每层环带宽度
- 外放阶段持续时间
- 可选的外圈停顿
- 回卷阶段持续时间
- 外放与回卷各自的效果载荷
- 同一目标在每个阶段的命中上限

约束：
- 逻辑层应优先把它实现成“按时间顺序结算多层环带命中”的共享入口，而不是依赖物理碰撞体实时扫描的单英雄专属圆环脚本。
- 默认应以施法瞬间记录的中心点为整个序列的逻辑锚点，不要求实时跟随施法者后续移动；如果某个技能确实要跟随，必须在对应英雄文档中显式说明。
- 每段命中的伤害、治疗、状态和日志仍应继续复用统一结算、统一状态系统和统一事件机制，不要新增“海啸命中”“波纹状态”这类只服务单英雄的新底层。
- 如果第一阶段暂时只有一个样例，也应按共享的“径向扫波能力”落地，不要直接写成单英雄专属分支。

### 固定方向往返路径命中能力

第一阶段技能系统应允许少量技能以“记录施法起点与方向锚点，沿固定直线外放后再按原路回收”的方式表达回力镖、回旋刃、往返飞矛这类技能。

这套能力至少应能表达：
- 方向锚点来源，例如 `当前主目标位置快照`、`指定目标点`
- 外放最大距离
- 路径宽度
- 外放阶段持续时间
- 可选的最远点停顿
- 回收阶段持续时间
- 外放与回收各自的效果载荷
- 同一目标在每个阶段的命中上限
- 回收终点模式，例如 `施法起点快照` 或后续可扩的 `施法者当前位置`
- 外放与回收的显式阶段标记或等价表现入口

约束：
- 逻辑层应优先把它实现成“按同一条固定路径分两段结算命中”的共享入口，不要依赖物理碰撞体实时扫线或写成单英雄专属 boomerang 脚本。
- 默认应以施法瞬间记录的起点与方向为整个往返序列的逻辑锚点；第一阶段回收终点默认收敛为 `施法起点快照`，若未来要实时追踪施法者，必须先在对应英雄文档中显式说明。
- 外放与回收阶段的伤害、治疗、状态、事件和日志仍应继续复用统一结算与统一状态系统，不要新增“回旋命中”“回镖阶段”这类只服务单英雄的新底层。
- 外放与回收若需要不同投射物拖尾、音效或命中火花，应提供显式的阶段标记或等价表现入口，让表现层按技能配置区分，不要再靠 `skillId` 私下猜分支。
- 如果第一阶段暂时只有一个样例，也应按共享的“往返路径命中能力”落地，不要直接写成单英雄专属分支。

### 部署型固定战斗代理体能力

第一阶段技能系统应允许少量技能创建 `部署型固定战斗代理体`，用于表达沙兵、固定炮台、阵地触发点这类“留在场上、但不是英雄本体”的战场锚点。

这套能力至少应能表达：
- 生成锚点模式，例如 `指定位置`、`目标附近`、`施法者附近`
- 持续时间
- 同时存在上限
- 超上限时是否按最旧实例替换
- 是否可移动、是否可被选中、是否可受伤
- 作用半径或联动半径
- 代理体的动作模式，例如 `定时脉冲`、`跟随所有者普攻触发`、`跟随当前技能触发`

约束：
- 这类正式参数必须跟 `SkillData` 或对应技能效果配置走，不允许长期散写在单英雄脚本或表现层里。
- 如果某个代理体只是“在固定区域替所有者补打一段普攻 / 技能命中”，应优先走共享的 `owner-driven proxy strike` 语义，不要直接膨胀成独立 `RuntimeHero + 独立 AI`。
- 代理体造成的伤害、治疗、状态和日志，仍应继续复用统一结算、统一状态系统和统一事件机制，不要再为部署物并行维护第二套战斗流程。
- 如果某个代理体按设计不作为英雄单位，必须在共享规则里显式声明它不参与默认索敌、threat 后撤、最小间距、body blocking 和计分语义，而不是把这些排除条件散写成单英雄特例。

### 临时分身单位能力

第一阶段技能系统允许少量技能创建 `临时分身单位 / CloneUnit`，用于表达镜像、影分身、短时复制体这类“会移动、会索敌、可被攻击、但不是正式出战英雄”的单位。

这套能力不同于 `部署型固定战斗代理体`：
- 部署型固定战斗代理体默认是战场锚点，不作为英雄单位，不进入默认索敌、最小间距、受压后撤或击杀计分语义。
- 临时分身单位默认按临时英雄单位处理，可以移动、索敌、普攻、被选中、受伤和受控，但不占用 BP / 选手出战槽位，也不进入死亡复活与击杀计分。

这套能力至少应能表达：
- 分身来源，例如复制的 `HeroDefinition`、来源运行时单位、创建者运行时单位和所属阵营
- 分身类型，例如 `MirrorCopy`、未来可扩的 `ShadowClone`
- 持续时间与消失方式，例如到时消失、被击杀、创建者死亡时清理
- 属性缩放，例如最大生命、攻击力、防御、攻速、移速和攻击距离
- 技能参数策略，例如沿用源技能倍率与范围，或在个别分身类型上显式配置技能缩放；第一版镜像复制默认依靠分身自身属性缩放降低强度，不对 `clone-safe` 小技能再做统一二次折减
- 行为权限，例如是否可普攻、是否可用小技能、是否可用大招、是否可创建新的分身
- 交互权限，例如是否可被选中、是否可受伤、是否可被控制、是否参与最小间距和位置收束
- 复制规则，例如是否复制当前 Buff / Debuff、是否复制技能模板、是否复制外观
- 归属规则，例如分身造成伤害、治疗、状态和击杀时应归属到创建者还是分身自身

默认约束：
- 分身单位必须带有统一 `isClone` 或等价运行时标记。
- `CloneUnit` 不能再复制 `CloneUnit`。
- `CloneUnit` 不能释放创建分身、创建召唤物、创建部署型固定代理体、大招、一局一次技能、复活、永久成长或局外成长效果。
- `CloneUnit` 默认不继承源目标当前 Buff、Debuff、护盾、持续伤害、持续治疗、控制、临时形态或一局一次资源状态。
- `CloneUnit` 造成的伤害、治疗、状态和击杀，应在统一事件与日志中同时保留“实际行动单位”和“归属创建者”，避免统计与表现无法追溯。
- `CloneUnit` 死亡或到时消失不触发普通英雄复活，不增加击杀分，不回写任何英雄、技能、选手或 BP 静态数据。
- 如果源英雄的小技能不满足 `clone-safe` 条件，分身应只使用普攻，不应为了“复制完整体验”而绕开共享技能安全规则。

实现优先级：
- 第一版优先服务镜法师这类“复制敌方英雄弱化副本”的需求。
- 第一版可以先只允许分身使用明确可复用的普攻和 `clone-safe` 小技能；复杂被动、召唤、分身、复活、永久成长、特殊资源和英雄私有计数器默认禁用。
- 如果未来其他英雄也使用分身类效果，应复用同一套 `CloneUnit` 数据和运行时清理规则，不要新建另一套平行的影子单位系统。

### 技能按已有状态层数结算与击杀后触发

第一阶段技能系统应允许在统一模板内表达“读取目标已有正式状态，再决定额外收益”的能力。

这套能力至少应能表达：
- 读取目标当前某类正式状态的有效层数或等价累计值
- 按该层数为技能追加伤害、治疗、护盾或其他正式收益
- 仅对带有指定状态的目标生效，而不是无条件命中所有目标
- 目标因指定技能或指定效果死亡后，在死亡位置追加一次范围效果

约束：
- 这类状态读取必须优先通过统一状态系统查询，不要在英雄脚本里长期并行维护第二套“私有层数计数器”
- 被读取的状态应按配置指定 `状态资产 / 主题键 / 来源技能组`，不要把所有同大类状态混算；例如“中毒结算”不应默认把燃烧、流血也一起算进去
- `主题键` 的职责是表达“这些正式状态能否被同一类技能统一读取”；它不应自动把不同来源的实例并进同一个叠层池
- 默认叠层语义仍按“同一来源、同一技能或同一普攻 on-hit 载荷”分别计数；如果要让同一来源内多个施毒入口共用一组层数上限，应通过显式的局部叠层池配置表达，而不是靠主题键偷偷合并
- 默认语义应是“读取但不消耗层数”；如果某个技能要清空、减少或转化层数，必须在配置和文档里显式声明
- “击杀后触发范围效果”应优先以 `死亡位置快照` 作为范围锚点，不要求先建立可长期存在的尸体对象
- 如果同一次技能结算会先对一批初始目标造成本体伤害，再由击杀触发后续范围效果，则应先让这批初始目标各自完成一次本体结算，再处理由这些击杀产生的后续效果；前一个目标触发的尸爆或等价连锁，不应让后一个初始目标跳过自己的本体结算
- 击杀后的后续效果仍应复用统一的范围伤害、范围附状态、持续区域或共享事件载荷，不要把连锁收割逻辑直接写成某个英雄专属的大段脚本

### 伤害事件驱动的共享计数与阈值触发

第一阶段技能系统应允许在统一模板内表达“施法者对目标造成指定类型伤害时，为目标附加共享计数标记；达到阈值后立即触发正式效果”的能力。

这套能力至少应能表达：
- 监听来源范围，例如普攻、主动技能、持续伤害，或某技能所属的全部伤害入口
- 命中后附加指定正式状态或主题化计数标记
- 同一英雄的多个来源共享一组本地叠层池，例如普攻、小技能和大招共用同一感电计数
- 达到阈值时追加正式效果，例如眩晕、额外伤害、治疗或护盾
- 阈值触发后的层数处理规则，例如清空、消耗固定层数或保留剩余层数

约束：
- 这类计数必须优先落在统一状态系统里，不要在英雄脚本里长期并行维护私有计数器
- 监听入口应优先复用统一的伤害事件与技能命中事件，不要为单个英雄再写一套隐形 `onDamage` 流程
- 如果某个主动小技能附带一条常驻被动监听，这仍然视为同一个正式小技能位的组成部分，不额外新增独立被动栏
- 若多个来源共用同一组计数，归并规则必须显式声明为“同一英雄本地共享”或等价配置，不要靠 `heroId` 或技能名在代码里偷偷特判
- 阈值触发出来的 `眩晕`、额外伤害、治疗或护盾，仍应继续复用统一状态系统、统一数值结算和统一事件机制，不要写成“先自己判断满层，再直接改 RuntimeHero”式特例

### 受限状态事件驱动的自我计数与转化

第一阶段技能系统允许极少量反控制英雄在唯一小技能或大招内声明“受到负向限制时计数 / 转化”的共享能力。

这套能力至少应能表达：
- 监听施法者自己受到敌方负向状态、减速或负向强制位移请求
- 按状态类别过滤，例如只监听 `眩晕`、`击飞`、`嘲讽`、负数 `MoveSpeedModifier` 或带控制标记的强制位移
- 在正常受控时为施法者附加主题化自我计数标记
- 在短时控制转化窗口内，把后续控制或减速尝试转化为攻击力、攻速或其他正式属性收益
- 同一来源、同一技能或同一持续区域的最小触发间隔
- 计数的层数上限、持续时间刷新、死亡和复活清除规则
- 主动技能消耗层数后的额外动作，例如一次追加斩击或短时属性收益

约束：
- 这类监听仍然属于正式技能配置的一部分；如果由主动小技能附带常驻监听，它仍占用唯一小技能位，不额外新增独立被动栏
- 计数标记应优先落在统一状态系统或等价共享运行时计数入口，不要在单英雄脚本里长期维护私有层数字段
- 控制转化窗口必须复用 `docs/planning/stage-01-status-effect-decisions.md` 中的窄语义，不得顺手扩成无敌、不可选中、减伤、完整驱散或完整法术免疫
- 被转化的控制尝试、获得的层数、属性收益和消耗层数后的动作，都应进入统一事件或调试日志，方便核对反控制收益是否过量
- 若某个技能允许在硬控期间触发自我负面清除，必须显式声明这是“仅用于自我清除的受控触发入口”，不要让所有主动技能都绕过行为门禁

### 技能特效参数绑定规则

第一阶段技能特效必须与技能配置参数绑定，不允许长期独立维护一套仅供表现层使用的“假参数”。

至少遵守以下规则：
- `范围型特效` 的可见范围应绑定技能的 `范围半径` 或对应效果的 `radiusOverride`
- `持续型特效` 的存在时长应绑定技能效果的 `durationSeconds`
- `周期触发型特效` 的重复播放频率应绑定技能效果的 `tickIntervalSeconds`
- `投射物表现` 的飞行速度、起点与命中目标应绑定技能或普攻的投射物参数

约束：
- 不允许为了“先看起来差不多”而长期手写一套与技能参数脱节的特效缩放、持续时间或播放频率
- 如果某个特效资源本身尺寸与技能参数不一致，应通过统一换算规则适配，不要为单个英雄散写魔法数字
- 表现资源配置可以决定“用哪个特效”，但技能配置负责决定“这个特效按多大范围、持续多久、多久触发一次”

## 普攻配置模板化

第一阶段普攻也必须配置化，不允许长期写死在代码里。

建议普攻统一配置至少包含：
- `伤害倍率`
- `攻击间隔` 或攻速换算参数
- `射程`
- `是否投射物`
- `投射物速度`
- `命中附带效果列表`
- `同目标持续普攻叠层配置`
- `命中特效引用`
- `可选的固定顺序子段列表`
- `子段专属效果类型 / 目标类型 / 目标策略 / 倍率`
- `目标阵营完全缺失时的显式回退子段规则`
- `子段专属命中附带效果与表现标记`

说明：
- 这样才能稳定支持近战、远程、投射物、暴击表现，以及“普攻附加减攻 / 减防 / DOT / HOT”等共享扩展。
- 如果普攻只有一种形态，可以只填写基础字段；如果是固定轮转型普攻，则应优先用共享子段列表表达轮转顺序和每段表现差异，不要把轮转计数器或特效分支藏在 `RuntimeHero` 之外的单英雄脚本或 `BattleView` 里。
- 如果某个轮转型普攻需要在“敌方已空、但友方仍可治疗”这类窗口里持续出手，也应通过上述显式回退规则表达，不要让运行时靠“没打出去就原地卡住”或“偷偷跳段”这两种隐式行为碰运气。

### 同目标持续普攻叠层能力

第一阶段普攻系统应允许少量英雄在持续攻击同一个目标时，把同一次普攻链之外的长期收益叠起来，用于表达单挑升温、压制前排或越打越快这类机制。

这套能力至少应能表达：
- 叠层触发入口，例如 `普攻命中当前目标`
- 运行时绑定目标，例如使用目标的 `runtimeId` 或等价唯一标识
- 每次命中增加的层数
- 层数上限
- 每层带来的正式收益，例如 `AttackSpeedModifier`
- 清层条件，例如切换目标、目标死亡、目标不可选中、施法者死亡或复活
- 临时覆写条件，例如某个大招持续期间把输出收益视为满层

约束：
- 同目标叠层属于普攻运行时状态，不应回写到 `HeroDefinition`、`BasicAttackData` 或表现层对象里。
- 目标绑定必须能区分同一个英雄配置在场上出现的多个运行时实例，不能只按 `heroId` 绑定。
- 如果英雄需要在当前目标仍合法时尽量不切目标，应通过 AI 的共享目标保持参数表达；不要把目标保持写成完整仇恨系统，也不要让它绕过嘲讽、不可选中、死亡等正式目标覆盖规则。
- 层数收益仍应继续复用统一属性修正、动作覆写、事件和日志入口，不要在普攻计时器里直接藏一个英雄专属倍率。

### 普攻弹射链能力

第一阶段普攻系统应允许少量英雄在一次主目标命中后，继续把同一次普攻链按规则弹到附近其他敌人。

这套能力至少应能表达：
- 额外弹射次数上限
- 每跳搜索半径
- 每跳的伤害或治疗倍率
- 搜索锚点语义，例如 `以上一跳命中点为中心`
- 同一普攻链中的唯一命中规则
- 弹射链提前终止条件，例如附近没有合法新目标
- 首段与各次弹射的表现标记、跳数或等价事件参数

约束：
- 弹射命中仍属于“同一次普攻”的后续子段，不应重新开启攻击前摇、额外消耗普攻 CD，也不应把自己当成新一轮正式普攻再递归触发更多链条。
- 每跳命中的伤害、治疗、状态、日志和击杀归属，仍应继续复用统一结算与统一事件机制，不要在英雄脚本里私下生成第二套伪技能命中流程。
- 若设计要求“同一目标在同一次普攻链中最多命中一次”，这条唯一命中集合应作为正式共享语义表达，不要由单英雄脚本临时维护私有 `HashSet` 当长期方案。
- 表现层应能直接读取首段 / 弹射段的显式标记、跳数或起止目标，让主投射物、弹射拖尾和命中火花按统一事件挂接；不要让 `BattleView` 继续靠 `heroId` 猜“这次是不是弹射”。

## 首批 6 个英雄的定位

第一阶段首批 6 个英雄直接按职业原型做“模板英雄”。

建议模板方向：
- 战士：近战持续输出型
- 法师：范围爆发型
- 刺客：切后排突进型
- 坦克：前排承伤控制 / 保护型
- 辅助：治疗增益或保护控制型
- 射手：远程持续输出型

说明：
- 第一批英雄不追求复杂主题包装。
- 第一批英雄的主要职责是验证框架、职业差异和技能系统。

## 英雄命名风格

第一阶段英雄命名采用：
- `职业原型名`
- 或 `开发代号`

说明：
- 第一阶段不以世界观包装为优先。
- 原型名和代号更利于快速开发和沟通。

## 标签系统

第一阶段需要简单英雄标签系统。

标签用于职业之外的补充描述，例如：
- `近战`
- `远程`
- `爆发`
- `持续输出`
- `治疗`
- `突进`
- `控制`

约束：
- 第一阶段标签系统保持简单。
- 可用字符串数组或枚举数组实现。
- 标签是补充分类，不替代职业字段。

## 英雄说明文档要求

第一阶段要求每个英雄都要有一份说明文档。

文档用途：
- 说明英雄定位
- 说明技能意图
- 说明目标职业模板
- 说明设计边界
- 说明是否存在特例逻辑

说明：
- 这会增加一点前期文档成本。
- 但对于 AI 协作开发，这能显著降低后续理解偏差。

### 英雄说明文档目录规则

英雄具体设定文档统一放在：

- `docs/heroes/<class>/<english-name>.md`

职业目录固定为：

- `warrior`
- `mage`
- `assassin`
- `tank`
- `support`
- `marksman`

说明：
- `docs/planning/` 继续承载规则、阶段决策和实现约束
- `docs/heroes/` 只承载单个英雄的具体设定、技能说明和设计边界
- 每份英雄文档至少应包含：英文名、中文名、英雄分类、普攻类型、小技能、大招、设计边界

## 设计边界要求

第一阶段必须为每个英雄定义设计边界，防止设计失控。

示例：
- 坦克不做超高全屏输出
- 辅助不做超高爆发核心输出
- 刺客不做超远程站桩输出
- 射手不做超大范围强控核心角色

说明：
- 设计边界是控制首批英雄复杂度和职业辨识度的重要工具。
- 每个英雄说明文档里都应明确记录“该英雄不做什么”。

## 首批英雄覆盖的系统特性

首批 6 个英雄不只是职业各来一个，还必须故意覆盖不同系统能力，确保战斗框架真正被验证到。

首批英雄整体至少要覆盖：
- 近战普攻
- 远程投射物普攻
- 范围伤害
- 位移
- 治疗或增益
- 眩晕控制
- 大招范围判断

说明：
- 如果首批英雄没有覆盖这些能力，系统即使看似完成，也仍然未经充分验证。

## 对后续 AI 的约束

后续 AI 在新增英雄或调整英雄时，必须遵守以下规则：
- 优先复用统一英雄结构、统一普攻模板、统一技能模板、统一状态系统。
- 没有充分理由时，不要新增完全特例化实现。
- 所有特例都必须标记，并写入英雄说明文档。
- 新英雄必须归属六类职业之一。
- 新英雄必须包含稳定 ID、标签和说明文档。
- 第一阶段新增英雄时，应优先补齐系统覆盖面，而不是盲目堆数量。
- 如果新增英雄依赖“普攻命中附带效果”，优先扩 `BasicAttackData` 的共享配置能力，不要把这类逻辑散写到 `RuntimeHero`、AI 或单英雄被动脚本里。
