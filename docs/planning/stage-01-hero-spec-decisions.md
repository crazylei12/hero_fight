# 第一阶段英雄开发规范决策

最后更新：2026-04-11

## 文档用途

这份文档记录第一阶段关于英雄定义、配置、扩展和文档约束的正式决策。后续 AI 在新增英雄、调整英雄、设计技能、扩展职业模板时，必须以本文件为直接依据。

如果后续规则变化，应直接修改本文件，而不是继续依赖问题清单中的原始回答。

## 第一阶段目标

第一阶段英雄系统的目标是：
- 建立统一、可扩展的英雄配置规范
- 支持首批 6 个职业模板英雄
- 为后续持续新增英雄提供稳定格式
- 避免英雄逻辑完全碎片化
- 允许少量特例，但必须被显式管理

## 英雄数据分层

第一阶段英雄数据固定拆成三层：
- `基础数据`
- `战斗逻辑配置`
- `表现资源配置`

### 基础数据

用于描述英雄的身份和战斗基础信息，例如：
- 英雄 ID
- 英雄名称
- 职业
- 基础属性
- 标签
- 设计说明或调试备注

### 战斗逻辑配置

用于描述英雄如何参与战斗，例如：
- 普攻配置
- 小技能配置
- 大招配置
- AI 行为模板
- 特例逻辑标记

### 表现资源配置

用于描述英雄的表现资源引用，例如：
- 头像
- 立绘或占位图
- 2D 战斗单位预制体或平面角色资源
- 如有必要，也允许临时引用 3D 占位预制体
- 动画资源
- 投射物资源
- 特效资源
- 音效资源

说明：
- 这三层必须明确分离。
- 不允许把所有信息长期堆进一个混乱的大对象里。
- 第一阶段默认资源方向应优先兼容 `Sprite / SpriteRenderer / 2D 角色 prefab`。
- 如果某个字段同时能挂 2D 与 3D 资源，规划意图仍然是 `2D 正式表现优先，3D 仅作临时调试占位`。

## 英雄配置内容范围

第一阶段每个英雄配置文件至少应包含以下内容：
- `英雄ID`
- `英雄名称`
- `职业`
- `基础属性`
- `普攻配置`
- `小技能配置`
- `大招配置`
- `AI 行为标签`
- `表现资源引用`
- `调试备注`

约束：
- 第一阶段只保留和战斗相关的字段。
- 第一阶段不要求加入剧情、背景故事、台词、稀有度等包装字段。

## 英雄内容归属固定规则

第一阶段后续新增英雄时，必须先按“谁拥有这部分内容”来决定放置位置，不允许边做边猜。

固定归属如下：
- `HeroDefinition` 只作为英雄总入口和装配点
- `HeroStatsData` 只负责英雄基础属性
- `BasicAttackData` 只负责普攻规则与普攻表现参数
- `SkillData` 只负责该技能自己的战斗参数、效果参数与技能表现参数
- `HeroVisualConfig` 只负责英雄本体级表现资源
- `RuntimeHero` 只负责战斗中的临时状态
- 表现层控制脚本只负责把事件和配置转成画面，不负责决定战斗结果

### HeroDefinition 的固定职责

`HeroDefinition` 必须是“英雄装配清单”，不是大杂烩逻辑脚本。

它可以拥有：
- 英雄 ID、名称、职业、标签
- `baseStats`
- `basicAttack`
- `activeSkill`
- `ultimateSkill`
- 职业或英雄 AI 模板 ID
- 特例逻辑标记与说明
- `visualConfig`

它不应长期拥有：
- 技能内部效果细节
- 技能专属特效参数
- 运行时血量、CD、目标、位置
- 复杂技能执行代码
- 技能命中特殊判定的硬编码实现

说明：
- 英雄换技能时，应只替换 `SkillData` 引用，而不是再回头修改英雄本体里一堆技能细节字段。
- `HeroDefinition` 可以引用技能，但不应吞掉技能自己的配置边界。

### HeroVisualConfig 的固定职责

`HeroVisualConfig` 只负责“这个英雄本体长什么样、用什么基础表现资源”，不负责承载所有技能表现。

它适合拥有：
- 英雄头像
- 英雄战斗 prefab
- 英雄 animator controller
- 英雄本体常驻挂件或默认材质
- 普攻投射物资源
- 英雄通用受击表现或死亡表现的资源引用
- 可选的通用施法/受击兜底资源

它不应长期拥有：
- 某个小技能专属范围特效
- 某个大招专属持续区域特效
- 某个技能独有的音效节奏参数
- 依赖单个技能 ID 才能成立的表现配置

规则：
- 只要某个表现资源是“换了技能就应跟着一起换”的，它就不属于 `HeroVisualConfig`，而属于对应 `SkillData` 或普攻配置。

### BasicAttackData 的固定职责

普攻相关内容必须尽量跟普攻走，不应散落到英雄本体或 UI 脚本里。

`BasicAttackData` 至少应负责：
- 普攻伤害倍率
- 攻速换算所需参数
- 普攻射程覆盖
- 是否投射物
- 投射物速度
- 普攻命中表现需要的参数或资源引用

说明：
- 如果未来某个英雄更换武器型普攻，优先改 `BasicAttackData` 或其关联表现配置，不要把变化写死在 `BattleView`。

### SkillData 的固定职责

技能自己的东西，必须尽量都跟技能走。

`SkillData` 应负责：
- 技能 ID、名称、描述
- 技能类型、目标类型、释放距离、范围、CD
- 技能效果列表
- 技能大招决策模板
- 技能表现资源引用
- 技能表现所需的正式参数入口

技能表现资源引用至少应优先放在技能侧，而不是英雄侧，例如：
- 施法特效
- 命中特效
- 持续区域特效
- 技能专属投射物
- 技能音效

规则：
- 只要某个表现资源与某个技能强绑定，并且英雄换技能时它也应一起更换，这部分就必须放到 `SkillData` 或技能专属表现配置中。
- 技能的范围、持续时间、脉冲频率等正式参数，必须由技能配置或技能效果配置提供，不允许长期在表现层另存一套。

### RuntimeHero 的固定职责

运行时状态只能放在 `RuntimeHero` 或其他运行时对象中。

至少包括：
- 当前生命值
- 当前目标
- 当前位置
- 技能冷却
- 死亡/复活状态
- 当前状态效果
- 对局内统计

规则：
- 不允许把这些运行时状态回写到 `HeroDefinition`。
- 不允许让表现层脚本自己偷偷维护一份战斗真状态作为长期方案。

## 英雄目录与资源放置规则

第一阶段后续新增英雄时，目录也要固定，不允许“能放哪就放哪”。

推荐固定结构如下：
- `Assets/Data/Heroes/<heroId>/`
- `Assets/Data/Skills/<heroId>/`
- `Assets/Prefabs/Heroes/<heroId>/`
- `Assets/Prefabs/Projectiles/<heroId>/`
- `Assets/Prefabs/VFX/Skills/<skillId>/`
- `Assets/Art/Heroes/<heroId>/`
- `Assets/Audio/Skills/<skillId>/`

放置规则：
- 英雄总配置资产放在 `Assets/Data/Heroes/<heroId>/`
- 该英雄默认使用的技能资产放在 `Assets/Data/Skills/<heroId>/`
- 英雄本体 prefab、角色展示资源放在 `Assets/Prefabs/Heroes/<heroId>/` 或 `Assets/Art/Heroes/<heroId>/`
- 普攻投射物 prefab 放在 `Assets/Prefabs/Projectiles/<heroId>/`
- 技能专属特效 prefab 放在 `Assets/Prefabs/VFX/Skills/<skillId>/`
- 技能专属音效放在 `Assets/Audio/Skills/<skillId>/`

补充规则：
- 同一个技能如果未来要给多个英雄复用，仍然以 `skillId` 为主目录，不转回英雄目录。
- 同一个英雄的“本体资源”与“技能资源”要分开，不要全部塞进一个英雄文件夹里。
- 第一阶段示例资源可以暂时简化命名，但后续新增正式英雄时应尽量按以上目录收束。

## 复杂表现特例的固定落点

复杂表现特例允许存在，但必须有固定落点，不能继续无限堆进 `BattleView` 这一类总视图脚本。

规则如下：
- `BattleView` 只负责通用视图同步、事件订阅和表现分发
- 通用技能特效优先走技能配置直接实例化
- 复杂技能表现应放到单独的表现控制脚本中
- 如果某个英雄的大招需要单独动画编排、分段特效或自定义时间轴，应建立独立脚本，不要把完整实现直接塞进总视图文件

推荐落点：
- `Assets/Scripts/UI/Presentation/Skills/`
- 或 `Assets/Scripts/UI/Presentation/Heroes/<heroId>/`

示例约束：
- 不应长期出现一串 `if skillId == xxx` 的大段特效实现都写在 `BattleView`
- 如果某个技能需要特殊表现，优先写成 `SkillPresentationController`、`MeteorPresentationController` 这类单独脚本
- `BattleView` 最多只保留“选择哪个控制器/哪个 prefab”的分发层职责

一句话规则：
- 英雄本体的东西跟英雄走
- 普攻的东西跟普攻走
- 技能的东西跟技能走
- 运行时状态只放运行时
- 复杂表现特例必须拆出总视图脚本

## 英雄 ID 规则

第一阶段英雄 ID 规则固定为：
- `职业前缀 + 三位序号 + 英文名字`

示例格式：
- `warrior_001_bladeguard`
- `mage_001_firemage`
- `assassin_001_shadowstep`
- `tank_001_ironwall`
- `support_001_sunpriest`
- `marksman_001_longshot`

约束：
- ID 必须稳定、唯一。
- 不允许后续随意改已有英雄 ID。
- 英雄英文名部分应简短、可读、可用于资源命名。

## 职业枚举

第一阶段代码中直接固定六类职业枚举：
- 战士
- 法师
- 刺客
- 坦克
- 辅助
- 射手

说明：
- 第一阶段不采用开放字符串分类。
- 如果未来真的要扩新职业，再升级数据结构和枚举体系。

## 六个职业的第一阶段基础定位

### 战士

- 近战或近距离持续作战
- 输出和生存相对均衡
- 可以带控制能力
- 如果带控制，优先是 `单点控制`

### 法师

- 相对脆弱
- 依赖技能打出高爆发或范围伤害
- 大招通常具备明显范围压制能力

### 刺客

- 高机动
- 优先威胁敌方后排脆皮
- 强调切入、突进、收割

### 坦克

- 前排承伤
- 限制敌方推进
- 具备明确控制能力
- 控制往往更强、范围更大

### 辅助

- 负责治疗、增益、保护或控制
- 也允许具备较强、范围较大的控制效果
- 不以高额爆发输出为主要职责

### 射手

- 远程持续输出
- 对攻速、暴击、输出位置更敏感
- 依赖稳定站位持续打伤害

## 默认职业 AI 模板

第一阶段要求每个职业至少有一个默认 AI 行为模板。

说明：
- 默认模板用于形成统一职业行为基线。
- 具体英雄可以在模板基础上扩展或覆盖。
- 这不是要求所有英雄完全相同，而是要求先有继承起点。

职业模板方向示例：
- 战士：压前排，优先接敌，持续输出
- 法师：保持中后排，找敌方密集区释放技能
- 刺客：寻找切入路径，优先后排脆皮
- 坦克：顶在前排，优先接敌并打出控制
- 辅助：靠近友军，根据局势治疗、增益或控制
- 射手：维持安全输出位置，持续攻击可攻击目标

## 特例逻辑规则

第一阶段允许少量特例逻辑，但必须显式标记。

规则如下：
- 默认必须先尝试用统一技能系统、统一目标系统、统一状态系统实现。
- 只有在统一系统无法表达时，才允许补充自定义逻辑。
- 所有特例逻辑必须在配置中显式标注。
- 所有特例逻辑必须在对应英雄文档中说明用途和原因。

说明：
- 不允许无节制地为每个英雄各写一套完全独立脚本。
- 允许特例，是为了避免统一系统刚起步时过度僵化。

## 技能配置模板化

第一阶段技能配置必须模板化。

每个技能都应走统一技能配置结构，而不是每个技能拥有完全不同的数据形态。

建议统一字段至少包含：
- `技能ID`
- `技能名称`
- `技能类型`
- `目标类型`
- `释放条件`
- `射程`
- `范围半径`
- `伤害或治疗倍率`
- `冷却时间`
- `状态效果列表`
- `动画/特效资源引用`

说明：
- 模板化是后续批量扩英雄的前提。
- 即使存在少量特例，也必须建立在统一技能模板之外的补充层，而不是替代模板本身。

### 技能特效参数绑定规则

第一阶段技能特效必须与技能配置参数绑定，不允许长期独立维护一套仅供表现层使用的“假参数”。

至少遵守以下规则：
- `范围型特效` 的可见范围应绑定技能的 `范围半径` 或对应效果的 `radiusOverride`
- `持续型特效` 的存在时长应绑定技能效果的 `durationSeconds`
- `周期触发型特效` 的重复播放频率应绑定技能效果的 `tickIntervalSeconds`
- `投射物表现` 的飞行速度、起点与命中目标应绑定技能或普攻的投射物参数

约束：
- 不允许为了“先看起来差不多”而长期手写一套与技能参数脱节的特效缩放、持续时间或播放频率
- 如果某个特效资源本身尺寸与技能参数不一致，应通过统一换算规则适配，不要为单个英雄散写魔法数字
- 表现资源配置可以决定“用哪个特效”，但技能配置负责决定“这个特效按多大范围、持续多久、多久触发一次”

## 普攻配置模板化

第一阶段普攻也必须配置化，不允许长期写死在代码里。

建议普攻统一配置至少包含：
- `伤害倍率`
- `攻击间隔` 或攻速换算参数
- `射程`
- `是否投射物`
- `投射物速度`
- `命中特效引用`

说明：
- 这样才能稳定支持近战、远程、投射物、暴击表现等扩展。

## 首批 6 个英雄的定位

第一阶段首批 6 个英雄直接按职业原型做“模板英雄”。

建议模板方向：
- 战士：近战持续输出型
- 法师：范围爆发型
- 刺客：切后排突进型
- 坦克：前排承伤控制型
- 辅助：治疗增益或保护控制型
- 射手：远程持续输出型

说明：
- 第一批英雄不追求复杂主题包装。
- 第一批英雄的主要职责是验证框架、职业差异和技能系统。

## 英雄命名风格

第一阶段英雄命名采用：
- `职业原型名`
- 或 `开发代号`

说明：
- 第一阶段不以世界观包装为优先。
- 原型名和代号更利于快速开发和沟通。

## 标签系统

第一阶段需要简单英雄标签系统。

标签用于职业之外的补充描述，例如：
- `近战`
- `远程`
- `爆发`
- `持续输出`
- `治疗`
- `突进`
- `控制`

约束：
- 第一阶段标签系统保持简单。
- 可用字符串数组或枚举数组实现。
- 标签是补充分类，不替代职业字段。

## 英雄说明文档要求

第一阶段要求每个英雄都要有一份说明文档。

文档用途：
- 说明英雄定位
- 说明技能意图
- 说明目标职业模板
- 说明设计边界
- 说明是否存在特例逻辑

说明：
- 这会增加一点前期文档成本。
- 但对于 AI 协作开发，这能显著降低后续理解偏差。

### 英雄说明文档目录规则

英雄具体设定文档统一放在：

- `docs/heroes/<class>/<english-name>.md`

职业目录固定为：

- `warrior`
- `mage`
- `assassin`
- `tank`
- `support`
- `marksman`

说明：
- `docs/planning/` 继续承载规则、阶段决策和实现约束
- `docs/heroes/` 只承载单个英雄的具体设定、技能说明和设计边界
- 每份英雄文档至少应包含：英文名、中文名、英雄分类、普攻类型、小技能、大招、设计边界

## 设计边界要求

第一阶段必须为每个英雄定义设计边界，防止设计失控。

示例：
- 坦克不做超高全屏输出
- 辅助不做超高爆发核心输出
- 刺客不做超远程站桩输出
- 射手不做超大范围强控核心角色

说明：
- 设计边界是控制首批英雄复杂度和职业辨识度的重要工具。
- 每个英雄说明文档里都应明确记录“该英雄不做什么”。

## 首批英雄覆盖的系统特性

首批 6 个英雄不只是职业各来一个，还必须故意覆盖不同系统能力，确保战斗框架真正被验证到。

首批英雄整体至少要覆盖：
- 近战普攻
- 远程投射物普攻
- 范围伤害
- 位移
- 治疗或增益
- 眩晕控制
- 大招范围判断

说明：
- 如果首批英雄没有覆盖这些能力，系统即使看似完成，也仍然未经充分验证。

## 对后续 AI 的约束

后续 AI 在新增英雄或调整英雄时，必须遵守以下规则：
- 优先复用统一英雄结构、统一普攻模板、统一技能模板、统一状态系统。
- 没有充分理由时，不要新增完全特例化实现。
- 所有特例都必须标记，并写入英雄说明文档。
- 新英雄必须归属六类职业之一。
- 新英雄必须包含稳定 ID、标签和说明文档。
- 第一阶段新增英雄时，应优先补齐系统覆盖面，而不是盲目堆数量。
