# 第一阶段状态效果决策

最后更新：2026-04-22

## 文档用途

这份文档记录第一阶段状态效果系统的正式范围和落地方向。后续 AI 在扩展技能、英雄、数值、日志和战斗表现时，必须以本文件为依据。

它主要解决以下问题：
- 第一阶段到底正式支持哪些状态效果
- 哪些效果应归入统一状态系统，而不是散写成技能特例
- 在当前 `game/` 代码基础上，状态系统下一步应怎么扩

## 第一阶段目标

第一阶段状态系统的目标是：
- 用一套统一机制覆盖属性增减、硬控、持续伤害/治疗、特殊生存与目标规则
- 让首批英雄和后续英雄都能复用同一套运行时规则
- 保持第一阶段最小原型节奏，不一次性扩成完整 MOBA/MMORPG 状态库

## 状态效果与技能效果的边界

- `技能效果` 用于一次性结算，例如直接伤害、直接治疗、瞬时位移、创建持续区域
- `状态效果` 用于附着在单位身上的持续影响，例如属性变化、硬控、持续伤害/治疗、无敌、不可选中
- 一个技能可以同时包含多段技能效果和多个状态效果
- “受压后撤”与“最近 threat 记忆”属于 `AI 决策 / 运行时行为输入`，不属于状态效果正式集合
- `位移` 需要在本文件里一起约束，因为它和 `击飞`、`不可选中` 的边界紧密相关
- 第一阶段默认把 `位移` 视为 `技能效果`，不视为状态效果主集合的一员
- `击飞` 在第一阶段默认拆成两部分：
  - 控制语义由状态系统处理
  - 如果还需要明显水平位移，再额外叠加独立位移效果

## 第一阶段正式支持的状态效果范围

### 属性变化类

第一阶段正式支持以下属性变化：
- 攻击力上升 / 下降
- 防御力上升 / 下降
- 攻速上升 / 下降
- 移速上升 / 下降

说明：
- 同一种属性变化通过 `magnitude` 的正负表达增益或减益
- 属性变化类状态的 `magnitude` 统一按“百分比修正的小数形式”表达，不按固定点数表达
- 运行时语义固定为：`最终属性 = baseValue * (1 + 所有同类 magnitude 之和)`
- 示例：
  - `+20%` 写成 `0.2`
  - `-35%` 写成 `-0.35`
  - `+200%` 写成 `2.0`
- 第一阶段先不把暴击率、暴击伤害、攻击距离、最大生命值等更多属性修正列为正式必做集合
- 如果首批英雄后续确实需要这些扩展属性 Buff / Debuff，再增量补充，不提前铺太大范围

### 治疗修正类

第一阶段正式支持：
- `受治疗率修正`

语义规则：
- 这是一种挂在 `被治疗目标` 身上的统一状态语义，作用对象是目标最终真正入血的治疗结果，而不是施法者侧的“治疗强度”
- `magnitude` 继续使用百分比修正的小数形式：
  - `+25%` 写成 `0.25`
  - `-40%` 写成 `-0.4`
  - `-100%` 写成 `-1.0`
- 统一结算语义固定为：`finalHealReceived = baseHeal * max(0, 1 + 所有受治疗率修正 magnitude 之和)`
- 它同时作用于 `直接治疗` 和 `持续治疗`，不额外拆成两套规则
- `护盾` 不视为治疗，不受这类修正影响
- 当累计结果为 `-1.0` 时，表示该目标处于 `100% 禁疗`

补充约束：
- 第一阶段正式支持的是 `目标侧受治疗率修正`，不额外扩出一套独立的“施法者治疗量加成 / 减成”底层；若只是想让某个治疗技能更强，优先直接调整技能倍率
- 这条语义既能表达 `减疗 / 禁疗`，也能表达少量 `增疗`；首批明确需求优先围绕减疗和禁疗使用

### 控制类

第一阶段正式支持两种硬控：
- `眩晕`
- `击飞`

语义规则：
- 两者都会立即打断当前攻击或施法
- 两者都会阻止移动、普攻和主动施法
- 第一阶段的 `击飞` 默认表示“短时间空中硬控”，不自动附带水平击退
- 如果某个技能确实要把目标推开或拉走，应组合独立的位移效果，而不是把所有位移都塞进击飞语义里
- 第一阶段不做控制递减、韧性、控制免疫链等复杂系统

第一阶段额外正式支持一种 `强制攻击控制`：
- `嘲讽`

语义规则：
- `嘲讽` 通过统一状态系统表达，不单独引入完整仇恨系统
- `嘲讽` 期间，目标会把状态来源视为当前强制目标，并优先向其移动、尝试普攻
- `嘲讽` 期间，目标不能主动释放技能
- `嘲讽` 默认不阻止移动，也不阻止普攻；它和 `眩晕` / `击飞` 不属于同一层级的硬控
- 若嘲讽来源死亡、变成不可选中或不再是敌方单位，嘲讽应立即失效，不要继续留下“无来源控制”
- 第一阶段不做“嘲讽附带锁头仇恨记忆”“嘲讽结束后继续保留仇恨”“多个来源抢控制权”的额外系统

### 持续类

第一阶段正式支持：
- `持续伤害`
- `持续治疗`

语义规则：
- 两者都必须带 `durationSeconds` 和 `tickIntervalSeconds`
- 第一阶段按离散 tick 结算，不按逐帧持续流逝结算
- 每次 tick 都必须走统一伤害/治疗结算与日志事件
- 周期性效果若需要来源归属，必须保留施加者和来源技能信息，避免 DOT 击杀丢失归属
- 对 `DamageOverTime` 这类会按来源属性换算强度的周期伤害，默认只在施加时快照来源侧输入，例如攻击力；目标当前防御等减伤应在每次 tick 时重新参与统一伤害结算，不与 DOT 强度一起冻结

### 主题化 DOT 命名与层数读取

- `Poison / Burn / Bleed / Curse` 这类名称，第一阶段优先视为共享 `DamageOverTime` 的主题化表达，而不是每来一种主题就新增一个底层状态类型
- 如果某个英雄或技能需要判断“目标当前是否带毒”，应按配置指定的 `状态资产 / 主题键 / 来源技能组` 做匹配，不要把所有 `DamageOverTime` 都一概视为“中毒”
- `statusThemeKey` 这类主题键只负责“这是不是 poison / burn / bleed 主题”的查询语义，不直接决定 DOT 的叠层上限、刷新目标或重施加归组
- 如果某个技能需要读取“当前毒层”或等价的持续伤害层数，应优先从统一状态系统按指定 `statusThemeKey` 汇总所有匹配正式状态的 `有效层数` 或等价累计结果；这类读取允许跨不同来源、不同技能聚合，只要它们同属该主题
- `stackGroupKey` 如有配置，只应用于同一来源内的可选叠层池覆盖，用来表达“同一英雄的多个施毒入口共享一组层数上限”等局部规则；它不应跨不同来源合并 DOT 实例
- 这类层数读取只负责读正式状态，不改变 DOT 仍然属于统一状态系统的事实；日志、来源归属、叠层上限和重施加规则继续沿用统一状态语义

### 特殊保护与目标规则类

第一阶段正式支持：
- `护盾`
- `无敌`
- `不可选中`
- `分担伤害`

语义规则：
- `护盾` 通过统一状态系统附着在单位身上，优先吸收即将受到的伤害，再结算生命值减少
- `护盾` 的 `magnitude` 表示当前护盾值，重复施加时按配置决定是否刷新持续时间与护盾数值
- `护盾` 的 `magnitude` 是固定数值语义，不走属性变化类那套百分比修正规则
- `护盾` 不等同于治疗，不会抬高当前生命值，也不改变最大生命值
- `无敌` 只阻止伤害生效，包括普攻伤害、技能伤害、持续伤害
- `无敌` 不默认等同于不可选中，也不默认等同于免控
- `不可选中` 会阻止单位成为 `普攻`、`单体技能`、`明确指定目标的直接技能` 的合法目标
- `不可选中` 应同时影响 `AI 索敌` 和 `直接命中时的合法性复检`
- `不可选中` 默认不等同于无敌；若一个单位只是不可选中，仍可被非指向范围效果命中
- `不可选中` 不应阻止“目标就是自己”的技能生效；例如 `Self`、包含自身的友方技能、或明确允许自施法的技能，仍可对施法者本人合法生效
- 上述“自己对自己施法”的例外只适用于技能，不适用于普攻
- `分担伤害` 默认作为附着在 `被保护者` 身上的状态表达；状态的 `source` 是承担分担伤害的单位
- `分担伤害` 的 `magnitude` 统一表示分担比例，使用 `0 ~ 1` 的小数语义；例如 `0.35` 表示把该单位本次实际受到伤害中的 `35%` 转给来源单位承担
- `分担伤害` 的分摊基准是 `护盾吸收后的实际伤害`，不是护盾前毛伤害；如果本次伤害被目标自身护盾完全吸收，则不再额外转移
- 分担出去的那部分伤害仍然要走承担者自己的统一受伤入口，因此承担者的 `护盾`、`无敌`、日志和击杀归属都继续按统一规则处理
- 若承担者死亡、离场失效或不再是该目标的合法友方单位，`分担伤害` 应立即过期，而不是在承担者复活后自动恢复
- 第一阶段默认不继续把“已经被分担出去的伤害”再次向外连锁转移，避免循环引用和调试复杂度快速膨胀
- 第一阶段不做“完全脱离战场”“从世界中暂时移除”“失去所有碰撞和交互”这类更重的相位化系统

## 位移效果在本文件中的归属

- 要写在这里一起，因为位移和 `击飞`、`不可选中`、目标合法性会互相影响
- 但第一阶段默认把 `位移` 归为 `技能效果`，不归为 `状态效果`
- 第一阶段的典型位移包括：突进、闪现、短距离 reposition、推开、拉拽、击退
- `击飞` 之所以保留在状态效果里，是因为它的核心语义是“一段持续时间内的硬控”
- `推开 / 拉拽 / 击退` 等位置变化，第一阶段仍建议作为独立位移技能效果处理
- 只有未来真要做“持续牵引”“持续冲锋”“强制移动期间附带特殊行为规则”时，再考虑把位移升级为单独状态体系

## 统一语义规则

### 死亡与复活

- 第一阶段状态效果默认在死亡时清空
- 复活后不继承死亡前状态
- 复活后技能 CD 仍按既有规则不重置

### 叠层与重施加

- 硬控默认 `maxStacks = 1`，重复施加时只刷新持续时间，不叠更高层
- `无敌` 与 `不可选中` 默认 `maxStacks = 1`，重复施加时只刷新持续时间
- `分担伤害` 默认 `maxStacks = 1`；不同来源的分担伤害实例允许并存，最终按所有有效来源的比例汇总，但总分担比例不应超过 `100%`
- 对 `分担伤害` 而言，“同一来源”优先按承担者和来源技能识别；同一承担者、同一技能的重施加应刷新原实例，而不是仅因为比例变化就额外叠出新实例
- 属性变化、持续伤害、持续治疗允许按配置决定是否叠层
- 属性变化、持续伤害、持续治疗的 `maxStacks` 默认按“同一来源、同一技能或同一普攻 on-hit 载荷”分别计数；不同来源或不同技能的同类型实例允许并存
- 这些可叠层实例的最终效果继续按所有有效实例的 `magnitude` 累加；达到该组上限后，重复施加只刷新该组中已有实例的持续时间，不覆盖其他来源或其他技能带来的同类型状态
- 第一阶段不做“只保留最强值”“不同来源优先级对抗”“复杂覆盖表”等额外系统，除非后续实际证明不够用

### 行为阻断

- 后续实现里不应继续在多个模块里散写 `HasStatus(Stun)` 这种单点判断
- 统一规则应收束为“是否允许移动 / 普攻 / 施法 / 选目标 / 受伤”的集中查询
- `眩晕` 和 `击飞` 都属于 `硬控`，会影响同一组行为门禁
- `嘲讽` 主要影响“当前目标选择”和“是否允许主动施法”
- `不可选中` 主要影响“能否成为直接目标”
- `不可选中` 对自目标技能存在显式例外，但这个例外应由统一技能目标校验入口集中处理，不要散写在各个英雄特例里
- `无敌` 主要影响“伤害是否真正生效”

### 事件与日志

- 状态添加、移除、周期 tick 必须能进入统一事件流
- 周期性伤害和治疗应继续复用统一的 `DamageApplied` / `HealApplied` 方向事件，不要单独偷偷结算
- 状态日志至少应能说明：谁对谁施加了什么状态、持续多久、关键状态何时结束
- `无敌`、`不可选中` 这类关键状态的开始和结束，应尽量有明确事件，方便调试目标丢失和伤害失效问题
- `分担伤害` 生效后造成的承担伤害，也应继续复用统一的 `DamageApplied` 事件，并在伤害来源种类上标记为“由分担伤害转入”的那次受伤
- 如果某个状态的“运行时状态来源”与“实际施加者”不是同一个单位，事件与日志应能同时保留这两个信息，不要在 apply/remove 两端混用成一套含糊的 `source`

## 第一阶段明确暂不纳入底层正式集合的效果

以下效果暂不作为第一阶段状态系统的正式必做内容：
- `沉默`
- `定身`
- `缴械`
- `净化 / 驱散`
- `法术免疫`
- `带保留状态的复活效果`

原因：
- 除了已经确认纳入的 `嘲讽` 外，其余这些效果会显著扩大行为门禁、位置系统、优先级系统或数值结算的复杂度
- 当前第一阶段更需要先把统一状态系统做稳，而不是一次性做全

## 当前代码基础上的落地方向

当前 `game/` 里已经有一套可扩展但仍偏早期的状态骨架，主要入口包括：
- `game/Assets/Scripts/Data/StatusEffectType.cs`
- `game/Assets/Scripts/Data/StatusEffectData.cs`
- `game/Assets/Scripts/Heroes/RuntimeStatusEffect.cs`
- `game/Assets/Scripts/Heroes/StatusEffectSystem.cs`
- `game/Assets/Scripts/Heroes/RuntimeHero.cs`
- `game/Assets/Scripts/Battle/BattleEvents.cs`

建议按“最小增量改造”推进，而不是整套推翻重写。

当前实现里，`StatusEffectSystem.cs` 已开始承担统一的运行时状态入口，主要负责：
- 状态应用与重施加
- 行为门禁查询
- 属性修正查询
- 周期状态 tick 推进
- 护盾吸收与过期状态清理
- 分担伤害目标与比例归并

这一步的目标是先把最容易继续发散的状态职责收口，避免后续把状态判定继续散写回 `RuntimeHero`、`BattleSimulationSystem` 和技能执行流程里。

### 第一步：补齐正式状态类型

建议在当前 `StatusEffectType` 基础上，至少补齐：
- `KnockUp`
- `DamageOverTime`
- `Invulnerable`
- `Untargetable`
- `Shield`
- `DamageShare`
- `HealTakenModifier`

说明：
- 现有的攻击力、防御力、攻速、移速、持续治疗等类型可以继续保留
- 第一阶段不必为了追求抽象优雅，马上把所有属性变化重构成更复杂的通用表达式
- 当前更重要的是先把状态能力覆盖面补完整

### 第二步：扩展状态配置数据

建议优先补充到 `StatusEffectData` 的字段：
- `tickIntervalSeconds`
- `activeSkillCooldownCapSeconds`
- 如有需要，再补一个非常轻量的周期状态判断入口，避免把 tick 规则散写在多个模块里

现有字段可以继续沿用：
- `durationSeconds`
- `magnitude`
- `maxStacks`
- `refreshDurationOnReapply`

说明：
- `眩晕` / `击飞` 主要依赖持续时间
- 属性变化主要依赖 `magnitude`
- DOT / HOT 需要额外的 `tickIntervalSeconds`
- `activeSkillCooldownCapSeconds` 是可选的运行时上限字段；当它大于 `0` 时，状态持续期间会把小技能剩余 CD 压到不高于该值，适用于忍者大招这类临时高频释放窗口

### 第三步：让运行时状态保存来源信息

建议 `RuntimeStatusEffect` 在运行时额外保存：
- 施加者
- 来源技能
- 剩余 tick 计时

原因：
- DOT / HOT 需要正确记日志
- DOT 击杀需要保留归属
- 后续表现层也更容易知道某个状态来自哪次施法

### 第四步：收束行为判定入口

建议把当前状态相关的行为门禁收成统一查询，例如：
- `CanMove`
- `CanAttack`
- `CanCastSkills`
- `HasHardControl`
- `CanBeDirectTargeted`
- `CanReceiveDamage`

说明：
- 当前分散判断主要会出现在：
  - `BattleSimulationSystem`
  - `BattleSkillSystem`
  - `DamageResolver`
  - `BattleAiDirector`
- 这些入口不应继续各自只判断 `Stun`
- 一旦正式支持 `击飞`、`无敌`、`不可选中`，分散判断会很快失控

### 第五步：把周期效果改成离散 tick

当前如果存在按 `deltaTime` 持续回血的实现，建议尽快改为离散 tick 方案。

原因：
- 更容易对齐日志和表现
- 更容易和 `tickIntervalSeconds` 对齐
- 更容易控制 DOT / HOT 的强度预期
- 更符合后续持续区域和单位附着状态的统一节奏

### 第六步：优先补齐无敌和不可选中

推荐最小做法：
- `无敌` 只接到统一伤害入口，不额外扩成全免疫万能开关
- `不可选中` 先接入索敌、单体技能选目标、直接命中合法性复检
- 不要把 `不可选中` 做成“直接从战场列表中删除”

### 第六步补充：把护盾收进统一受伤入口

推荐最小做法：
- `护盾` 作为统一状态系统中的一种正式状态表达，而不是继续散写成技能局部变量
- 先在统一受伤入口做“护盾先吸收、剩余伤害再扣生命值”的最小语义
- 第一阶段不急着扩成多层优先级、护盾爆炸、护盾转治疗等高级系统

### 第六步补充：把受治疗率修正接入统一治疗入口

推荐最小做法：
- 在统一治疗入口集中读取目标身上的 `HealTakenModifier`
- 直接治疗与 `HealOverTime` tick 都走同一套目标侧倍率修正，不单独维护两份规则
- 多个受治疗率修正实例按 `magnitude` 累加，并在最终结算时对结果做 `>= 0` 的钳制
- `-1.0` 代表 `100% 禁疗`
- 不要把“减疗 / 禁疗”散写成某个技能命中后临时改写治疗脚本的私有分支

### 第七步：先把击飞做成“硬控 + 可选位移”的组合

第一阶段不建议把 `击飞` 一开始就做成复杂的物理抛物线系统。

推荐最小做法：
- 逻辑层把它当作一种短时硬控
- 表现层可额外播放抬升或弹起效果
- 如果技能需要明显位置改变，再复用现有位移 / reposition 能力组合

这样能先拿到玩法语义，而不会把时间都花在复杂轨迹系统上。

### 当前实现补充：击飞与强制位移的数据契约

- `KnockUp` 继续保留在统一状态系统里，负责“空中硬控”语义
- 若技能还需要把目标打退、拉近或轻微抛起，应额外组合独立的 `SkillEffectType.ApplyForcedMovement`
- `ApplyForcedMovement` 的最小数据契约应包含：
  - 方向模式
  - 水平位移距离
  - 位移持续时间
  - 表现层抬升高度
- 运行时上，强制位移不应被塞回状态列表里；它应作为英雄运行时的独立移动过程存在
- 第一阶段的最小语义可约束为：
  - 强制位移期间，单位不执行自发移动、普攻或施法决策
  - 水平落点仍受场地边界限制
  - “飞到空中”的主要语义由 `KnockUp` + 表现层高度偏移表达，而不是完整物理抛射

### 当前实现补充：击飞的默认画面表现

- 第一阶段的 `击飞` 默认采用 `共享表现层反馈`，不要让每个英雄各自重写一套受击飞表现
- 当前推荐的统一画面语言是：
  - 目标本体通过表现层高度偏移离开地面
  - 地面阴影保留在原地并缩小、变淡，用来强调“角色已经离地”
  - 起飞瞬间播放一次地面脉冲
  - 空中阶段保留轻量地面圈提示
  - 落地瞬间再播放一次更明显的落地脉冲
  - 如果同时存在明显水平位移，再叠加沿位移方向的短拖尾
- 这套表现的目标是：
  - 让玩家一眼区分“只是被推开”和“真的被打到空中”
  - 保持顶视角下的可读性，不依赖复杂 3D 抛物线或真实物理
  - 把击飞反馈沉淀成共享规则，而不是武术家或单个英雄的特例演出

## 推荐实现顺序

后续真正开始编码状态系统增强时，推荐顺序如下：

1. 扩 `StatusEffectType` 与 `StatusEffectData`
2. 给 `RuntimeStatusEffect` 补来源与 tick 计时
3. 把 `RuntimeHero` / `StatusEffectSystem` 的查询入口统一
4. 先完成 `无敌` 与 `不可选中`
5. 再完成 `持续伤害` / `持续治疗`
6. 再补 `分担伤害`
7. 最后补 `击飞`
8. 再把示例英雄或测试技能接进去验证

## 对后续 AI 的约束

后续 AI 在实现第一阶段状态系统时，必须遵守：
- 不要为每个新状态各写一套专属运行时小系统
- 不要把 `无敌` 写成“顺便禁掉所有控制和所有交互”的万能状态
- 不要把 `不可选中` 和 `无敌` 混成同一个概念
- 不要把 `不可选中` 做成“直接从战斗上下文中移除”
- 不要把 `击飞` 默认等同于“带水平位移的复杂物理系统”
- 不要让 DOT / HOT 绕开统一伤害/治疗结算
- 不要继续在多个模块里散写只针对 `眩晕` 的判断
- 如果状态系统的数据契约或语义变化影响现有文档，必须同步更新文档
