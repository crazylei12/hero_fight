# 第一阶段插件使用规则

最后更新：2026-04-11

## 文档用途

这份文档用于约束第一阶段原型开发中对当前项目已安装 Unity 插件的使用方式。

它要解决的问题是：
- 哪些插件可以优先使用
- 哪些插件只能用于表现层或编辑器提效
- 哪些插件当前阶段默认不要接入核心战斗系统
- 后续 AI 进入 `game/` 写代码时，应该在什么时机考虑这些插件

这份文档的目标不是鼓励引入更多插件，而是避免：
- 明明已经装了合适工具却重复造轮子
- 把强插件误接成第一阶段核心框架，导致范围膨胀
- 不同 AI 对插件取舍标准不一致

## 第一阶段总原则

- 第一阶段仍然优先使用 Unity 原生能力和现有统一架构。
- 插件只能服务于“更快落地最小原型”，不能反向主导项目架构。
- 战斗核心逻辑、数值结算、状态系统、事件机制、战斗输入输出对象，默认不得依赖第三方插件作为唯一实现前提。
- 若某插件会明显增加学习成本、工程耦合或迁移成本，应默认谨慎接入。
- 已安装不等于必须使用；只有在能直接降低当前阶段实现成本时才接入。

## 当前已确认存在的插件

当前项目里已经确认存在以下插件或大型第三方工具目录：
- `DOTween / DOTween Pro`
- `A Pathfinding Project`
- `Feel`
- `PlayMaker`
- `Adventure Creator`
- `Emerald AI`
- `Easy Save 3`
- `Febucci Text Animator`

- `vFavorites`

说明：

- 如果后续新增插件，应补充更新本文件，而不是默认自由使用。

## 第一优先级：推荐接入的插件

这些插件适合当前第一阶段原型，且 AI 默认可以主动考虑使用。

### DOTween / DOTween Pro

定位：
- `表现层`
- `UI 层`
- `少量编辑器辅助`

适合用途：
- UI 进出场动画
- HUD 数值跳动
- 结果页过渡
- 角色受击位移表现
- 投射物、提示圈、警告条等非核心规则动画
- 镜头或屏幕轻量过渡

接入时机：
- 已经有稳定的战斗主循环后
- 开始补战斗表现、UI 反馈时
- 需要替代手写协程动画时

强制约束：
- 不要让 DOTween 决定战斗是否命中、何时结算伤害或何时允许技能生效。
- 不要把规则层的时序绑定到某个 Tween 是否播放完成。
- 动画失败不应改变战斗结果。

结论：
- `默认允许主动使用`

### A Pathfinding Project

定位：
- `运行时逻辑层` 中的移动路径求解辅助
- `表现层` 的移动路径可视化辅助

适合用途：
- 单位绕障
- 非纯直线接敌
- 地图从纯空地扩到带简单阻挡时的寻路
- 刺客切入、远程走位等需要较稳定路径规划的情况

接入时机：
- 当前地图仍是纯平面、无障碍时，可以先不用
- 当单位移动开始因为障碍、拥挤或绕路出现明显问题时，再优先考虑接入

强制约束：
- A* 只负责“怎么走到目标附近”。
- 目标选择、技能释放、仇恨/优先级、胜负判定仍必须保留在项目自己的战斗系统里。
- 不要把整个单位 AI 决策外包给路径插件。

结论：
- `推荐作为移动层增强工具使用`

### Feel

定位：
- `表现层`
- `少量 UI 层反馈`

适合用途：
- 受击反馈
- 击杀反馈
- 屏幕震动
- 短暂停顿
- 闪白、缩放、音效联动
- 浮字和打击感反馈编排

接入时机：
- 战斗规则跑通后
- 开始提升“命中感”“大招感”“结果反馈感”时

强制约束：
- Feel 只消费战斗事件，不决定战斗规则。
- 时间缩放、震屏等反馈不能破坏 60 秒规则、复活计时和技能 CD 的统一时钟。
- 若使用反馈序列，必须能在关闭反馈时仍保持战斗正确运行。

结论：
- `推荐作为事件驱动反馈层使用`

## 第二优先级：可选使用的插件

这些插件在第一阶段不是必须，但某些场景下可以提高效率。

### Easy Save 3

定位：
- `外围功能`
- `调试辅助`
- `后续存档能力预留`

适合用途：
- 保存最近使用的阵容
- 保存调试配置
- 保存平衡测试参数
- 后续接最小本地存档

接入时机：
- 当 HeroSelect、调试菜单或平衡工具需要持久化配置时
- 不应先于核心战斗系统接入

强制约束：
- 第一阶段不要因为接存档而重写战斗核心结构。
- 存档层应读取统一输入/结果对象，而不是自己维护另一套事实来源。

结论：
- `可选`

### Febucci Text Animator

定位：
- `UI 层`

适合用途：
- 战斗播报
- 结果标题动画
- 关键提示文本
- 技能名称弹出

接入时机：
- 当基础 HUD、结果页和播报逻辑已经存在后

强制约束：
- 它只负责文本表现。
- 不要让文本播放节奏反向控制战斗流程。

结论：
- `可选，偏 UI 表现`

### vFavorites

定位：
- `编辑器效率工具`

适合用途：
- 快速定位常用资源
- 整理高频操作目录

接入时机：
- 任意阶段都可辅助使用

强制约束：
- 它不属于项目功能依赖。
- 不要把流程设计成离不开某个个人编辑器收藏结构。

结论：
- `可自由使用，但不作为项目功能方案`

## 第三优先级：默认不作为第一阶段核心方案

这些插件不是不能碰，而是当前阶段默认不应拿来承载核心战斗原型。

### PlayMaker

原因：
- 更适合关卡流程、可视化状态机、小型交互逻辑
- 不适合作为当前统一战斗核心、统一数值结算、统一状态系统的主承载层

允许用途：
- 小型流程原型
- 临时演出或界面状态控制
- 编辑器验证小逻辑

默认不要做：
- 不要把主战斗循环写成 PlayMaker FSM
- 不要把英雄核心 AI、伤害结算、状态系统大量转移到 PlayMaker

结论：
- `默认不用作核心战斗方案`

### Emerald AI

原因：
- 它是一整套通用角色 AI 方案
- 当前项目已经明确要保留自己的职业模板、技能逻辑、状态系统和战斗规则
- 直接接成核心 AI 会显著增加耦合和替换成本

允许用途：
- 参考其组件拆分思路
- 复用极少量非核心表现部件时需非常谨慎

默认不要做：
- 不要把英雄运行时 AI 建立在 Emerald AI 上
- 不要让 Emerald AI 接管目标选择、技能时机或死亡复活流程

结论：
- `第一阶段默认不接核心`

### Adventure Creator

原因：
- 偏剧情、交互、冒险流程框架
- 与当前 5v5 自动战斗原型目标不匹配

结论：
- `第一阶段默认不用`

### Pixel Crushers Dialogue System

原因：
- 偏对话、任务、剧情表达
- 与当前阶段核心目标无关

结论：
- `第一阶段默认不用`

## Odin Inspector 说明

当前项目中未确认存在 `Odin Inspector / Sirenix` 目录或对应工程内容。

规则：
- 后续 AI 不应默认假设项目已经安装 Odin Inspector。
- 在没有确认装入工程前，不要编写依赖 Odin 特性的必需字段、特性或自定义抽屉方案。
- 如果后续用户实际安装并希望启用，应在本文件追加一节说明其使用边界。

当前结论：
- `视为未安装`

## 第一阶段推荐使用顺序

如果后续 AI 在实现中考虑插件，优先顺序如下：

1. `DOTween / DOTween Pro`
2. `Feel`
3. `A Pathfinding Project`
4. `Easy Save 3`
5. `Febucci Text Animator`

补充说明：
- 若当前任务是“先把战斗跑通”，前三项也都不是必需前置。
- 推荐顺序表达的是“已需要该能力时优先考虑谁”，不是“项目一开始就全部接入”。

## 按开发阶段的插件选择建议

### 阶段 A 到阶段 D：工程骨架与基础战斗循环

优先策略：
- 默认先不用第三方插件解决核心战斗规则
- 允许直接使用少量 `DOTween` 做占位 UI 过渡，但不是必须

重点：
- 先把 BattleManager、RuntimeHero、索敌、普攻、死亡复活跑通

### 阶段 E 到阶段 G：数值、状态、技能、职业 AI

优先策略：
- 仍以自研统一系统为主
- 只有当移动问题明显阻碍验证时，再考虑 `A Pathfinding Project`

重点：
- 不要因为已有 AI 插件就绕开既定架构

### 阶段 H 到阶段 J：首批英雄、UI、日志、打击感

优先策略：
- 优先接入 `DOTween`
- 视表现需求接入 `Feel`
- 视文本表现需求接入 `Febucci Text Animator`
- 视调试与阵容保存需求接入 `Easy Save 3`

重点：
- 插件主要用于提升可读性、反馈和工作效率

## 对后续 AI 的明确要求

后续 AI 在 `game/` 目录开始实现时，必须遵守：

- 不要因为项目里装了很多插件，就默认都要接入。
- 先问“当前阶段是否真的需要这个能力”，再问“是否值得用插件实现”。
- 若某插件会让核心逻辑与第三方框架深度绑定，默认拒绝。
- 若某插件只影响表现层或编辑器效率，且能显著省时，可以优先考虑。
- 如果实际决定正式接入某插件到核心工作流，应在对应实现任务里更新本文件，记录：
  - 为什么接
  - 接到哪一层
  - 哪些边界不能被它接管
