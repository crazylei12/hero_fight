# 第一阶段战斗规则决策

最后更新：2026-04-22

## 文档用途

这份文档记录第一阶段竞技场原型已经确认的战斗规则。后续 AI 在实现战斗系统、英雄行为、技能系统、伤害结算和日志系统时，必须以本文件为直接依据。

本文件优先级高于战斗问题清单中的建议文本。若后续规则变化，应直接修改本文件。

## 第一阶段战斗目标

第一阶段战斗系统的目标是：
- 支持 `5v5` 自动战斗
- 支持死亡、复活、重新投入战斗
- 支持计时、计杀敌数、加时决胜
- 支持基础职业差异
- 支持统一的英雄和技能开发规范
- 支持后续扩展更复杂的职业行为状态机

## 伤害与基础属性

### 伤害类型

- 第一阶段 `不区分物理伤害和法术伤害`。
- 所有伤害统一走同一套伤害结算逻辑。
- 因此第一阶段不做：
  - 物抗
  - 魔抗
  - 技能伤害类型细分

### 英雄通用基础属性

第一阶段每个英雄至少具有以下通用属性：
- `生命值`
- `攻击力`
- `防御力`
- `攻速`
- `移速`
- `暴击率`
- `暴击伤害`
- `攻击距离`

说明：
- `暴击` 第一阶段就需要支持。
- 当前不强制加入独立的 `法术强度` 属性。
- 如果后续技能体系需要扩展更丰富的技能倍率，可在后续版本补充额外属性字段。

## 普攻规则

### 普攻模式

- 近战英雄普攻采用 `瞬时命中`。
- 远程英雄普攻采用 `投射物`。

### 普攻行为

- 英雄进入攻击范围后停下进行攻击。
- 攻击完成后，若目标仍有效，则继续攻击。
- 若目标死亡、丢失或离开可追击逻辑，则重新索敌。
- 当前实现基线中，普攻与主动释放技能统一采用 `0.2 秒前摇 + 0.2 秒后摇`。
- 前摇开始时即占用本次攻击/施法机会并进入动作锁，前摇结束后才真正结算命中、投射物或技能效果；后摇结束前不能移动、普攻或继续施法。

### 普攻命中附带效果

- 第一阶段允许普攻在基础伤害或基础治疗之外，通过统一配置附带额外效果。
- 这类附带效果应优先复用正式的 `状态效果` 或统一效果载荷，不单独为单个英雄发明“普攻被动专属底层”。
- 当前优先支持的方向是：普攻命中后附加属性 Buff / Debuff、持续伤害 / 持续治疗等已经在统一状态系统正式支持范围内的内容。
- 叠层上限、持续时间刷新、来源归属和日志记录，都应继续遵循统一状态系统与统一事件机制，不要在普攻流程里另写一套局部规则。
- 如果某个英雄的核心特色只是“普攻命中后附加一个 Debuff”，这仍应视为 `BasicAttackData` 的正式配置能力，而不是额外占用一个被动技能槽。

### 固定轮转型普攻

- 第一阶段允许少量英雄把一次“普攻动作”定义为按固定顺序轮转的多个正式子段，例如一次对敌造成伤害、下一次对友造成治疗。
- 每个普攻子段都应独立声明自己的 `效果类型`、`目标类型`、`目标策略`、倍率和命中附带效果，不要在同一次命中里临时改写目标阵营或结算类型。
- 轮转顺序不应因为当前理想目标缺失而偷偷跳过到下一段；是否回退到最近友军、自己或直接放弃本次出手，应由该子段的正式目标策略或回退策略显式决定。
- 但若某个子段的 `目标阵营` 在当前时刻已经完全没有合法目标，例如伤害子段时全场没有合法敌方目标，而配置中又显式声明可回退到某个治疗子段，则允许直接执行这次“阵营缺失回退”，避免单位进入无意义的空转等待。
- 这类“阵营缺失回退”必须作为正式配置能力表达；它的行为需要明确：是仅本次借用回退子段继续出手，还是把轮转序号一并推进，都不应由表现层或单英雄脚本临时猜测。
- 当前轮转序号属于 `RuntimeHero` 或等价运行时对象的一部分；如有需要，可在出生或复活时按配置重置到固定起手子段。
- 逻辑事件与表现分发应能携带显式的普攻子段标记，例如 `variantKey`；这样同一英雄的伤害弹、治疗弹或其他轮转子段，才能稳定挂接不同投射物、命中特效与音效，而不依赖表现层猜当前轮次。

## 索敌与行为基础规则

### 默认索敌规则

- 第一阶段默认优先选择 `最近敌人`。
- 默认敌方普攻索敌 `不锁定到目标死亡`；单位在每次基础战斗决策时，都应重新评估并切换到当前最近的可攻击敌方目标。
- 但架构上仍应支持通过共享目标类型覆盖默认索敌逻辑，例如治疗普攻选择低血量友军、保护型普攻优先寻找正在威胁友方后排的敌人。
- 当前工程已不再保留刺客专用或共享的敌方普攻偏好目标类型；敌方伤害普攻默认收敛到 `最近敌人` 规则，若未来真要新增新的敌方普攻模板，必须重新写入文档而不是悄悄回加代码。

后续方向示例：
- 辅助优先靠近友军并为低血量队友服务
- 坦克优先卡前排或限制敌方前压

### 仇恨系统

- 第一阶段 `不做完整仇恨系统`。
- 若后续需要嘲讽或强制攻击，优先作为技能效果实现，而不是先做完整底层仇恨框架。
- 当前第一阶段已正式支持 `嘲讽` 这一类“强制把来源视为当前目标”的轻量效果，但它仍必须落在统一状态系统和统一目标选择覆盖里，不升级成完整仇恨表。

### 移动规则

第一阶段采用以下规则：
- 英雄出生后自动向敌方区域前进。
- 进入攻击范围后停下攻击。
- 目标死亡或丢失后重新索敌。
- 若当前没有可攻击目标，则继续移动。
- 战斗中所有存活英雄按 `中心点距离` 保持至少 `1 世界单位` 的最小间距；若移动、复活或位移后出现重叠，应由运行时位置收束重新拉开。

补充约束：
- 这条最小间距是统一战斗空间规则，不依赖 prefab collider 或物理系统。
- 该数值必须低于当前最短近战实际接敌距离，避免近战英雄因为最小间距过大而攻击不到目标。

### 投射物普攻英雄的受压后撤基线

- 仅 `普攻配置 usesProjectile = true` 的英雄，默认接入这条通用行为。
- 当该英雄最近短时间内被敌方英雄造成了 `有效的直接伤害`，且当前没有先进入技能释放、也不能立刻打出下一次普攻时，可以优先短时后撤。
- 这条短时 threat 记忆只用于行为判断，不等同于完整仇恨系统。
- `DOT / 持续伤害` 不用于刷新这条 retreat threat。
- 近战、短手或已经压近的威胁可以正常触发后撤；远程 threat 只有在压进异常近距离时，才视为足够危险。
- 正常的远程对远程互点，不应仅仅因为“刚刚吃到伤害”就触发机械式双后撤。
- threat 窗口结束后，单位应恢复原本的索敌、前压、普攻与技能流程，不进入长期逃跑状态。

## 死亡、复活与出生点

### 死亡与复活规则

- 英雄血量归零后死亡。
- 死亡后进入复活等待时间。
- 复活等待时间固定为 `5 秒`。
- 复活后 `满血`。
- 每局开场时，系统会先为每个英雄在 `己方出生区` 内分配一个本局出生位置。
- 复活后回到 `己方本局分配的出生位置`。
- 复活后 `技能 CD 不重置`。

### 架构要求

由于比赛胜负依赖持续复活后的交战，因此战斗系统必须天然支持：
- 多次死亡
- 多次复活
- 英雄反复进出战场
- 复活后重新加入索敌和技能流程

### 击杀、助攻与击杀参与

- 第一阶段除击杀计分外，也正式支持 `助攻` 统计与共享 `击杀参与` 判定，供英雄被动、战斗结果和日志复用。
- 当前共享助攻窗口默认取目标死亡前最近 `8 秒` 内的有效贡献。
- 敌方贡献默认可来自：直接伤害、持续伤害、强制位移、硬控或其他正式负向状态效果；这些贡献应优先通过统一伤害、状态和事件入口登记，不要为单个英雄私下补一套“参与击杀计数器”。
- 友方贡献默认可来自：治疗、护盾、无敌、不可选中、分担伤害或其他正式正向状态效果；这类保护型贡献可在己方完成击杀时计为助攻。
- 某个英雄若声明“击杀参与后触发”，默认同时包括 `击杀` 与 `助攻`，除非该英雄文档显式写明只吃击杀。

## 技能规则

### 技能结构

第一阶段每个英雄统一拥有：
- 一个小技能
- 一个大招

补充说明：
- 第一阶段默认不额外设置独立的被动技能栏位。
- 允许极少量英雄把唯一的小技能位设计成 `被动型小技能`；这仍然占用小技能位，不额外增加第四个技能槽。

### 技能资源

- 第一阶段不做蓝量或能量。
- 主动型小技能采用 `纯 CD`。
- 被动型小技能不参与主动释放和冷却循环。
- 第一阶段主动释放型大招采用 `一局一次资源`，不会因为等待冷却结束而在同一局内再次释放。

### 小技能释放规则

- 主动型小技能为 `全自动释放`。
- 当 CD 转好且存在合法释放目标时，可以立即释放。
- 主动型小技能应支持基础目标选择策略。
- 若某英雄的小技能被定义为 `被动型小技能`，则该技能不走主动释放判断、不占用自动施法时机，但仍占用唯一的小技能位。

### 大招释放规则

- 大招在游戏中具有极高重要性。
- 大招应具备明显强于小技能的战局影响力。
- 大招不是单纯“CD 更长的小技能”，而是可能改写战局的关键技能。
- 第一阶段就必须为大招预留更强的释放逻辑。
- 主动释放型、一局一次的大招在数值设计上，也必须按“战局改写技能”处理，不应只给到略高于小技能的结果强度；正式数值锚点见 `docs/planning/stage-01-ultimate-numeric-guidelines.md`。

大招通用定位：
- 效果更强
- 范围可能更大
- 目标选择更讲究时机
- 释放条件不应简单等同于“小技能 CD 好了就放”

大招释放原则：
- 第一阶段仍以 `主动释放型` 大招为默认方案；这类大招仍然是自动释放。
- 主动释放型大招在当前第一阶段规则下，默认 `一局只能释放一次`。
- 允许极少量英雄把大招位定义为 `被动型` 或 `条件触发型`；这类特例必须在英雄文档里显式说明偏离原因、触发条件和强度边界。
- 但大招需要独立的释放判断逻辑。
- 伤害型大招应倾向在敌人较多、收益更高时释放。
- 对直线穿阵、路径伤害或冲锋斩击这类技能，收益判断允许按“固定方向路径上预计命中的敌人数”计算，不必强行退化成圆形范围人数判断。
- 防御型或保护型大招应倾向在友方局势危险时释放。

说明：
- 第一阶段即使只做基础版，也必须从架构上区分“小技能释放决策”和“大招释放决策”。
- 不能把大招实现成和小技能完全同一套条件检查。
- 被动型或条件触发型大招仍然占用正式大招位，不应被当成额外白送的独立被动栏。
- 第一阶段大招释放时机不应主要按职业硬编码分支决定，而应优先采用可配置的释放决策模板。
- 第一阶段应允许 `BattleInput` 中的蓝红双方分别配置自己的团队级大招策略，不要求两边共用同一档。
- 团队级策略至少包含两条轴线：
  - 开大时机：`Early / Standard / Late`
  - 开大协同：`Together / Standard / Separate`
- 为避免同一边英雄在首次接敌或团战形成时集体交大，`Separate` 档应能在某英雄成功释放主动大招后的短时间内，明显延后其同阵营其他英雄的大招判定；这条延后默认只作用于同阵营，不影响敌方阵营。
- `Together` 档应能让同阵营在短时间内更容易形成跟大，不应只做“较轻的错峰”。
- 具体模板方案见 `docs/stage_01/stage-01-ultimate-cast-decision-templates.md`。
- 具体数值设计应区分：瞬时纯伤害、持续伤害、瞬时治疗、持续治疗，以及带明显控制 / 保护的大招；持续技能打满时的总量应高于瞬时技能，而带强控制 / 强保护时应下调 raw 伤害或治疗预算。

### 技能目标策略

第一阶段技能目标系统至少要支持基础策略，供小技能和大招共用或扩展，例如：
- `最近敌人`
- `当前敌方目标（若当前目标失效则本次不施放）`
- `最低血量敌人`
- `范围内造成伤害最高的敌人`
- `敌方最后排英雄`
- `指定职业或标签优先的敌方英雄`
- `范围内敌人数最多的位置`
- `最低血量友军`
- `最低血量友方远程单位`
- `当前被敌方近身威胁的友方远程单位`

补充约束：
- 目标策略应支持最小的“先过滤、再排序、最后回退”组合，例如优先敌方刺客；若当前没有合法刺客目标，则再走配置的回退目标策略
- 回退后的具体行为由回退目标类型自身语义决定，不应在英雄文档里默认写成“必定重新抓最近敌人”
- 这类过滤与回退规则应尽量沉淀为共享目标策略或统一 helper，不要把“没找到指定职业时怎么办”散写成各个英雄自己的私有分支
- `最低血量敌人` 与 `最低血量友军` 的共享排序语义默认都应先比较 `当前绝对生命值`；若当前生命值并列，再比较 `当前生命值比例`。若仍并列，友方目标优先更近者，敌方目标优先更远者，分别贴合保护技能减少无谓走位和收割技能优先切后排残血的第一阶段意图。
- 如果未来确实需要“最低生命值比例敌人”这类不同语义，应新增显式目标策略名称，而不是继续复用 `最低血量敌人` 再在不同英雄里偷偷写出不同排序。
- 保护型区域技能允许采用“先尝试以受压后排友军为区域锚点；若当前没有危险锚点，再回退到敌方密集区域”的共享决策形式，不要把这类双锚点判断散写成单英雄专属大招 AI
- 若某个友方目标技能显式声明 `allowsSelfCast = true`，那么共享目标选择、直目标合法性复检和范围命中收集都必须允许施法者自己成为合法目标；不要出现配置上“允许自施法”，但运行时 helper 仍把自己从 `ThreatenedAlly` 等目标类型里排除的情况

### 友方正向效果的合法目标补充

- 第一阶段允许极少量英雄声明 `不接受来自其他友军的正向效果` 这类共享目标合法性约束。
- 当单位带有这类约束时，其他友军提供的 `直接治疗`、`持续治疗`、`护盾`、`无敌`、`不可选中`、`分担伤害`、`正向属性 Buff`、`正向受治疗率修正`，以及等价的范围正向效果，都不应再对其生效。
- 这类过滤应同时作用于友方普攻目标选择、技能目标选择、范围命中收集、直接命中合法性复检，以及 AI 的治疗 / 增益决策，避免把技能和普攻浪费在无效目标上。
- 这条约束默认只屏蔽“其他友军 -> 该单位”的正向效果；该单位自己的自我强化、自我治疗，以及敌方施加的负向效果不受影响。

## 第一阶段技能类型范围

第一阶段先做最基础的技能类型集合，但架构必须为扩展预留空间。

建议纳入第一阶段最小集合：
- `单体伤害`
- `范围伤害`
- `单体治疗`
- `增益 Buff / 属性上升`
- `减益 Debuff / 属性下降`
- `持续伤害`
- `持续治疗`
- `位移`
- `被动型常驻 / 条件型自我强化`

说明：
- 第一阶段不追求一次性做全技能系统。
- 但技能框架不能写死成只支持伤害一种效果。
- 技能带来的短时 `吸血`、临时攻速覆写或等价的伤害转治疗效果，优先作为技能执行期间的正式动作覆写或共享效果载荷表达，不单独扩成全局常驻基础属性系统。
- 与状态效果直接相关的正式范围见 `docs/planning/stage-01-status-effect-decisions.md`。

## 部署型固定战斗代理体

- 第一阶段允许少量技能创建 `部署型固定战斗代理体`，用于表达沙兵、固定炮台、阵地触发点等限时战场锚点。
- 这类代理体默认不是 `英雄单位`，不参与出生、死亡、复活、击杀计分、阵营人数统计或前后排站位判断。
- 这类代理体默认不进入英雄默认索敌集合，也不参与面向英雄的范围命中收集、`usesProjectile` 远程英雄的受压后撤 threat 记忆、“被贴身”语义、最小间距收束或 body blocking。
- 是否可被选中、是否可受伤、是否可移动，应由技能配置显式声明；第一阶段允许直接采用 `不可移动 + 不可选中 + 无敌` 的最小版本。
- 代理体造成的伤害、治疗、日志和击杀归属，默认仍挂回所属英雄及来源普攻 / 技能，不单独引入独立记分单位。
- 这类代理体应至少支持 `持续时间`、`同时存在上限`、`超上限时按最旧实例替换` 这三项共享规则。

## 控制效果

- 第一阶段正式支持两种硬控：`眩晕` 和 `击飞`。
- 两者都必须纳入统一状态系统。
- 第一阶段的 `击飞` 默认表示短时空中硬控，不默认附带水平击退；若需要推/拉位移，应作为独立效果组合。
- 第一阶段额外支持一种轻量强制攻击控制：`嘲讽`。
- `嘲讽` 不等同于硬控；它的最小语义是“强制把来源作为当前目标，并禁止主动释放技能”，不额外引入完整仇恨系统。
- 除 `嘲讽` 外，第一阶段仍不急于同时支持沉默、定身、缴械等更多控制体系。

## 特殊状态

- 第一阶段还支持 `护盾`、`无敌`、`不可选中` 和 `分担伤害`。
- `护盾` 优先作为统一状态系统表达，先吸收伤害，再结算生命值减少。
- `无敌` 只阻断伤害生效，不自动等同于不可选中或免控。
- `不可选中` 阻断成为普攻和单体直接技能的合法目标，但不自动等同于无敌。
- `分担伤害` 默认把目标在护盾结算后的实际伤害按比例转给状态来源单位承担，承担者继续走自己的护盾、无敌、日志和击杀归属流程。
- `不可选中` 不应让单位免疫非指向范围效果、路径效果或持续区域效果；这类效果在命中收集阶段不应继续按“直接目标合法性”过滤。
- 详细语义与位移边界见 `docs/planning/stage-01-status-effect-decisions.md`。

## Buff / Debuff 系统

- 第一阶段必须实现 `统一状态系统`。
- 所有增益和减益效果都应尽量走统一的状态机制。
- 不接受每个技能各写一套散乱状态逻辑作为长期方案。

说明：
- 这是后续扩展英雄数量时最关键的底层之一。
- 控制、增益、减益、持续伤害/治疗、特殊保护与目标规则都应尽量建立在统一状态框架之上。
- 详细状态语义、正式效果范围与实现边界见 `docs/planning/stage-01-status-effect-decisions.md`。

## 阵容与站位规则

### 选人规则

第一阶段选人阶段先不做复杂限制：
- 可以随意选英雄
- 暂不限制职业数量
- 暂不限制双方是否重复英雄
- 暂不同步做复杂阵容合法性校验

说明：
- 第一阶段重点是验证战斗框架。
- 阵容平衡和选人约束可后续再收紧。

### 站位规则

- 第一阶段采用 `按职业前后排优先级自动分层站位`。
- 坦克英雄默认最靠近中线，战士与其他近战 / 短手英雄稍后，远程 / 投射物英雄默认更靠后。
- 同一局内各英雄的具体出生位置会在己方出生区中 `随机分配并打散`，不再固定成整齐槽位。
- 出生点纵向默认散布在地图中间带附近，不贴近上下边框。
- 第一阶段不做手动布阵编辑器。

## 战斗日志

第一阶段需要 `简易战斗日志`，至少记录关键事件，架构上允许后续扩展。

建议至少支持以下关键事件：
- `战斗开始`
- `英雄出生`
- `目标变更`
- `技能释放`
- `伤害`
- `治疗`
- `死亡`
- `复活`
- `击杀计分变化`
- `进入加时`
- `战斗结束`

说明：
- `普攻命中` 可以视调试需求决定是否默认记录。
- 第一阶段日志重点是辅助规则验证和平衡调试，而不是做完整回放。

## 对后续 AI 的约束

后续 AI 在实现第一阶段战斗系统时，必须遵守以下规则：
- 不要擅自引入物理/法术双伤害体系。
- 不要省略暴击系统。
- 不要把大招做成和小技能完全相同的释放逻辑。
- 不要跳过统一状态系统。
- 不要在第一阶段提前做过多复杂阵容限制。
- 所有实现都必须以“后续可继续加职业行为状态机”为前提。
