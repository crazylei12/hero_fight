# 项目基础设定

最后更新：2026-04-22

## 文档用途

这份文档用于定义项目的基础方向。后续任何 AI 在参与开发、写代码、拆任务、补文档时，都必须先阅读本文件，再执行工作。

如果后续设定发生变化，应直接更新本文件，不要依赖聊天记录猜测项目方向。

## 项目目标

这是一个使用 Unity 引擎开发的战斗策略游戏实验项目。

项目长期目标：
- 做出一款核心体验接近《Teamfight Manager》的游戏。
- 支持单人玩法。
- 预留局域网联机与 P2P 联机能力。
- 支持存档。

项目当前目标：
- 第一阶段只做一个最小可运行的竞技场战斗原型。
- 先把英雄自动战斗系统做扎实。
- 建立可持续扩展的英雄开发规范和数据结构。

## 已确认设定

### 平台与技术

- 平台只支持 `Windows`。
- 引擎固定为 `Unity LTS`。
- 编程语言优先使用 `C#`。
- 优先使用 Unity 原生能力。
- 没有明确必要时，不引入重型第三方插件。
- 所有系统尽量数据驱动。

### 项目用途

- 本项目主要用于实验 AI 协作开发能力。
- 玩家范围不需要面向公开市场，主要供作者本人和朋友试玩。

### 游戏大方向

- 单人与联机同等重要。
- 开发顺序上先做单机，联机先预留接口，不在第一阶段实现。
- 核心循环为：
  - 经营队伍
  - BP
  - 自动战斗
  - 赛季推进
- 比赛中完全自动，不做实时人工操作。
- 游戏不强调题材和剧情包装，优先保证纯战斗体验。

### 战斗基础设定

- 基础单位是 `英雄`。
- 第一阶段基线队伍规模为 `5v5`，长期需兼容 `3v3 / 4v4 / 5v5`。
- 单小局目标时长为 `60 秒`。
- 战斗视角和观感方向参考《Teamfight Manager》的对战界面。
- 最终英雄分类固定为六类：
  - 战士
  - 法师
  - 刺客
  - 坦克
  - 辅助
  - 射手

### 第一阶段原型方向

- 第一阶段只做 `本地单机 1v1 对战模拟`。
- 第一阶段地图为 `纯平面竞技场`。
- 第一阶段空间尺度固定使用 `世界单位` 作为基础空间单位，不再按具体场景临时改口径。
- 第一阶段默认导入的 2D 战斗场景贴图按 `100 像素 = 1 世界单位` 处理。
- 第一阶段默认竞技场 `arena_stage01_flat` 的正式尺寸固定为 `48 x 27 世界单位`。
- 第一阶段默认只交付 `5v5` 战斗内容与规则验证；如需为未来 `3v3 / 4v4` 预留结构，应通过模式配置和运行时参数实现，不要把队伍人数写死在战斗输入、出生、UI 和大招判断里。
- 第一阶段只做简单选人，不做 Ban，不做完整 BP。
- 第一批英雄做 `6` 个，对应六个分类各 `1` 个代表英雄。
- 第一阶段每个英雄统一采用：
  - 普攻
  - 一个小技能
  - 一个大招
- 第一阶段小技能采用 `纯 CD`，不做蓝量或能量。
- 第一阶段主动释放型大招采用 `一局一次资源`，不按可重复 CD 技能处理。
- 第一阶段允许先使用简单 AI，但架构上要为分类状态机扩展预留接口。
- 第一阶段统一伤害类型，不区分物理和法术。
- 第一阶段英雄基础属性至少包含：
  - 生命值
  - 攻击力
  - 防御力
  - 攻速
  - 移速
  - 暴击率
  - 暴击伤害
  - 攻击距离
- 第一阶段胜负规则采用：
  - 英雄死亡后等待 `5 秒`复活
  - 复活后满血回出生点
  - 技能 CD 不重置
  - 常规时间 `60 秒`
  - 时间结束比较击杀数
  - 同分进入加时，下一次击杀决胜
- 第一阶段大招必须具备独立于小技能的更强释放逻辑。
- 第一阶段需要统一状态系统和简易战斗日志。

### 第一阶段英雄规范方向

- 英雄数据固定拆成：
  - 基础数据
  - 战斗逻辑配置
  - 表现资源配置
- 第一阶段英雄配置只保留战斗相关字段。
- 英雄 ID 规则采用：`职业前缀 + 三位序号 + 英文名字`
- 代码中职业枚举固定为六类。
- 每个职业至少要有一个默认 AI 行为模板。
- 技能配置必须模板化。
- 普攻配置也必须模板化。
- 第一阶段允许少量特例逻辑，但必须标记并写入文档。
- 第一阶段需要简单标签系统。
- 每个英雄都要有一份说明文档。
- 每个英雄都要有设计边界说明。

### 第一阶段架构方向

- 工程采用四层结构：
  - 数据配置层
  - 运行时逻辑层
  - 表现层
  - UI 层
- 战斗核心逻辑不直接依赖 UI。
- 战斗核心逻辑不直接依赖表现资源。
- 英雄静态配置与运行时实例必须分离。
- 需要统一的战斗事件机制。
- 需要独立的数值结算模块。
- 需要把 AI 决策与执行系统分离。
- 需要独立的统一状态系统模块。
- 选人界面只负责组装战斗输入数据。
- 需要统一的战斗输入配置对象。
- 需要统一的战斗结果对象。
- 需要统一随机数服务。
- 日志系统分为战斗事件日志和调试日志。
- 需要为未来联机预留表现与逻辑解耦边界。
- 需要先规范项目目录。
- 需要明确禁止部分逻辑写在错误层级或 MonoBehaviour 中。
- 核心战斗逻辑尽量可脱离场景对象测试。
- 需要统一战斗管理器作为入口。

### 经营与长期系统方向

- 长期会做经营向内容。
- 除装备/道具系统外，其他经营系统后续都可能纳入。
- 角色差异不仅来自数值成长，还来自行为倾向和风格特性。
- 角色特性举例：
  - 偏进攻
  - 偏保命
  - 偏追击
  - 偏拉扯

### 存档与美术方向

- 存档方式为 `自动存档 + 多手动存档位`。
- 早期开发允许使用占位资源。
- 第一阶段战斗表现方向固定为 `平面 2D / 2.5D 可读性优先`，不以真实 3D 模型表现为前提。
- 第一阶段单位表现默认优先使用 `Sprite`、`SpriteRenderer`、`2D 角色预制体` 或等价的平面资源方案。
- 如果工程里临时使用 3D 几何体、3D prefab、灯光或透视相机，它们只应被视为 `调试占位`，不能反向定义正式美术方向。
- 最终画面目标接近《Teamfight Manager》的完成度和可读性。

## 当前阶段范围

第一阶段不是做完整游戏，而是做一个最小对战系统。

第一阶段必须包含：
- 一个竞技场场景
- 英雄自动战斗
- 英雄开发规范
- 英雄属性编辑器
- 英雄选择界面
- 简易战斗日志

第一阶段明确不做：
- 赛季系统
- 经营系统
- 联机功能本体
- 完整 BP
- 复杂剧情
- 正式美术资源
- 完整战斗回放

## 第一阶段工作原则

- 优先做可运行战斗，不先做外围系统。
- 优先建立统一英雄规范，不接受每个英雄各写一套散乱逻辑。
- 优先保证可调试、可扩展、可加新英雄。
- UI 先够用，不追求最终表现。
- 所有新需求都要先判断是否属于“第一阶段最小对战系统”。
- 所有战斗逻辑都必须兼容死亡复活、60 秒记人头、平局加时下一杀决胜。

## 对后续 AI 的要求

任何 AI 在执行任务前，必须遵守以下规则：

- 不要把范围扩展到完整游戏，除非文档明确解锁该阶段。
- 当前优先级永远高于远期设想。
- 如果某项实现会显著拖慢第一阶段原型，应优先选择更简单的方案。
- 写代码时优先考虑英雄系统的统一规范和后续扩展能力。
- 新增规划、规则、阶段执行文档时，统一放在 `docs/planning/` 下。
- 新增单个英雄的具体设定文档时，统一放在 `docs/heroes/` 下，并按职业子目录管理。
- 新问题、新决策、新阶段范围，必须落到文档中，不要只留在对话里。
