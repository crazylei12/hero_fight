# 第一阶段项目启动检查清单

最后更新：2026-04-11

## 文档用途

这份文档用于指导第一阶段在 Unity 中真正开始落地时的第一批具体动作。

和 `stage-01-first-implementation-plan.md` 不同，这份文档更接近执行清单。它的目标是：
- 让后续 AI 明确当前该做什么
- 让项目从“只有文档”进入“开始搭工程”
- 降低第一次建 Unity 项目时的随意性

## 使用方式

后续 AI 在真正开始搭建 Unity 工程前，应先阅读以下文档：
- `docs/planning/project-foundation.md`
- `docs/planning/stage-01-arena-decisions.md`
- `docs/planning/stage-01-combat-rules-decisions.md`
- `docs/planning/stage-01-status-effect-decisions.md`
- `docs/planning/stage-01-hero-spec-decisions.md`
- `docs/planning/stage-01-architecture-decisions.md`
- `docs/planning/stage-01-first-implementation-plan.md`

然后按本清单执行。

## 当前阶段目标

把项目推进到这样一个状态：
- Unity 工程已创建
- 目录结构已建立
- 基础场景已存在
- 英雄和技能的数据结构已落地
- 战斗主循环骨架已存在
- 可以开始进入第一轮代码实现

## 启动前确认

在开始搭工程前，先确认以下事项：

- [ ] 当前项目目标仍然是“第一阶段最小自动战斗原型”
- [ ] 当前不实现联机功能本体
- [ ] 当前不实现赛季、经营、完整 BP
- [ ] 当前优先做统一系统，不优先做内容堆积
- [ ] 当前优先保证架构清晰、日志可用、后续可扩展
- [ ] 当前战斗表现方向仍然是 `平面 2D / 2.5D 可读性优先`

## 第 1 组：创建 Unity 工程

- [ ] 创建 `Unity LTS` 项目
- [ ] 确认项目可以正常打开、编译、进入编辑器
- [ ] 记录所用 Unity 版本到项目文档中
- [ ] 确认项目平台目标为 `Windows`
- [ ] 暂不引入不必要的第三方插件

完成标准：
- Unity 工程可正常打开
- 没有基础编译错误

## 第 2 组：建立目录结构

在 `Assets` 下至少建立以下目录：

- [ ] `Assets/Scripts/Core`
- [ ] `Assets/Scripts/Battle`
- [ ] `Assets/Scripts/Heroes`
- [ ] `Assets/Scripts/UI`
- [ ] `Assets/Scripts/Data`
- [ ] `Assets/Data/Heroes`
- [ ] `Assets/Data/Skills`
- [ ] `Assets/Prefabs`
- [ ] `Assets/Scenes`
- [ ] `Assets/Art`
- [ ] `Assets/Audio`

建议后续可补：
- [ ] `Assets/Materials`
- [ ] `Assets/Animations`
- [ ] `Assets/VFX`
- [ ] `Assets/UI`

完成标准：
- 目录结构清晰
- 没有把脚本、数据、资源混在一起

## 第 3 组：创建基础场景

至少创建以下场景占位：

- [ ] `MainMenu`
- [ ] `HeroSelect`
- [ ] `Battle`

建议：
- [ ] 把场景统一放在 `Assets/Scenes`
- [ ] 配置 Build Settings，确保能在场景间切换

完成标准：
- 三个场景能被打开
- 项目具备最小的流程骨架

## 第 4 组：建立核心数据脚本

优先创建最小数据定义脚本：

- [ ] `HeroClass`
- [ ] `HeroTag`
- [ ] `HeroStatsData`
- [ ] `BasicAttackData`
- [ ] `SkillData`
- [ ] `HeroVisualConfig`
- [ ] `HeroDefinition`
- [ ] `BattleInputConfig`
- [ ] `BattleResultData`

要求：
- [ ] 数据结构优先放在 `Assets/Scripts/Data`
- [ ] 不要把运行时逻辑混进静态配置结构
- [ ] ScriptableObject 与纯数据结构边界清楚

完成标准：
- 可以在 Unity 中创建英雄定义资产
- 一个英雄定义能容纳第一阶段所需全部战斗字段

## 第 5 组：建立基础枚举和通用类型

优先落地的枚举或基础类型包括：

- [ ] 职业枚举
- [ ] 标签定义
- [ ] 技能类型枚举
- [ ] 目标类型枚举
- [ ] 状态效果类型枚举
- [ ] 阵营枚举
- [ ] 战斗结束原因枚举

要求：
- [ ] 枚举命名稳定
- [ ] 不要后面频繁重命名基础枚举
- [ ] 枚举要服务于第一阶段，不要为了未来过度设计

## 第 6 组：建立战斗主入口骨架

优先创建：

- [ ] `BattleManager`
- [ ] `BattleContext`
- [ ] `BattleClock`
- [ ] `BattleScoreSystem`
- [ ] `BattleEndResolver`
- [ ] `BattleEventBus`
- [ ] `BattleRandomService`

要求：
- [ ] 战斗管理器是统一入口
- [ ] 输入对象和结果对象边界清晰
- [ ] 不要由 UI 直接充当战斗驱动器

完成标准：
- 能够接受一份战斗输入配置并启动基础战斗流程
- 即使还没有单位，也能完成倒计时和结束判定骨架

## 第 7 组：建立运行时英雄实例骨架

优先创建：

- [ ] `RuntimeHero`
- [ ] 英雄当前状态容器
- [ ] 英雄当前位置/目标数据结构
- [ ] 技能 CD 运行时数据结构
- [ ] 复活计时运行时字段

要求：
- [ ] 静态配置和运行时实例必须分离
- [ ] 不要直接修改 HeroDefinition 作为运行时状态

完成标准：
- 能从一个英雄定义创建一个运行时英雄实例

## 第 8 组：建立基础战斗事件流

优先建立关键事件定义：

- [ ] `BattleStart`
- [ ] `UnitSpawned`
- [ ] `TargetChanged`
- [ ] `AttackPerformed`
- [ ] `SkillCast`
- [ ] `DamageApplied`
- [ ] `UnitDied`
- [ ] `UnitRevived`
- [ ] `ScoreChanged`
- [ ] `BattleEnded`

要求：
- [ ] UI 通过事件读取状态变化
- [ ] 日志通过事件记录过程
- [ ] 表现层通过事件触发反馈

完成标准：
- 基础事件可以被订阅和广播

## 第 9 组：建立单位基础循环最小骨架

优先做以下能力的占位实现：

- [ ] 单位生成
- [ ] 单位出生点分配
- [ ] 自动向敌方区域移动
- [ ] 默认索敌
- [ ] 普攻触发入口
- [ ] 生命值减少
- [ ] 死亡
- [ ] 5 秒复活
- [ ] 回出生点重新加入战斗

要求：
- [ ] 先保证循环闭合
- [ ] 不要一开始就追求复杂职业差异
- [ ] 不要因为临时调试方便，就把单位表现路线默认带成 3D 模型游戏

完成标准：
- 即使只有占位资源，也能看到 5v5 基础循环跑起来

## 第 10 组：建立数值与状态系统骨架

优先创建：

- [ ] `DamageResolver`
- [ ] `HealResolver`
- [ ] `CriticalResolver`
- [ ] `StatusEffectSystem`
- [ ] `StatusEffectInstance`

要求：
- [ ] 数值结算独立
- [ ] 暴击通过统一随机数服务处理
- [ ] 状态系统不要写死在技能脚本里
- [ ] 周期状态使用统一 tick 入口
- [ ] 硬控统一走行为门禁判断

完成标准：
- 普攻伤害能走统一结算
- `眩晕` 和 `击飞` 有统一入口
- `持续伤害` / `持续治疗` / `无敌` / `不可选中` / `分担伤害` 能通过统一状态机制表达

## 第 11 组：建立技能系统骨架

优先创建：

- [ ] 技能释放入口
- [ ] 技能目标选择接口
- [ ] 技能 CD 管理
- [ ] 小技能释放条件
- [ ] 大招释放条件
- [ ] 技能执行基础流程

要求：
- [ ] 小技能和大招走统一技能系统
- [ ] 但大招释放判断必须独立于小技能
- [ ] 先做最小技能效果池
- [ ] 技能特效的范围、持续时间、脉冲频率等表现参数绑定技能正式参数，不额外漂移

最小技能效果池：
- [ ] 单体伤害
- [ ] 范围伤害
- [ ] 单体治疗
- [ ] 增益 Buff / 属性上升
- [ ] 减益 Debuff / 属性下降
- [ ] 持续伤害
- [ ] 持续治疗
- [ ] 位移
- [ ] 眩晕 / 击飞

完成标准：
- 一个英雄可以通过统一配置释放技能并产生结果
- 技能表现不会与技能结算范围、持续时间、脉冲频率长期脱节

## 第 12 组：建立职业 AI 模板骨架

优先创建：

- [ ] AI 决策接口
- [ ] 战士模板
- [ ] 法师模板
- [ ] 刺客模板
- [ ] 坦克模板
- [ ] 辅助模板
- [ ] 射手模板

要求：
- [ ] 决策与执行分离
- [ ] 默认索敌规则可被职业覆盖
- [ ] 大招释放条件可由职业或英雄扩展

完成标准：
- 不同职业开始呈现基础行为差异

## 第 13 组：创建首批 6 个英雄占位配置

优先为以下六类各创建一个英雄定义：

- [ ] 战士
- [ ] 法师
- [ ] 刺客
- [ ] 坦克
- [ ] 辅助
- [ ] 射手

要求：
- [ ] 每个英雄都使用统一配置结构
- [ ] 每个英雄都具备普攻、小技能、大招
- [ ] 每个英雄都绑定默认职业 AI 模板
- [ ] 每个英雄都补说明文档

完成标准：
- 六个英雄定义资产都存在
- 可以进入后续战斗调试阶段

## 第 14 组：建立最小 UI 通路

优先创建：

- [ ] HeroSelect 基础界面
- [ ] 双方英雄选择入口
- [ ] 战斗启动按钮
- [ ] Battle HUD 占位
- [ ] 击杀数显示
- [ ] 倒计时显示
- [ ] Result 界面占位

要求：
- [ ] UI 只读核心状态或提交输入
- [ ] 不直接承担战斗规则判定

完成标准：
- 可以完成“选人 -> 开战 -> 结束 -> 结果”通路

## 第 15 组：建立日志与调试入口

优先创建：

- [ ] 战斗事件日志记录器
- [ ] 调试日志接口
- [ ] 最小日志输出面板或控制台输出方案

要求：
- [ ] 先能追踪关键事件
- [ ] 后续再考虑更好看的日志面板

完成标准：
- 出现战斗异常时能从日志初步定位问题

## 第一批真正动手时的推荐顺序

推荐按以下顺序开始实现：

1. [ ] 创建 Unity 工程
2. [ ] 建立目录和场景
3. [ ] 写数据结构和 ScriptableObject
4. [ ] 写 BattleManager 与 BattleInput / BattleResult
5. [ ] 写 RuntimeHero
6. [ ] 写 BattleEventBus 与 BattleClock
7. [ ] 写基础单位出生、移动、索敌、普攻
8. [ ] 写死亡、复活、计分、加时
9. [ ] 写数值结算和状态系统
10. [ ] 写技能系统
11. [ ] 写职业 AI 模板
12. [ ] 创建 6 个英雄配置
13. [ ] 串 HeroSelect 和 Battle HUD
14. [ ] 补日志和调试输出

## 明确不要在启动阶段做的事

- [ ] 不要先做联网房间
- [ ] 不要先做赛季推进
- [ ] 不要先做经营系统
- [ ] 不要先做复杂英雄特例
- [ ] 不要先做正式美术
- [ ] 不要先做高级镜头表现
- [ ] 不要把调试用 3D 几何体、3D prefab 或透视相机误当成第一阶段正式表现路线
- [ ] 不要先做复杂 UI 动效
- [ ] 不要在没有统一模板前快速堆 20 个英雄

## 启动阶段完成标志

当下面这些条件全部满足时，可以认为“第一阶段项目已经完成真正启动”：

- [ ] Unity 工程已建立
- [ ] 目录规范已落地
- [ ] 场景骨架已存在
- [ ] 数据结构已落地
- [ ] 战斗主入口已存在
- [ ] 基础事件流已存在
- [ ] 运行时英雄实例已存在
- [ ] 单位基础循环已跑通
- [ ] 数值与状态系统骨架已存在
- [ ] 技能系统骨架已存在
- [ ] 至少 6 个英雄配置资产已创建
- [ ] 最小 UI 通路可用
- [ ] 日志系统可用

## 对后续 AI 的执行要求

后续 AI 开始真正创建工程时：
- 必须先读规划文档，不得跳过
- 必须按当前阶段目标做事，不得提前扩范围
- 每完成一项关键内容，应同步更新文档或进度记录
- 若发现文档与实现冲突，应先修正文档或提出冲突点，不要私自改方向
