# 第一阶段架构决策

最后更新：2026-04-11

## 文档用途

这份文档记录第一阶段原型在工程结构、代码分层和模块边界上的正式决策。后续 AI 在搭建 Unity 工程、创建脚本目录、实现战斗系统和 UI 时，必须以本文件为直接依据。

如果后续架构规则变化，应直接修改本文件，而不是继续依赖问题清单中的原始回答。

## 第一阶段架构目标

第一阶段架构的目标是：
- 保证战斗逻辑、表现层、UI 层解耦
- 保证配置数据和运行时状态分离
- 保证后续可以继续扩职业状态机、联机和更多英雄
- 避免“能跑但难维护”的场景脚本堆叠式实现
- 为 AI 连续接手开发提供稳定结构

## 总体分层

第一阶段明确采用四层结构：
- `数据配置层`
- `运行时逻辑层`
- `表现层`
- `UI层`

### 数据配置层

负责静态配置和资源引用，例如：
- ScriptableObject
- JSON
- 英雄静态配置
- 技能静态配置
- 地图或对战参数配置

### 运行时逻辑层

负责核心战斗规则，例如：
- 战斗流程推进
- AI 决策
- 技能释放逻辑
- 伤害与治疗结算
- 状态系统
- 胜负判定
- 计时与加时

### 表现层

负责视觉与听觉呈现，例如：
- 动画
- 特效
- 投射物表现
- 受击反馈
- 音效

### UI 层

负责用户交互和信息显示，例如：
- 选人界面
- 战斗 HUD
- 日志面板
- 结果界面

## 核心解耦规则

### 战斗核心不直接依赖 UI

- 战斗核心逻辑不得直接依赖具体 UI。
- UI 只作为观察者、读取者或输入发起者。
- 战斗胜负判断、计分、状态变化不能写死在 UI 脚本中。

### 战斗核心不直接依赖表现资源

- 战斗逻辑先决定“发生了什么”，再由表现层决定“怎么表现”。
- 逻辑层不能把特效播放、动画是否完成作为规则前提。
- 表现层应消费逻辑事件，而不是反向控制逻辑结算。

补充约束：
- 表现层中的技能特效参数必须优先读取技能运行时或技能静态配置中的正式参数，例如 `范围半径`、`持续时间`、`tickIntervalSeconds`
- 不允许在表现层长期额外维护一套脱离技能配置的范围、时长、触发频率参数
- 如果特效资源的原始尺寸与技能参数不一致，应在表现层建立统一换算函数或适配器，而不是让每个技能单独写死经验值

补充结构约束：
- 英雄本体级资源与技能级资源必须分开归属
- 英雄换技能时，应主要通过替换技能配置实现，不应要求回头修改英雄本体里的技能专属表现字段
- 技能专属表现资源应优先挂在技能配置或技能专属表现配置上，而不是长期堆在英雄总配置上
- `BattleView` 这类总视图脚本只能承担通用同步与分发职责，不应长期沉积大量按 `heroId` 或 `skillId` 分支的复杂特效实现
- 如果某个技能需要复杂表现编排，应拆到独立的表现控制脚本，而不是继续扩写总视图入口

## 配置与运行时分离

### 英雄静态配置与运行时实例必须分离

- 静态配置是英雄模板，不可在战斗中直接修改。
- 运行时实例保存本局战斗中的临时状态，例如：
  - 当前生命值
  - 技能 CD
  - 当前位置
  - 当前目标
  - 当前状态效果
  - 是否死亡
  - 是否正在复活中

说明：
- 这是复活、状态系统、联机同步、复战和统计的基本前提。

## 事件机制

第一阶段必须建立统一的战斗事件机制。

事件示例包括：
- `OnBattleStart`
- `OnUnitSpawned`
- `OnTargetChanged`
- `OnAttackPerformed`
- `OnSkillCast`
- `OnDamageApplied`
- `OnUnitDied`
- `OnUnitRevived`
- `OnScoreChanged`
- `OnBattleEnded`

用途：
- UI 通过事件刷新 HUD
- 日志系统记录关键事件
- 表现层播放动画、特效和音效
- 调试系统可捕获内部流程

补充要求：
- 表现层应优先消费 `技能释放`、`持续区域创建`、`持续区域脉冲`、`命中` 等事件来驱动特效
- 周期性伤害或治疗特效的重播节奏，应与对应运行时事件或对应技能的 `tickIntervalSeconds` 保持一致

约束：
- 不优先使用大量脚本之间的直接相互调用替代统一事件流。

## 关键独立模块

### 数值结算模块

第一阶段必须把数值结算独立成单独模块。

至少负责：
- 伤害计算
- 暴击判定
- 防御减伤计算
- 治疗计算
- Buff / Debuff 修正

约束：
- 不允许把这些结算逻辑分散写死在各个技能脚本和攻击脚本中。

### AI 决策模块与执行模块分离

第一阶段必须把“AI 决策”与“角色执行”分离。

建议理解：
- 决策模块负责：
  - 选择目标
  - 选择释放哪个技能
  - 判断是否该移动
- 执行模块负责：
  - 移动
  - 普攻
  - 施法
  - 切换目标

说明：
- 后续扩职业状态机时，这个分离是必要前提。

### 状态系统模块

第一阶段必须有独立的统一状态系统模块。

至少负责：
- 状态添加
- 状态持续时间管理
- 状态叠加规则
- 状态移除
- 状态对属性或行为的影响

说明：
- 状态系统属于运行时逻辑层，不应散落在各技能特例实现中。

## 选人与战斗入口

### 选人界面的职责

选人界面只负责组装战斗输入数据。

选人界面不应：
- 直接驱动战斗内部对象
- 直接改写战斗核心状态
- 自己拼接内部运行时对象关系

选人界面应输出的信息包括：
- 我方阵容
- 敌方阵容
- 地图配置
- 时间限制
- 日志开关
- 其他对战参数

### 战斗输入配置对象

第一阶段必须有统一的战斗输入配置对象。

它负责集中描述一场战斗启动所需的输入，例如：
- 我方阵容
- 敌方阵容
- 出生点布局
- 时间限制
- 是否开启日志
- 其他实验参数

说明：
- 这对调试、复战、未来联机和自动测试都很重要。

### 战斗结果对象

第一阶段必须有统一的战斗结果对象。

它至少应包含：
- 获胜方
- 双方击杀数
- 是否进入加时
- 各英雄统计
- 日志摘要

说明：
- 结果界面、后续存档、复盘和统计都应读取统一结果对象。
- 不允许由 UI 自己临时拼接结果结构作为长期方案。

## 随机数入口

第一阶段必须有统一随机数服务。

随机数服务至少用于：
- 暴击
- 随机目标选择或打散逻辑
- 技能散布
- 后续 AI 随机倾向

说明：
- 不建议在各处直接散用 `Random`。
- 统一随机数入口更利于调试、复战和未来联机扩展。

## 日志系统结构

第一阶段日志系统需要分成两类：
- `战斗事件日志`
- `调试日志`

### 战斗事件日志

用于记录玩家可读或开发者可读的战斗过程，例如：
- 战斗开始
- 技能释放
- 击杀发生
- 进入加时
- 战斗结束

### 调试日志

用于记录更细的内部信息，例如：
- AI 为什么选择该目标
- 某技能为什么未释放
- 某状态为什么被移除

说明：
- 第一阶段即使调试日志不对玩家开放，架构上也必须预留。

## 联机预留边界

第一阶段必须为未来联机预留结构边界，但不实现联机功能本体。

预留原则：
- 战斗结果尽量由逻辑层决定
- 表现层只消费逻辑结果
- UI 不直接改战斗核心状态
- 未来可以考虑同步输入、同步状态或同步事件

说明：
- 第一阶段不是做联机，但必须避免把逻辑、UI、表现写死到无法同步的程度。

## 项目目录规范

第一阶段应先规范项目目录。

建议目录方向：
- `Assets/Scripts/Core`
- `Assets/Scripts/Battle`
- `Assets/Scripts/Heroes`
- `Assets/Scripts/UI`
- `Assets/Scripts/Data`
- `Assets/Data/Heroes`
- `Assets/Data/Skills`
- `Assets/Prefabs`
- `Assets/Scenes`
- `Assets/Art`
- `Assets/Audio`

说明：
- 后续 AI 必须尽量遵守目录规范，不要随意新增风格混乱的目录。

## 明确禁止的写法

第一阶段明确要求一些逻辑不要直接写在 MonoBehaviour 里或不要写死在错误层级。

至少包括：
- 纯数值结算不要全部写在 MonoBehaviour 中
- 英雄静态配置不要散落在场景对象里
- 战斗胜负判断不要写死在 UI 脚本里
- UI 脚本不要直接充当战斗核心控制器
- 表现对象不要直接充当规则裁判

说明：
- 给 AI 明确负面约束很重要。
- 第一阶段可以简单，但不能野生生长。

## 可测试性要求

第一阶段要求核心战斗逻辑尽量可脱离场景对象测试。

含义：
- 不一定立刻写完整自动化测试
- 但核心结算、状态、战斗结果判定、部分 AI 决策应尽量不完全依赖 Unity 场景生命周期

说明：
- 这会显著降低后续重构成本。
- 也更适合 AI 编写和验证纯逻辑模块。

## 战斗管理器

第一阶段需要统一的战斗管理器作为入口。

战斗管理器负责：
- 接收战斗输入配置对象
- 创建并初始化战斗
- 启动战斗流程
- 推进战斗计时与状态
- 监听结束条件
- 产出战斗结果对象

说明：
- 它会成为选人界面、战斗系统、日志系统、结果界面和未来联机之间的重要枢纽。

## 对后续 AI 的约束

后续 AI 在第一阶段搭建工程时，必须遵守以下规则：
- 优先按四层结构组织代码和资源。
- 不要把 UI、表现和战斗逻辑写死到一起。
- 不要混淆英雄静态配置和运行时状态。
- 不要省略统一事件机制、统一数值结算模块、统一状态系统。
- 不要让选人界面直接操作战斗内部实例。
- 不要在项目早期无序扩目录。
- 所有重要核心模块都要尽量为未来联机和更复杂职业行为预留边界。
