# 第一阶段首轮实施计划

最后更新：2026-04-17

## 文档用途

这份文档用于指导后续 AI 和开发过程中的具体落地顺序。

前面的文档已经定义了项目目标、战斗规则、英雄规范和架构边界；这份文档要解决的问题是：
- 现在实际应该先做什么
- 哪些任务必须先完成
- 哪些任务不要过早开始
- 从空 Unity 工程到第一个可运行战斗原型的推荐顺序是什么

这份文档面向执行，不面向讨论。

## 当前总目标

第一阶段的当前目标不是做完整游戏，而是做出一个最小可运行的 5v5 自动战斗竞技场原型。

完成标准：
- 可以从选人界面选择双方英雄
- 进入战斗场景
- 双方英雄自动战斗
- 支持普攻、小技能、大招
- 支持死亡、复活、重新投入战斗
- 支持 60 秒计时、击杀计分、加时决胜
- 支持基础战斗日志
- 可以结算并显示结果

## 执行原则

- 先搭框架，再补内容。
- 先做通路，再做细节。
- 先保证逻辑可运行，再补表现。
- 先做统一系统，再做英雄个性。
- 第一批英雄只做够验证系统，不做花哨扩展。

## 当前不该做的事

在首轮实现中，以下内容明确不要提前开始：
- 联机功能本体
- 赛季系统
- 经营系统
- 完整 BP
- 高质量美术资源
- 正式音效资源
- 完整回放系统
- 复杂战术状态机
- 大量新英雄扩充

## 推荐实施顺序

### 阶段 A：建立工程骨架

目标：创建一个干净、可持续扩展的 Unity 项目基础结构。

任务：
- 创建 Unity LTS 项目
- 建立基础目录结构
- 创建最小场景结构
- 建立脚本命名空间或基础组织规则
- 建立规划文档与项目目录的一致关系

建议优先创建的目录：
- `Assets/Scripts/Core`
- `Assets/Scripts/Battle`
- `Assets/Scripts/Heroes`
- `Assets/Scripts/UI`
- `Assets/Scripts/Data`
- `Assets/Data/Heroes`
- `Assets/Data/Skills`
- `Assets/Prefabs`
- `Assets/Scenes`
- `Assets/Art`
- `Assets/Audio`

完成标准：
- Unity 工程能正常打开
- 目录结构已经就位
- 至少有 `MainMenu`、`HeroSelect`、`Battle` 三个基础场景占位

### 阶段 B：先建立静态数据结构

目标：把英雄和技能的配置结构先定下来，为后续一切运行时逻辑提供输入。

任务：
- 创建职业枚举
- 创建英雄标签结构
- 创建英雄基础属性数据结构
- 创建普攻配置结构
- 创建技能配置结构
- 创建英雄总配置结构
- 创建表现资源引用结构
- 创建战斗输入配置结构
- 创建战斗结果结构

建议先完成的数据类或 ScriptableObject：
- `HeroClass`
- `HeroTag`
- `HeroStatsData`
- `BasicAttackData`
- `SkillData`
- `HeroDefinition`
- `HeroVisualConfig`
- `BattleInputConfig`
- `BattleResultData`

完成标准：
- 可以在 Unity Inspector 中创建一个英雄配置资产
- 一个英雄配置能完整描述第一阶段战斗所需信息

### 阶段 C：建立战斗运行时核心骨架

目标：先让一场战斗有“开始、推进、结束”的主干，而不是先做具体技能。

任务：
- 创建战斗管理器
- 创建战斗上下文或战斗状态容器
- 创建运行时英雄实例结构
- 创建战斗事件机制
- 创建统一随机数服务
- 创建基础计时系统
- 创建比分系统
- 创建加时判定流程
- 创建战斗结束流程

建议优先实现的运行时类：
- `BattleManager`
- `BattleContext`
- `BattleClock`
- `BattleScoreSystem`
- `BattleEventBus`
- `BattleRandomService`
- `RuntimeHero`
- `BattleEndResolver`

完成标准：
- 可以用假数据启动一场空战斗
- 战斗可以计时到 60 秒并正确结束
- 击杀数和加时机制有明确入口

### 阶段 D：建立单位基础生存循环

目标：先让英雄“出生、移动、攻击、死亡、复活”跑起来。

任务：
- 创建出生点系统
- 创建单位生成流程
- 创建单位移动逻辑
- 创建默认索敌逻辑
- 创建普攻执行逻辑
- 创建生命值系统
- 创建死亡处理
- 创建复活流程
- 创建重新入场逻辑

说明：
- 这一阶段先不追求职业差异最大化。
- 先保证基础循环可靠。

完成标准：
- 5v5 双方单位能出生并接敌
- 单位会自动索敌和普攻
- 单位死亡后 5 秒复活并回到出生点
- 复活后能重新加入战斗

### 阶段 E：建立数值结算与状态系统

目标：把伤害、暴击、防御、治疗、属性 Buff / Debuff、眩晕、击飞、持续伤害 / 治疗、无敌、不可选中、分担伤害这些底层规则统一起来。

任务：
- 创建数值结算模块
- 创建伤害计算流程
- 创建暴击计算流程
- 创建防御减伤规则
- 创建治疗流程
- 创建统一状态系统
- 创建状态添加/移除/持续时间刷新机制
- 创建状态 tick 机制
- 实现硬控：`眩晕`、`击飞`
- 实现至少一种属性 Buff 和一种属性 Debuff
- 实现 `持续伤害` 与 `持续治疗`
- 实现 `无敌` 与 `不可选中`
- 实现 `分担伤害`

建议优先实现的模块：
- `StatResolver`
- `DamageResolver`
- `HealResolver`
- `CriticalResolver`
- `StatusEffectSystem`
- `StatusEffectInstance`

完成标准：
- 普攻和技能都能复用统一结算逻辑
- 眩晕与击飞能稳定影响英雄行为
- Buff / Debuff、持续伤害 / 治疗、无敌、不可选中、分担伤害不需要靠技能脚本各自硬写状态逻辑

### 阶段 F：建立技能系统

目标：让小技能和大招基于统一框架运行。

任务：
- 创建技能释放入口
- 创建技能目标选择逻辑
- 创建小技能释放规则
- 创建大招释放规则
- 创建技能 CD 系统
- 创建统一技能执行流程
- 创建通用连续动作序列执行能力
- 创建技能效果基础类型
- 创建可配置的大招释放决策模板

第一阶段最小技能效果池：
- 单体伤害
- 范围伤害
- 单体治疗
- 增益 Buff / 属性上升
- 减益 Debuff / 属性下降
- 持续伤害
- 持续治疗
- 位移
- 眩晕 / 击飞控制

说明：
- 小技能与大招必须走同一技能系统，但要保留不同释放判断逻辑。
- 大招优先级和释放时机不能写成“小技能 CD 好了就放”。
- 大招释放逻辑优先采用模板化条件与兜底规则，不按职业硬编码。
- 如果某个技能需要在短时间内连续执行多次普攻或多次重复释放同一技能，应优先走共享的“连续动作序列”入口，不要直接写成单英雄专属循环。
- 这套序列能力至少要能承接：重复次数、动作间隔、目标失效后的刷新策略，以及执行期间的临时参数覆写，例如临时射程。
- 与状态系统相关的正式范围见 `docs/planning/stage-01-status-effect-decisions.md`。
- `位移` 继续作为技能效果保留；它与 `击飞` 的边界也统一在上述状态文档中约束。

完成标准：
- 任意英雄都可以通过统一配置释放小技能和大招
- 至少已有一类技能通过共享的连续动作序列入口运行，而不是靠单英雄特例循环
- 大招具备单独的时机判断

### 阶段 G：建立职业 AI 模板

目标：先做每个职业的默认行为模板，而不是先做每个英雄的复杂个性脚本。

任务：
- 创建 AI 决策接口
- 创建基础决策流程
- 创建战士默认模板
- 创建法师默认模板
- 创建刺客默认模板
- 创建坦克默认模板
- 创建辅助默认模板
- 创建射手默认模板
- 为 `usesProjectile = true` 的英雄接入统一的受压后撤判断

说明：
- 第一版可以简单。
- 关键是架构上已经允许职业覆盖默认索敌和技能释放时机。
- 这条后撤判断只做“短时拉开危险 threat”的最小版本，不扩成完整仇恨系统或完整走 A 状态机。

完成标准：
- 六个职业在目标选择和技能时机上开始呈现基础差异
- 投射物普攻英雄在面对近身威胁时，已经具备统一的最小后撤能力

### 阶段 H：创建首批 6 个模板英雄

目标：用最少数量的英雄验证最多系统能力。

任务：
- 创建 6 个英雄配置资产
- 为每个英雄填写基础属性
- 为每个英雄配置普攻
- 为每个英雄配置小技能
- 为每个英雄配置大招
- 为每个英雄绑定职业 AI 模板
- 为每个英雄补一份说明文档

首批英雄必须覆盖：
- 近战普攻
- 远程投射物
- 范围伤害
- 位移
- 治疗或增益
- 至少一种硬控（`眩晕` 或 `击飞`）
- 大招范围判断

完成标准：
- 6 个英雄都能进入战斗并完整执行自己的普攻和技能逻辑

### 阶段 I：建立最小 UI 通路

目标：把“选择英雄 -> 进入战斗 -> 查看结果”完整串起来。

任务：
- 创建英雄选择界面
- 支持选择双方阵容
- 将选人结果写入战斗输入对象
- 从选人界面进入战斗场景
- 创建基础战斗 HUD
- 显示倒计时
- 显示双方击杀数
- 创建基础结果界面

说明：
- 这一阶段 UI 只要求可用，不要求美术完整。

完成标准：
- 玩家能够操作一整轮最小流程

### 阶段 J：建立日志与调试面板

目标：保证第一阶段原型可调试、可解释。

任务：
- 创建战斗事件日志记录器
- 创建调试日志接口
- 创建基础日志面板或输出方式
- 记录关键事件
- 支持快速查看胜负原因和关键技能触发

完成标准：
- 能看到一场战斗的重要过程
- 遇到英雄异常时可以从日志快速定位问题

## 推荐首轮开发里程碑

### 里程碑 1：空战斗骨架

完成条件：
- 有基础项目目录
- 有战斗管理器
- 有战斗输入对象
- 有战斗结果对象
- 有计时和结束逻辑

### 里程碑 2：单位基础循环跑通

完成条件：
- 5v5 单位能出生
- 能移动、索敌、普攻
- 能死亡和复活
- 能统计击杀数

### 里程碑 3：技能与状态系统跑通

完成条件：
- 小技能和大招都能释放
- 统一数值结算可用
- 眩晕 / 击飞、持续伤害 / 治疗、无敌、不可选中可用
- 大招有单独释放逻辑

### 里程碑 4：首批 6 英雄可玩

完成条件：
- 六个职业模板英雄全部配置完成
- 战斗差异明显
- 无需手动改代码就能替换或调整英雄配置

### 里程碑 5：最小玩法通路完成

完成条件：
- 能选人
- 能开战
- 能结算
- 能看日志

## 建议的脚本优先级

后续 AI 真正开始写代码时，建议优先顺序如下：

1. `BattleInputConfig`
2. `BattleResultData`
3. `HeroDefinition`
4. `SkillData`
5. `RuntimeHero`
6. `BattleManager`
7. `BattleEventBus`
8. `BattleClock`
9. `BattleScoreSystem`
10. `DamageResolver / HealResolver / CriticalResolver`
11. `StatusEffectSystem`
12. `BasicAttackSystem`
13. `SkillExecutionSystem`
14. `AiDecisionSystem`
15. `HeroSelect UI`
16. `Battle HUD`
17. `Result UI`
18. `Battle Log`

## 给后续 AI 的明确执行指令

后续 AI 如果开始实际搭建工程，应该遵守以下顺序：
- 先建目录和数据结构
- 再建战斗主循环
- 再建生存循环
- 再建数值和状态系统
- 再建技能系统
- 再建职业 AI 模板
- 再创建 6 个模板英雄
- 最后补最小 UI 通路

不要反过来先做：
- 花哨 UI
- 联机结构细节
- 大量英雄内容
- 美术表现打磨
