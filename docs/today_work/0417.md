这样做的目的：

- 英雄基础属性和技能细节不会混在一起

- 先保证“批量调数”和“追加配置”不会静默造脏数据

### 这次我实际做了什么

### 1. 给 SkillData 补了通用序列配置

我新增了：

当前这套配置支持的核心字段包括：

  - `BasicAttack`
  - `SourceSkill`

### 2. 给 RuntimeHero 补了序列运行时状态

我新增了：

- `game/Assets/Scripts/Heroes/RuntimeCombatActionSequence.cs`

并在：

- `game/Assets/Scripts/Heroes/RuntimeHero.cs`

里接入了以下能力：

- 当前是否存在激活中的动作序列
- 序列剩余次数
- 序列节拍推进
- 序列期间的临时普攻射程覆写
- 死亡时清空序列
- 复活重置时清空序列
- 硬控或强制位移时中断序列

这轮还顺手补了一个很重要的行为收口：

- 当前若单位在前摇期间吃到硬控，`RuntimeHero.Tick()` 会清掉待执行动作

这一步不是额外扩功能，而是为了避免后续连续动作序列出现：

- “已经被控住了，但前摇结束后动作仍然照常结算”

### 3. 新增了共享序列推进系统

我新增了：

- `game/Assets/Scripts/Battle/BattleCombatActionSequenceSystem.cs`

它的职责是：

- 在战斗循环中推进当前序列
- 按配置决定下一跳执行的是：
  - 普攻
  - 还是同一个技能
- 处理目标刷新
- 处理目标切换事件
- 在序列结束或中断时清掉运行时状态

当前接入点是：

- `game/Assets/Scripts/Battle/BattleSimulationSystem.cs`

顺序上，序列推进发生在：

1. 先处理待结算动作
2. 再判断是否仍在动作锁 / 强制位移
3. 然后优先推进激活中的连续动作序列
4. 最后才回到普通的“放技能 / 找目标 / 普攻”流程

这个顺序的目的很明确：

- 一旦某个技能已经启动了连续动作序列，后续节拍应优先由序列驱动，而不是让普通 AI 决策把它打断

### 4. 把重复普攻和重复技能都接到了统一入口

这轮没有为射手单独写一套“连射脚本”。

而是分别在：

- `game/Assets/Scripts/Battle/BattleBasicAttackSystem.cs`
- `game/Assets/Scripts/Battle/BattleSkillSystem.cs`

上补了共享入口。

当前实现语义：

- 如果序列载荷是 `BasicAttack`
  - 每一跳会复用现有普攻命中、投射物、伤害 / 治疗结算路径
  - 但不会重复吃普通普攻 CD
- 如果序列载荷是 `SourceSkill`
  - 每一跳会复用现有技能目标选择和技能效果结算路径
  - 但不会重复吃技能冷却
  - 也不会在每一跳再次递归启动新的动作序列

这条“禁止递归再启动序列”是刻意加的保护：

- 否则以后某个带序列的技能如果被序列重复释放，会无限嵌套

### 5. 目标刷新策略已经进了共享契约

当前序列目标策略支持：

- `KeepCurrentTarget`
  - 目标一旦失效，序列直接结束
- `RefreshOnInvalid`
  - 当前目标失效时再刷新一个新目标
- `RefreshEveryIteration`
  - 每一跳都重新评估目标

这套策略已经同时支持：

- 重复普攻
- 重复技能

不是只在射手那边有效。

### 6. 临时射程覆写已经进入正式运行时

这轮已经把“序列期间临时覆写射程”做进了运行时，而不是停留在文档里。

当前已落地的是：

- `RuntimeHero.AttackRange`
  - 若当前存在激活中的序列，且序列声明了 `temporaryBasicAttackRangeOverride`
  - 会返回更大的临时射程

技能侧也已经预留：

- `temporarySkillCastRangeOverride`

并在序列重复施法时走单独的 `effectiveCastRange` 计算。

所以从系统角度看：

- 后续 `Longshot` 大招要做“全屏普攻射程”
- 以及未来某些英雄要做“序列期间临时加大技能施法范围”

这两类需求现在都已经有统一挂点。

### 这轮已经刻意规避的风险

### 1. 没把序列逻辑写成射手专属脚本

这轮最重要的约束就是：

- 不先做 `Longshot` 的孤立特例

### 2. 重复施法不会反复吃冷却

当前序列里重复技能时：

- 只在首次主动释放时正常吃 CD / 记录大招已释放
- 后续序列节拍中的重复释放只复用效果结算，不再重复扣 CD

这是刻意保留的统一语义：

- “动作序列是一次技能释放后的后续执行”

当前重复技能步骤会带：

- `suppressActionSequenceTrigger = true`

所以不会出现：

- 技能 A 启动序列
- 序列里又重复技能 A
- 重复技能 A 又启动一条新的序列

- 目标死亡就失效

- 需要刷新目标时，仍走现有目标选择逻辑

所以这轮没有偷偷引入：

- “序列期间无视目标合法性”

## 0417 续修：连续动作序列边界语义收口

这部分记录的是在审查意见基础上，对“通用连续动作序列能力”做的后续收口。

### 这轮一句话结论

- `0417check.md` 里列出的边界问题已经按运行时语义补了一轮。
- 当前序列系统不再只是“主路径能跑”，而是已经补上了：
  - 固定次数 / 固定持续时间两种重复模式
  - `intervalSeconds` 与前后摇的明确分工
  - 零时长位移 / 瞬时 dash 对序列的真实打断
  - `interruptFlags = None` 时当前步取消不再白白烧掉一次次数
  - 初次选目标时对序列临时射程覆写的消费

### 这次我实际收了什么

### 1. 补上了“固定持续时间重复”模式

这轮在 `CombatActionSequenceData` 里新增了：

- `repeatMode`
  - `FixedCount`
  - `FixedDuration`
- `durationSeconds`

现在这套能力已经同时支持：

- 固定次数重复
- 固定持续时间内重复

这意味着：

- 之前文档里要求的“通用连续动作序列至少支持次数 / 持续时间两种表达”，现在已经和代码契约一致

### 2. 明确了 `intervalSeconds` 的运行时语义

这轮把序列节拍改成了：

- 每次动作入队后，先等待这一步动作真正结束
- 然后才开始计算 `intervalSeconds`

也就是说现在的语义是：

- `windupSeconds + recoverySeconds`
  - 负责这一步动作自身的前后摇
- `intervalSeconds`
  - 负责“当前动作彻底结束后，到下一步动作开始前”的额外间隔

这样改完后：

- 不会再出现“`intervalSeconds` 写得比前后摇短就被静默吃掉”的问题

### 3. `interruptFlags = None` 时，当前步取消会返还次数

这轮给待执行动作补了 `IsActionSequenceStep` 语义，并在 `RuntimeHero.ClearCombatActionState()` 里做了回收。

当前行为变成：

- 如果某一步序列动作还在前摇 / 待触发阶段
- 又因为硬控或强制位移被取消
- 且整段序列本身没有被配置成因此终止

那么：

- 当前这一步会被取消
- 但固定次数模式下，这次机会会返还
- 序列会等待单位重新恢复可行动作，再继续后续步骤

这一步同时避免了另一类问题：

- `interruptFlags = None` 时，序列不会在硬控期间反复空转入队然后又被下一帧清掉

### 4. 强制位移打断已经覆盖到瞬时位移

这轮不再只依赖：

- `IsUnderForcedMovement`

而是补了一个短暂保留的“最近刚发生过强制位移”运行时标记。

现在以下路径都会留下序列可观察到的打断信号：

- 普通持续性强制位移
- `durationSeconds <= 0` 的瞬时位移
- dash 自身的瞬时位移重定位

同时：

- 瞬时位移现在也会先清掉当前待执行动作

所以文档里说的“被强制位移打断序列”，现在不再只对持续位移成立。

### 5. 初次选目标时也会吃到序列临时射程覆写

这轮在技能侧补了更明确的“技能初次选目标有效射程”计算。

当前如果一个技能自己带了 `actionSequence`：

- 初次选目标时，会取 `skill.castRange`
- 以及序列里的临时技能射程覆写
- 对于 `BasicAttack` 载荷，还会一起参考临时普攻射程覆写

这一步的直接目的就是避免：

- 技能明明开出来后是全屏连射
- 但初次找目标时仍然被旧 `castRange` 卡死

所以后续给 `Longshot` 接大招时：

- 即使忘了同步手改一个更大的 `castRange`
- 运行时也不会再天然和序列期射程脱节

### 6. 顺手收掉了 `DensestEnemyArea` 的空目标判断歧义

这轮把“哪些技能允许暂时没有 `primaryTarget`”的语义抽成了单独 helper。

当前结论是：

- 代码路径和设计意图已经一致
- 后续如果继续扩 area-anchor 型技能，不会再因为这块判断顺序让代码读起来像是“支持空目标”，实际又提前被 validator 吃掉

这轮中途第一次编译校验里，Unity 曾吐出一段 `BalanceSheetService` 相关的旧编译阻塞日志。

## 0417 审查收口：Shadowstep 大招 decision 语义与注释补齐

这部分记录的是在后续 review 指出 Shadowstep 大招实现里的几处隐含语义后，我做的一轮小范围收口。

### 这次我实际收了什么

### 1. 次要条件不再吃主条件 fallback

之前 `EvaluateUltimateDecision(...)` 里：

- 如果 secondary condition 也属于支持阈值覆写的类型
- 就会把同一份 `decision.fallback` 继续传给次要条件

这会导致 `Shadowstep` 当前的次要条件：

- `EnemyLowHealthInRange`

在 `35 秒` 后也被放宽成：

- `healthPercentThreshold = 1.0`

功能上虽然没直接出错，但这确实属于隐含副作用，不是原本的设计意图。

这轮已经改成：

- fallback 只传给 primary condition
- secondary condition 固定传 `null`

当前语义变成：

- 主条件可以按时间放宽
- 次要条件保持稳定，不跟着偷偷降门槛

### 2. `ConfigureShadowstepUltimate(...)` 现在只负责 decision，不再重建 skill 内容

之前 `ConfigureShadowstepUltimate(...)` 又重复做了一遍：

- `skillType = Buff`
- `targetType = BackmostEnemy`
- `castRange = 40`
- `effects.Clear()`
- 重新添加 `Untargetable + activeSkillCooldownCapSeconds = 1`

但这些实际上已经在：

- `CreateShadowstepUltimateSkill(...)`

里设置过。

这轮已经把重复部分删掉了。

当前职责边界变成：

- `CreateShadowstepUltimateSkill(...)`
  - 负责 skill 本体结构和效果载荷
- `ConfigureShadowstepUltimate(...)`
  - 只负责 ultimate decision 条件

这样和其他英雄的 `Configure*Ultimate(...)` 写法就一致了，后续维护时也更容易判断“哪一层才是数据真来源”。

### 3. 给 `searchRadius = 0.25f` 补了明确注释

这轮保留了 `0.25f` 这个实现方案，但已经直接在代码旁边写明：

- 第一阶段还没有“直接检查 primaryTarget 血量阈值”的 condition 类型
- 所以这里故意用一个极小搜索半径
- 让 `EnemyLowHealthInRange` 退化成“检查当前选中的后排目标自己是否低于 70% 血”

也就是说：

- 这个值不是随手拍的 magic number
- 而是当前条件系统能力下的明确变通实现

### 4. 给 `targetType = BackmostEnemy` 的自 Buff 语义补了明确注释
