## 这次我实际做了什么

### 1. 新增批量调数窗口

我新增了：

- `game/Assets/Scripts/Editor/BalanceSheetWindow.cs`

当前 Unity 菜单入口：

- `Fight/Tools/Balance Sheets`

窗口职责：

- 让用户选择一个相对项目根目录的表格目录
- 一键导出当前英雄/技能数值到表格
- 一键从表格导入并回写现有 ScriptableObject 资产
- 在窗口里直接提示当前的“不会自动覆盖调数”的安全规则

默认目录：

- `game/BalanceSheets/Stage01/`

### 2. 新增批量调数导入/导出服务

我新增了：

- `game/Assets/Scripts/Editor/BalanceSheetService.cs`

当前导出的表固定为 5 张：

- `heroes.csv`
- `basic_attacks.csv`
- `skills.csv`
- `skill_effects.csv`
- `skill_status_effects.csv`

另外会一起导出：

- `README_批量调数说明.md`

当前导出/导入规则：

- 表头格式是 `稳定英文键|中文名`
- 枚举值格式是 `英文枚举|中文说明`
- 导入时优先按左侧稳定键解析
- 如果枚举列只填中文，工具也会尝试按中文说明识别
- 布尔列支持：
  - `是/否`
  - `true/false`
  - `1/0`
- 空白单元格导入时会被跳过，不会把原值清空
- 导出本身是只读文件输出，不会再调用 `AssetDatabase.SaveAssets()` 去顺手保存其他脏资产

### 3. 表格内容按职责拆分，而不是压成一个大表

这轮没有把所有内容压进一张巨表，而是按数据职责拆开：

- `heroes.csv`
  - 负责 `HeroDefinition.baseStats`
- `basic_attacks.csv`
  - 负责 `HeroDefinition.basicAttack`
- `skills.csv`
  - 负责 `SkillData` 主配置
- `skill_effects.csv`
  - 负责 `SkillData.effects`
- `skill_status_effects.csv`
  - 负责 `SkillEffectData.statusEffects`

这样做的目的：

- 英雄基础属性和技能细节不会混在一起
- 后续做筛选、排序、公式复制时更清楚
- 嵌套结构不会被 Inspector 式思维拖垮

### 4. 支持对技能效果和附带状态做“连续追加式”批量维护

这轮导入工具对嵌套列表的策略是：

- `effectIndex` 用来定位或在末尾连续追加 `SkillEffectData`
- `statusIndex` 用来定位或在末尾连续追加 `StatusEffectData`

当前明确约束：

- 只能修改已有项，或在末尾连续追加 1 个新项
- 不允许跳号补洞
- `skill_status_effects.csv` 不会凭空创建父 `SkillEffectData`
- 如果目标 effect 不存在，必须先在 `skill_effects.csv` 里创建，或确保资产里已存在
- 删除表格中的某一行，不会自动删除资产中已有的旧项

这么做是刻意保守：

- 先保证“批量调数”和“追加配置”不会静默造脏数据
- 暂时不把“删除列表项”做成高风险自动行为

### 5. 把构建前的危险覆盖路径改成只补缺失

之前最危险的点是：

- `WindowsBuildExporter.ExportWindowsPlayer()`
- 在构建前会调用 `Stage01SampleContentBuilder.GenerateDemoContentForBuild()`
- 而旧实现里 `GenerateDemoContentForBuild()` 内部走的是：
  - `GenerateDemoContentInternal(overwriteExistingContent: true)`

这意味着：

- 只要走构建流程，就可能把已经手调好的英雄/技能数值重新写回 demo 默认值

这轮我已改成：

- `GenerateDemoContentForBuild()`
  - 现在走 `overwriteExistingContent: false`

当前默认语义变成：

- 只补齐缺失 demo 资产
- 不覆盖已经存在的英雄/技能调数

### 6. 把“恢复默认值”改成显式危险操作

这轮我在 `Stage01SampleContentBuilder.cs` 里新增了一个显式菜单：

- `Fight/Dev/Regenerate Demo Content From Defaults (Overwrite Existing Tuning)`

当前行为：

- 只有手动点这个菜单
- 且确认弹窗选择继续
- 才会真正执行：
  - `GenerateDemoContentInternal(overwriteExistingContent: true)`

也就是说：

- 现在“覆盖已有调数”不再是日常自动流程的一部分
- 它变成了一个带明确危险提示的人工操作

## 这轮已经刻意规避的风险

### 1. 不再让自动流程覆盖调好的数值

这是这轮最重要的防线。

当前只有一处仍然保留 `overwriteExistingContent: true`：

- `Stage01SampleContentBuilder.RegenerateDemoContentFromDefaults()`

而这条路径是：

- 手动菜单触发
- 带确认弹窗
- 名字里明确写了 `Overwrite Existing Tuning`

所以现在它是显式危险操作，不再是默认行为。

### 2. 不直接改 YAML 文本做批量维护

这轮没有走：

- 直接批量改 `.asset` YAML

而是继续让：

- `ScriptableObject` 资产作为运行时真数据源
- CSV 作为人类更容易批量维护的调数层

这样更符合当前项目已有的数据驱动结构。

### 3. 不把复杂嵌套技能压成一张超大表

这轮没有做“一张表承载所有字段”的方案。

原因：

- `SkillData.effects`
- `SkillEffectData.statusEffects`

本来就是嵌套结构，强行压扁到一张表会让维护成本迅速变差。
