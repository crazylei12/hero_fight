## 预期实现语义

### 1. 圣职者

预期语义如下：

- 英文名和项目代号固定为 `Sunpriest`
- 英雄 ID 固定为 `support_001_sunpriest`
- 普攻为远程投射物治疗普攻
- 普攻目标为友方，优先最低血量友军，允许包含自己
- 小技能 `Radiant Heal` 为单体直接治疗 + 单体护盾
- 大招 `Sun Blessing` 为持续区域治疗，不跟随施法者移动

### 2. 普攻通用化

预期不是“给 Sunpriest 单独写一套特殊普攻逻辑”，而是：

- `BasicAttackData` 允许描述普攻目标类型
- `BasicAttackData` 允许描述普攻效果类型
- `BattleSimulationSystem` 能按普攻目标类型重新选目标
- `BattleBasicAttackSystem` 能按普攻效果类型走伤害或治疗结算
- 投射物普攻仍走统一 projectile 流程

### 3. 护盾

预期不是技能脚本临时维护一份额外护盾值，而是：

- `Shield` 成为正式 `StatusEffectType`
- 伤害统一从 `RuntimeHero.ApplyDamage` 入口先扣护盾再扣血
- 护盾值存在运行时状态里，而不是 HeroDefinition 或技能特例字段里

### 4. 持续区域

这次按用户补充意见改成了更通用的方向。

预期不是：
- `PersistentAreaDamage`
- `PersistentAreaHeal`
- `PersistentAreaBuff`

而是：
- `SkillEffectType.CreatePersistentArea`
- 区域自身有目标阵营配置
- 区域自身有脉冲效果类型配置
- 区域脉冲时再决定是直接伤害、直接治疗，还是只施加状态

## 临时方案草案：远程受击空闲后撤

这一节用于记录 2026-04-16 新讨论出的一个临时实现方案。

约束：

- 这里只记录“准备怎么做”，还不是正式规则文档
- 等代码真正写完、实际观察效果、再做一次检视后，才决定是否写入 `docs/planning/` 下的正式文档

### 目标

希望增加一个通用行为：

- 当英雄的普攻属于 `投射物普攻`
- 且该英雄刚刚被敌方英雄打到
- 且自己当前没有执行可立即生效的攻击或技能动作
- 则优先向远离威胁来源的方向短暂移动
- 形成“打一下后拉开距离”的走 A / 拉扯感

这个方案当前主要面向：

- 射手
- 法师
- 以及其他未来同样使用投射物普攻的后排英雄

### 当前判断

当前代码基础下，这个能力更适合放在：

- `AI 决策 / 移动决策`

而不适合放在：

- 状态系统
- 某个单英雄特例脚本

原因：

- 这是一个“什么时候该后撤”的行为判断，不是单位身上的持续状态
- 现有架构已经要求 `AI 决策` 与 `执行` 分离
- 当前运行时也已经能通过 `BasicAttackData.usesProjectile` 区分投射物普攻英雄

### 现有代码上的落点判断

当前最合适的改动位置预计是：

- `game/Assets/Scripts/Heroes/RuntimeHero.cs`
- `game/Assets/Scripts/Battle/BattleBasicAttackSystem.cs`
- `game/Assets/Scripts/Battle/BattleSkillSystem.cs`
- `game/Assets/Scripts/Battle/BattleAiDirector.cs`
- `game/Assets/Scripts/Battle/BattleSimulationSystem.cs`

理由简述：

- `RuntimeHero` 适合保存“最近是谁打了我”的短时运行时记忆
- `BattleBasicAttackSystem` / `BattleSkillSystem` 里有实际命中和伤害生效入口
- `BattleAiDirector` 适合放通用的后撤判定和方向计算
- `BattleSimulationSystem` 适合接入真正的“这一帧是否后撤”分支

### 计划中的最小实现

第一版先按最小增量做，不先引入完整动作状态机。

#### 1. 给 RuntimeHero 增加短时 threat 记忆

计划增加类似字段或能力：

- `LastThreatSource`
- `LastThreatTimeSeconds`
- `RecordThreat(...)`
- `TryGetRecentThreat(...)`

用途：

- 让 AI 知道“我刚刚是被谁打到的”
- 只保留一个很短的最近威胁窗口，不做完整仇恨系统

#### 2. 在实际命中处记录 threat

记录时机以“真正命中/真正掉血”为准，而不是敌方刚出手时就记录。

预计接入点：

- 基础攻击命中
- 技能直接伤害命中
- 持续区域或持续伤害命中

最小规则：

- 只记录敌方来源
- 没有造成有效伤害时先不强行记入

#### 3. 在 BattleAiDirector 中补后撤判定

计划补充一组通用判断：

- 是否属于投射物普攻英雄
- 当前 threat 是否仍在有效时间窗内
- 当前敌人是否已经压得过近
- 当前是否值得后撤，而不是继续前压

第一版预计采用“带迟滞的距离带”而不是硬切开关。

例如可先按下面思路试：

- 触发后撤：敌人距离 `< 期望输出距离 - 0.4`
- 停止后撤：敌人距离 `>= 期望输出距离 + 0.2`
- threat 有效窗口：`0.6s ~ 0.8s`

这样做的目的：

- 减少原地来回抖动
- 避免单位每帧在“前进 / 后退”之间切换

#### 4. 在 BattleSimulationSystem 中插入 retreat 分支

预计接入顺序：

- 先正常做 `hero.Tick(...)`
- 先尝试技能释放
- 再取当前普攻目标
- 在默认前压 / 普攻前，额外判断是否应该后撤

第一版计划把“空闲状态”暂时解释为：

- 当前帧没有释放技能
- `AttackCooldownRemainingSeconds > 0`
- 允许移动

说明：

- 这是基于现有代码结构的务实定义
- 当前系统里还没有正式的“攻击前摇 / 施法读条 / 后摇”动作状态
- 如果后续真要更精确地区分“正在攻击”“正在施法”“空闲”，再补统一动作状态层

### 第一版明确不做的内容

为了避免范围膨胀，第一版先不做：

- 完整仇恨系统
- 完整职业状态机重构
- 攻击前摇 / 施法读条 / 后摇的正式动作状态系统
- 为每个远程英雄单独写一套拉扯特例
- 接入寻路插件解决复杂走位

第一版只追求：

- 在现有自动战斗循环里，给投射物普攻后排增加一个最小可观察的“受压后撤”行为

### 需要重点观察的风险

代码真正实现后，优先观察以下问题：

- 会不会在攻击距离边缘来回抖动
- 会不会因为后撤过多导致远程单位几乎不出手
- 会不会和当前技能释放逻辑打架
- 会不会让治疗型或辅助型投射物普攻出现不合理逃跑
- 会不会让近战或非投射物英雄误触发
- 会不会制造过多目标切换或日志噪声

### 2026-04-16 当日补充修正

根据后续讨论，这个临时方案又补了两条更明确的边界：

- `DOT / 持续伤害` 不再刷新 retreat threat
- `远程对远程` 的正常互点，不应仅仅因为“刚刚吃到伤害”就触发后撤

当前更准确的理解应改为：

- threat 仍然以“有效伤害真正生效”作为候选记忆来源
- 但 retreat 触发不能只看“我刚掉血了”
- 还要再额外判断这个 threat 是否属于“值得后撤的威胁”

第一版的具体收口方向：

- 近战 / 短手 / 贴脸威胁：允许正常触发 retreat
- 远程 threat：只有当它已经压进异常近距离时，才允许触发 retreat

这样做的目的：

- 避免两个远程互点时，双方 `A 一下 -> 一起后撤 -> 再一起走回去`
- 避免把“正在交战”误判成“应该拉扯”

### 预期实现语义

#### 1. 触发对象

预期只有满足下面条件的英雄才会进入这一套行为：

- 普攻配置是 `usesProjectile = true`
- 允许移动
- 最近短时间内被敌方英雄造成了有效伤害

预期不是：

- 所有远程标签英雄都自动后撤
- 所有被打的英雄都后撤
- 近战、坦克、战士也走这一套

#### 2. threat 记录时机

预期是：

- 只有在“伤害真正生效”后才记录 threat
- 记录来源必须是敌方单位
- threat 入口应覆盖：
  - 普攻伤害
  - 技能直接伤害

预期不是：

- 敌方刚出手就提前记 threat
- 治疗或护盾也会触发 threat
- 友方来源也会误记成 threat

#### 3. retreat 触发条件

预期第一版语义是：

- 本帧没有因为技能释放直接 `return`
- 当前还有一个有效的普攻目标
- 普攻仍在冷却中，暂时不能立刻出手
- threat 距离过近
- 符合后才优先向远离 threat 的方向移动

这里的“空闲状态”当前是近似定义，不是正式动作状态机。

#### 4. retreat 行为边界

预期是：

- 只是短时拉开距离
- threat 窗口过去后，应恢复原本的索敌 / 前压 / 普攻流程
- 不应长期进入逃跑状态
- 不应把战斗变成“全程 kite 到角落”

### 需要重点怀疑的点

下面这些点是这轮最值得对着代码找 bug 的地方。

#### A. “空闲状态”其实只是近似定义

相关文件：

- `BattleSimulationSystem.cs`
- `BattleAiDirector.cs`

当前实现里，“空闲”不是正式动作状态，而是近似成：

- 本帧没先放技能
- `AttackCooldownRemainingSeconds > 某个小阈值`

请重点检查：

- 这会不会让某些其实应该站定等出手的单位过早后撤
- 这会不会在未来接入攻击前摇 / 施法读条后变成结构冲突

#### B. 远程对远程互点是否仍会触发不自然双后撤

相关文件：

- `BattleAiDirector.cs`
- `RuntimeHero.cs`

当前已经明确收口为：

- `DOT / 持续伤害` 不再刷新 retreat threat
- 但仍需要检查“远程 threat 是否会被误判成该后撤的威胁”

请重点检查：

- 两个远程正常互点时，是否仍会出现双方同步后撤
- “远程 threat 只有压进异常近距离才算危险”这个条件是否足够稳定
- 会不会仍然出现 `A 一下 -> 双方都退 -> 再一起走回去` 的机械行为

#### C. support / 治疗型投射物英雄会不会后撤过头

相关文件：

- `BattleAiDirector.cs`
- `BattleSimulationSystem.cs`

当前规则按 `usesProjectile` 硬边界走，理论上会覆盖射手、法师，也可能覆盖治疗型投射物英雄。

请重点检查：

- Sunpriest 这类单位会不会因为 retreat 过于频繁而难以稳定治疗
- 当前 `Support` 的 `GetDesiredCombatRange(...)` 和 retreat 距离带是否冲突

#### D. retreat 是否会制造目标切换噪声

相关文件：

- `BattleSimulationSystem.cs`
- `BattleAiDirector.cs`

请重点检查：

- retreat 过程中是否会导致目标频繁重选
- 会不会制造过多 `TargetChangedEvent`
- 会不会出现“刚退一步又立刻前压再退”的来回拉扯
